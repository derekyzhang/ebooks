前言
================================================================================

关于本文档

本文档描述了AMD I/O虚拟化技术。AMD I/O虚拟化技术体现在称为I/O内存管理单元
（IOMMU）的系统级功能中。

目标读者

本文档提供了IOMMU的行为定义和相关设计说明。它面向参与AMD产品（使用x86/AMD64
微处理器架构）的操作系统内核软件和驱动程序、虚拟机管理程序（Hypervisor）以及固件
开发的系统设计人员、芯片组设计人员和程序员。目标用户应具有个人计算机设计和AMD
目标平台系统架构的相关经验。相关参考文献列表请参见第26页"相关文档"。

文档组织

• 第1章"IOMMU概述"（第29页）提供了AMD I/O虚拟化技术和IOMMU的介绍。
• 第2章"架构"（第47页）描述了IOMMU的操作以及控制其行为的寄存器和系统内存
  数据结构。
• 第3章"寄存器"（第191页）展示了IOMMU寄存器的格式，并描述了每个寄存器中的
  数据字段。
• 第4章"实现注意事项"（第269页）讨论了主要与IOMMU实现者相关的设计和实现
  问题。
• 第5章"I/O虚拟化ACPI表"（第275页）定义了用于描述IOMMU控制字段的平台配置
  信息的ACPI表。
• 附录"寄存器索引"（第297页）提供了所有IOMMU寄存器定义的索引。

约定与定义
================================================================================

符号约定

• 128
  没有字母后缀的数字为十进制，除非上下文另有说明。
• 1011b
  二进制值——在本例中为4位值。
• F0EA_0B02h
  十六进制值。可插入下划线字符以提高可读性。
• CR0–CR4
  寄存器范围，从寄存器CR0到CR4（含），低序寄存器在前。
• CR0[PE]
  引用寄存器中字段的符号——在本例中为CR0寄存器的PE字段。
• 7:4
  位范围，从第7位到第4位（含）。高序位在前。可插入逗号表示间隔。

定义

• 已访问位（A）。页表中的一个位，表示对应的内存已被读取或写入。通常由硬件设
  置为1。
• ACPI。高级配置和电源接口（Advanced Configuration and Power Interface），一种
  行业标准接口规范，用于操作系统主导的配置和其他管理。
• APIC。高级可编程中断控制器（Advanced Programmable Interrupt Controller）
  （参见型号82093AA和82489DX下的规范）。
• ARI。替代路由信息（Alternative Routing Information），是PCI-SIG规范，允许PCI
  设备拥有超过八个PCI功能但不超过256个。
• ATS。地址转换服务（Address Translation Service），是PCI-SIG规范，允许PCI外设
  向IOMMU或TA请求虚拟到物理地址的转换。转换结果可存储在IOTLB中。ATS在外设
  上是可选的。本规范要求地址转换服务1.1规范或更高版本。参见
  http://www.pcisig.com/specifications/iov/ats/。
• AVIC。AMD高级虚拟中断控制器（AMD Advanced Virtual Interrupt Controller）。
  参见《AMD64架构程序员手册，卷2：系统编程》中的"高级虚拟中断控制器"，
  订单号#24593（APM2），第15章。AVIC是Guest虚拟APIC的一种实现。允许处理器
  和IOMMU协调将中断直接传递到运行中的Guest虚拟机。
• BAR。PCI定义的基地址寄存器（Base Address Register）。
• BDF。PCI总线I/O设备标识符；由总线号、设备号和功能号串联而成。在本文档中也
  称为DeviceID。
• BIOS。指平台固件（基本输入/输出服务）。另见UEFI。
• 跳板缓冲区（Bounce Buffer）。位于低系统内存中的缓冲区，用于不支持64位寻址
  的设备的DMA传输。操作系统将DMA数据复制到该缓冲区或从该缓冲区复制到驱动
  程序使用的高内存中的实际缓冲区。
• 冷复位（Cold Reset）。通过移除并重新施加设备电源产生的复位。
• 脏位（D，Dirty bit）。页表中的一个位，表示对应的内存已被写入。通常由硬件设
  置为1。
• 设备排除向量（DEV，Device Exclusion Vector）。物理内存中的连续位数组。DEV
  表中的每一位代表4KB的物理内存页（包括系统内存和MMIO）。DEV表的打包方式
  如下：字节0的位[0]控制物理内存的前4KB；字节0的位[1]控制物理内存的第二个
  4KB；依此类推。
• DeviceID。由总线号、设备号和功能号组成的16位设备标识号，也称为BDF或BDFID。
  IOMMU使用它来选择地址转换或中断重映射操作的嵌套映射表。
• 设备处理复合体（Device Processing Complex）。外设上的计算单元，如专用功能
  （例如NIC、加密引擎）、图形处理单元（GPU）或加速计算元素（AC）。
• 设备表（Device Table）。系统内存中的表，将DeviceID映射到DomainID和页表根
  指针。
• 设备表项（DTE，Device Table Entry）。设备表中的一个条目。
• 直接内存访问（DMA，Direct Memory Access）。一种允许外设在不需要中央处理器
  干预的情况下访问内存的特性。
• 设备虚拟地址（DVA，Device Virtual Address）。设备在DMA事务中发出的未经转换
  的地址。如果IOMMU排除了该设备的转换，则此地址可对应系统物理地址；如果设备
  由Guest操作系统拥有和编程，则对应GPA。
• 域（Domain）。参见保护域。
• DomainID。软件选择的16位数字，用于标识一个域。
• 下行（Downstream）。从根复合体（Root Complex）到设备端点的流量。
• GART。图形地址重映射表（Graphics Address Remapping Table）。
• GPU。图形处理单元，通常用于图形特定的计算。
• GPGPU。用于通用计算的GPU。
• Guest。主机在其自己的虚拟环境中运行的应用程序或操作系统。
• Guest地址转换。GVA到GPA的转换。可由IOMMU或外设上的私有MMU提供服务。
• Guest物理地址（GPA，Guest Physical Address）。虚拟机中Guest操作系统使用的
  x86规范虚拟地址。GPA通过使用Guest页表转换Guest虚拟地址而创建。GPA可进一步
  转换为系统物理地址。
• Guest虚拟地址（GVA，Guest Virtual Address）。Guest应用程序使用的虚拟地址。
  GVA可转换为Guest物理地址。Guest虚拟地址被视为规范x86地址。
• Guest虚拟APIC。可选地，IOMMU可以支持将中断传递到Guest虚拟机而无需
  Hypervisor干预。Guest APIC在AMD虚拟中断控制器规范修订版1.0或更新版本中
  描述。
• 主机数据路径（HDP，Host Data Path）。一种功能单元，可将CPU线性寻址转换为
  GPU风格的平铺（tiled）或矩形寻址以提高性能。通常存在于高级图形处理外设中。
• 高内存（High memory）。在x86平台架构中，位于等于或大于4GB地址处的系统内存。
• 主机（Host）。负责运行Guest的系统软件层。另见嵌套分页和嵌套地址转换。
• 主机物理地址（HPA，Host Physical Address）。主机系统硬件用于访问物理资源
  （内存或MMIO寄存器）的物理地址。主机物理地址范围通常由操作系统或虚拟化环境
  中的Hypervisor管理，在大多数系统中与系统物理地址相同。
• Hypervisor。Hypervisor（HV）是计算机的控制软件。它管理物理硬件和虚拟机，
  允许多个操作系统在一个计算机系统上并发运行。
• IOMMU。指本规范定义的I/O内存管理单元。
• IOTLB。I/O转换后备缓冲区（I/O Translation Look-aside Buffer）。位于外设中的
  缓冲区，保存预转换的地址。有时称为"远程IOTLB"。IOTLB的一个例子是PCIe®
  地址转换缓存。
• IRTE。中断重映射表项（Interrupt Remapping Table Entry）。
• IVHD。I/O虚拟化硬件定义块（I/O Virtualization Hardware Definition block），
  是第5.2.2.1节[I/O虚拟化硬件定义（IVHD）块]中定义的ACPI表。
• IVMD。I/O虚拟化内存定义块（I/O Virtualization Memory Definition block），
  是第5.2.2.2节[I/O虚拟化内存定义（IVMD）块]中定义的ACPI表。
• IVRS。I/O虚拟化报告结构块（I/O Virtualization Reporting Structure block），
  是第5.2节[I/O虚拟化报告结构（IVRS）]中定义的ACPI表。
• LMA。本地内存地址（Local Memory Address）；对应外设上用于访问板载或私有内存
  的物理地址空间。在某些外设中，孔径（aperture）硬件将部分或全部本地内存地址
  空间映射到系统物理地址空间。孔径硬件通常由操作系统中的设备驱动程序管理。
• 本地内存（Local Memory）。外设上的内存，通常比系统内存访问更快，且通常与系
  统内存不一致（non-coherent）。部分本地内存可从CPU寻址（称为"公共"），部分
  可能从CPU不可访问（称为"私有"）。通常使用孔径机制选择本地内存的公共部分。
• 本地内存保护映射（Local Memory Protection Map）。一种硬件组件，在外设的本地
  内存中强制分离虚拟机上下文。
• 低内存（Low memory）。在x86平台架构中，位于4GB以下的系统内存。
• 主中止（Master-Abort）。一种PCI终止机制，当没有目标响应时允许主设备终止事务。
• 内存地址路由控制（MARC，Memory Address Routing Controls）。MARC是一个系统，
  为具有低延迟访问需求的Guest分配实时能力设备指定一个定义的孔径，以访问由
  Hypervisor控制的专用物理内存范围。需要低延迟系统内存访问路径的Guest分配设备
  的例子包括显示扫描输出引擎或帧缓冲区捕获设备。
• MMIO。内存映射I/O（Memory Mapped I/O）。对设备提供的内存映射资源的读或写访问。
• MMU。内存管理单元（Memory Management Unit）。
• 消息信号中断（MSI，Message Signalled Interrupt）。通过向系统定义的物理地址
  生成已发布写（posted write）来发出信号的中断。
• 嵌套地址转换（Nested address translation）。GPA到SPA的转换。可由IOMMU直接
  提供服务或由远程IOTLB提供。使用IOTLB需要ATS和/或PRI。
• 嵌套分页（Nested paging）。AMD64处理器中的可选特性，嵌套分页特性提供两级地
  址转换，从而消除了虚拟机管理器维护影子页表的需要。参见《AMD64架构程序员手
  册，卷2：系统编程》，订单号#24593（APM2）。
• NW。PCI-SIG术语（位），用于信号通知缺乏执行写操作的意图。
• NX。不可执行（No-execute）。页表项（PTE）字段，指示不应从引用的页面执行程序
  代码。
• 页表（Page Tables）。主内存中的表结构，用于将地址从一种表示转换为另一种表示。
• PASID。进程地址空间ID（Process Address Space ID），用于标识x86规范Guest虚拟
  机内的应用程序地址空间。在外设上用于隔离驻留在共享本地内存中的并发上下文。
  PASID和DeviceID一起唯一标识一个应用程序地址空间。参见PASID TLP前缀。
• PASID TLP前缀。IOMMU要求带PASID的虚拟地址使用PASID TLP前缀携带PASID值。
  另见PASID和TLP。参见PCI-SIG PASID TLP前缀ECN规范。
• PCI, PCI-SIG, PCIe®, PCI-X®。PCI-SIG是定义I/O连接技术（包括PCI、PCI-X和
  PCIe）的行业标准组织。更多信息请参见 http://www.pcisig.com/home。
• PDE。用于地址转换的页目录项（参见第80页图10中的示例）。
• 固定内存（Pinned memory）。始终保持在物理内存中的内存页。固定内存页可防止
  页管理软件将其用于其他目的。通常在DMA开始前必须固定内存页，DMA完成后可
  解除固定。
• 平台固件（Platform firmware）。控制平台启动和配置的固件或软件。平台固件通
  常实现为BIOS或UEFI。
• 平台安全处理器（PSP，Platform Security Processor）。也称为AMD安全处理器
  （AMD-SP）。向Hypervisor公开一组功能，用于SEV和SNP启用的Guest的生命周期
  管理。
• PPR。外设页请求（Peripheral Page Request）。当IOMMU收到有效的PRI请求时，
  它创建一个PPR消息以请求更改虚拟地址空间。
• PR, P。存在位（Present）。页表项（PTE）字段，指示指向的页表或物理页当前已
  加载到系统内存中。
• 预转换地址（Pretranslated address）。已由带IOTLB的外设转换为SPA的地址。
• 页请求接口（PRI，Page Request Interface）。PCI-SIG规范，定义外设如何向主机
  操作系统或Hypervisor请求内存管理服务（例如，为外设提供缺页服务）。PRI在外设
  上是可选的，但如果实现了PRI，则需要ATS。
• 私有MMU、设备MMU或设备页表。外设特定的机制，用于转换外设上生成的地址。
  在最简单的情况下，它生成一个位来指示输入地址是对外设本地内存还是系统内存的
  访问。当存在时，私有MMU、设备MMU或设备页表提供Guest地址转换。在GPU上，
  私有MMU、设备MMU或设备页表通常被称为内存控制器的VM组件。
• 保护域（Protection Domain）。可由多个设备共享的一组地址映射和访问权限。
• PSP。参见平台安全处理器。
• PTE。页表项（Page Table Entry）。页表转换条目，控制虚拟到物理地址的转换和
  内存页访问（参见第78页图9中的示例）。
• 保留（Reserved）。指定为保留的寄存器字段需要软件特殊处理。可写寄存器中的保
  留字段必须写入全零。读取时，软件不能依赖返回的值。
• 反向映射表（RMP，Reverse Map Table）。SEV-SNP数据结构，用于支持对内存访问
  的额外安全检查。更多详情请参阅《AMD64架构程序员手册，卷2：系统编程》中的
  "安全嵌套分页（SEV-SNP）"，订单号#24593（APM2）。
• 安全加密虚拟化（SEV，Secure Encrypted Virtualization）。一种允许虚拟机（VM）
  的内存内容使用该VM唯一的密钥进行透明加密的特性。
• 安全嵌套分页（SEV-SNP，Secure Nested Paging）。SNP提供VM内存的完整性保护，
  以帮助防止依赖Guest数据损坏、别名、重放和各种其他攻击向量的基于Hypervisor的
  攻击。
• 系统物理地址（SPA，System Physical Address）。直接用于寻址系统内存的地址。
  在SVM下，这也称为主机物理地址。参见HPA。
• 系统软件（System software）。管理系统硬件资源并控制低特权软件访问这些资源
  的特权软件。在非虚拟化环境中，操作系统是系统软件。在虚拟化环境中，
  Hypervisor（HV）是系统软件。
• TA。转换代理（Translation Agent），是PCI-SIG术语，指IOMMU表遍历器。
• 目标中止（Target-Abort）。一种PCI终止机制，当发生致命错误或目标永远无法响
  应时，允许目标终止事务。
• TLB。转换后备缓冲区（Translation Look-aside Buffer），通常在MMU内实现的地
  址转换信息缓存，用于提高转换速度。
• TLP。事务层数据包（Transaction Layer Packet），是PCIe术语，指非控制数据包。
  TLP数据包可能带有前缀。
• UEFI。指"统一可扩展固件接口"（Unified Extensible Firmware Interface）平
  台固件规范。参见 http://www.uefi.org/home/。另见BIOS。
• 未转换地址（Untranslated address）。由外设发出的虚拟地址（GVA或GPA），将由
  IOMMU转换为SPA。外设上对未转换地址的处理超出本规范的范围。
• 上行（Upstream）。从设备端点到根复合体的流量。
• 用户（User）, U/S, 用户/特权级别。IOMMU可以向外设提供特权级别信息。值0b表
  示允许特权级别访问，但不允许用户级别访问；1b表示允许用户和特权级别访问。根
  据上下文使用User和U/S术语。
• VM。虚拟机由Hypervisor创建和管理，使多个虚拟机可以共享单个硬件系统并运行
  独立的操作系统实例。

第1章  IOMMU概述
================================================================================

本章概述了IOMMU的功能，并介绍了几种使用模型。IOMMU的详细架构在第2章"架构"
中讨论。

I/O内存管理单元（IOMMU）通过添加对来自外设的DMA传输的地址转换和系统内存访问
保护支持，扩展了AMD64系统架构。IOMMU还有助于过滤和重映射来自外设的中断。

IOMMU实现了几项重要的系统级增强功能：
• 在64位系统上支持传统32位I/O设备（通常不需要跳板缓冲区和昂贵的内存复制）。
• 更安全的用户级应用程序对选定I/O设备的访问。
• 更安全的虚拟机Guest操作系统对选定I/O设备的访问。

IOMMU可用于：
• 替换现有的图形地址重映射表（GART）机制。
• 为不支持64位寻址的I/O设备重映射4GB以上的地址。
• 允许在虚拟机上运行的Guest操作系统直接控制分配的设备。
• 提供页粒度的设备系统内存访问控制。
• 允许设备直接访问用户空间I/O。
• 允许将中断直接传递给Guest操作系统。
• 过滤和重映射中断。
• 与选定的外设共享进程虚拟地址空间。
• 隔离/沙箱化设备，防止恶意DMA访问内存中安全敏感的操作系统和用户数据。
• 强制执行操作系统的数据访问安全策略。

IOMMU可以被认为是AMD64架构中两个设施的泛化：GART和设备排除向量（DEV）。
GART对系统物理地址空间的一小部分范围提供I/O设备访问的地址转换，DEV提供有限
程度的I/O设备分类和内存保护。通过适当的软件支持，IOMMU可以模拟GART或DEV的
功能。

IOMMU可选地提供重映射外设中断向量的能力。

--------------------------------------------------------------------------------
1.1 IOMMU功能概述
--------------------------------------------------------------------------------

IOMMU扩展了AMD64 DEV首次引入的保护域（或简称域）的概念。IOMMU允许系统中的
每个I/O设备被分配到特定的域和一组不同的I/O页表。当I/O设备尝试读取或写入系统
内存时，IOMMU拦截该访问，确定设备被分配到的域，并使用与该域关联的TLB条目或
与该I/O设备关联的I/O页表来确定该访问是否应被允许，以及要访问的系统内存中的
实际位置。

IOMMU可能包含对远程IOTLB的可选支持。带有IOTLB支持的受信任I/O设备可以与
IOMMU协作维护自己的地址转换缓存。这创建了一个可扩展的系统框架，其中I/O设备
可能具有不同的使用模型和工作集大小。具有IOTLB能力的I/O设备包含为自身需求定
制的私有TLB，创建了一个可扩展的分布式TLB系统。具有IOTLB能力的I/O设备的性能
不受IOMMU中实现的TLB条目数量的限制。具有IOTLB的外设可以发出未转换地址或从
IOTLB条目确定的预转换地址。预转换地址不会被IOMMU检查，除了验证外设在对应的
设备表项中设置了IOTLB使能位（I = 1）（参见图7和表7）。

IOMMU提供的主要系统资源包括：
• 使用基于内存的转换表进行I/O DMA访问权限检查和地址转换。
• 可选支持与AMD64长模式页表格式兼容的Guest转换表。
• 设备表，允许将I/O设备分配到特定域，并包含指向I/O设备页表的指针。
• 中断重映射表，IOMMU使用它为I/O设备中断提供权限检查和中断重映射。
• 可选的AMD64 Guest虚拟APIC机制，IOMMU使用它将中断传递给Guest虚拟机。
• 基于内存的队列，用于在IOMMU和系统处理器之间交换命令和状态信息。
• 可选支持外设页请求（PPR）日志。
• 缓解PPR和事件日志溢出的特性。
• 可选支持基于硬件的机制，允许特权I/O设备直接访问系统内存的定义区域。

IOMMU类似于处理器的内存管理单元，但它为外设的直接内存访问（DMA）而非处理器
的内存访问提供地址转换和页保护。

IOMMU在处理未转换的已发布请求时，不会向I/O设备直接指示转换失败。这与CPU的
MMU采用的缺页机制形成对比。

IOMMU支持的系统可以由多个处理器节点和设备节点组成，通过数据结构（如AMD
Infinity Fabric、数据结构链路或其他方式）相互连接。IOMMU只能处理通过其在系统
结构中的节点路由的内存事务。在具有到I/O设备的多条链路和总线的系统中，需要多
个IOMMU以确保每条I/O链路或总线都应用了适当的保护和转换。图1显示了一个具有两
个I/O互连树的平台示例。注意，两棵树的根节点都需要一个IOMMU。在此示例中，
IOMMU作为I/O集线器的一部分实现。只要符合相同的拓扑，其他实现也是可能的。

[图1: 示例平台架构]

IOMMU使用内存中的命令队列（命令缓冲区）来接受系统软件发起的显式转换缓冲区
无效命令。

可选地，IOMMU可能包含对使用地址转换服务（ATS）的外设的外设页请求（PPR）支
持。这为外设和软件创建了一种机制，以减少I/O期间对固定页的需求。IOMMU可能
包含对中断虚拟化的可选支持。这使用虚拟化Guest APIC（Guest APIC的一种实现
是高级虚拟中断控制器）和内存表将中断传递给Guest虚拟机。

--------------------------------------------------------------------------------
1.2 使用模型
--------------------------------------------------------------------------------

讨论了七种模型，以突出IOMMU在传统和虚拟化系统中的潜在用途。这些使用模型可以
增强系统安全性和稳定性。

1.2.1 替换GART

GART是AMD64系统设施，对图形孔径内的内存地址执行物理到物理的转换。GART的设计
目的是允许复杂的图形对象（如纹理映射）对图形协处理器来说看起来好像位于连续的
内存页中，即使大多数操作系统实际上将它们分散在随机分配的页面中。GART转换所有
对图形孔径的访问，包括主机处理器执行的加载和存储以及I/O设备执行的内存读写。
只有系统物理地址在GART孔径内的访问才会被转换；但是，转换结果可以是任何系统
物理地址。

要为I/O设备发起的访问设置等效的转换，主机操作系统必须：
• 构建指定所需转换的I/O页表。
• 在设备表中创建一个指向新构建的I/O页表的条目。
• 通知IOMMU新更新的设备表项。

此时，主机处理器和图形设备的所有访问都映射到与GART相同的页面。

如果主机操作系统更改了页保护或转换，它必须同时更新处理器页表和（如果未共享的
话）I/O页表，并向处理器和IOMMU发出适当的页无效命令。与处理器不同，IOMMU在
对I/O页表进行任何更改后都需要页无效命令。（AMD64处理器在对叶页表项进行添加
权限且不更改转换的更改后不需要页无效操作。）页表共享在第2.2.1节[更新共享表]
和第2.2.4节[共享AMD64处理器和IOMMU页表——GPA到SPA]中讨论。

由于IOMMU不提供重新启动对未映射或受保护地址的设备访问的设施，因此必须以适当
的权限映射设备可能访问的所有页面。在这方面，IOMMU类似于GART。

1.2.2 替换设备排除向量机制

设备排除向量（DEV）是AMD64安全虚拟机（SVM）架构引入的基本安全机制。与IOMMU
类似，DEV允许将I/O设备分类到不同的域。与每个域关联的是一个位向量，由物理页
地址索引，指示该域中的I/O设备是否被允许访问对应的物理页。

IOMMU提供保护和转换。如果只需要保护，软件可以创建指定所需保护的恒等映射I/O
页表。

1.2.3 32位到64位传统I/O设备映射

随着大容量物理内存的出现，依赖DMA的传统32位设备不再能任意访问系统内存。这使
操作系统复杂化，因为它们必须引入低内存和高内存的区分，并执行适当的记录以确保
传统I/O设备只被命令使用低内存执行传输。代价不仅仅是复杂性；例如，要从传统
I/O设备执行到高内存的传输，操作系统通常在低内存中分配一个跳板缓冲区，在低内
存中执行传输，然后将结果复制到高内存中的真实目标。对于磁盘控制器和网络接口等
高带宽I/O设备，跳板缓冲区分配和复制的性能开销可能很大。

在某些操作系统中，GART已被用于解决此问题。当操作系统希望在传统I/O设备和高内
存之间执行传输时，它分配GART孔径的一部分并将这些页面映射到高内存。然后它命令
I/O设备使用GART孔径内的地址执行传输，该地址必须位于低内存中。虽然这种方法避
免了跳板缓冲区复制的开销，但并不理想，因为相对较小的GART孔径必须由系统中的
所有传统I/O设备和任何图形处理器共享。设备驱动程序在GART孔径中的页面分配和释
放方面有额外的锁定和同步开销，由于序列化等待GART孔径可用，系统性能可能会
降低。

IOMMU提供了更好的解决方案。首先，IOMMU转换适用于I/O设备可以生成的全部地址范
围，而不是要求高内存传输只能映射在GART地址的狭窄范围内。此外，IOMMU将每个
I/O设备分配到不同域的能力意味着大量使用的I/O设备可以拥有自己的I/O页表集，而
不必与其他I/O设备争夺I/O页面的分配和释放。

1.2.4 用户模式设备访问

IOMMU在允许任意I/O设备被用户级进程安全控制方面起着至关重要的作用，因为由
IOMMU转换内存访问的I/O设备只能访问由关联的I/O页表显式映射的页面。因此，I/O
设备的访问可以被限制在用户进程合法有权访问的那些页面内。

为用户级I/O设置IOMMU与GART模拟类似，但有两个不同之处：首先，可映射地址范围
是I/O设备可生成地址的整个范围；其次，操作系统不一定需要在处理器页表中创建完
全等效的映射（尽管它很可能会这样做）。

即使在IOMMU的帮助下，启用用户级I/O设备访问也涉及许多设计考虑。保护和重映射
DMA是问题的一部分；另一部分是中断管理，IOMMU为此提供帮助。

与GART模拟的情况一样，系统软件必须评估是否需要将用户级进程控制的I/O设备可能
访问的所有页面锁定在内存中。如果I/O设备可以使用IOMMU可选实现的外设页请求
（PPR）机制在其接触页面时生成缺页，则可能可以避免需要锁定在内存中。

1.2.5 虚拟机Guest访问设备

IOMMU可用于允许未修改的虚拟机Guest操作系统直接访问I/O设备。这实际上只是允许
用户级访问I/O设备的特殊情况，但有一些值得单独提及的考虑因素。

首先，非VM感知的Guest当前没有办法通知其Hypervisor（HV）I/O设备可能访问哪些
页面，因此HV必须将整个Guest锁定在内存中。HV为Guest的I/O页表应该简单地将
Guest物理地址映射到系统物理地址。如果HV在嵌套分页下运行Guest并使用与IOMMU
兼容的嵌套页表，那么IOMMU可以直接共享该Guest的主机CPU页表。

通常，单个虚拟机（VM）Guest直接访问多个I/O设备。按设计，所有需要看到完全相
同I/O页转换的Guest中的I/O设备可以共享一个DomainID（参见第56页"数据结构"）。
如果属于给定VM Guest的所有I/O设备被分配到同一域，那么IOMMU可以在该Guest的
任何I/O设备之间共享转换缓存条目。

最后，当Guest内存使用主机系统上的大页分配时，Guest I/O吞吐量通常会显著提高。
这样I/O页表也可以使用大页，IOMMU更不容易在其转换缓存中发生抖动。

1.2.6 虚拟化IOMMU

IOMMU的设计使得它可以被希望向其VM Guest展示IOMMU假象的HV通过软件模拟。

运行非VM感知Guest的HV已经拦截并模拟其Guest访问PCI配置空间的尝试。因此，
IOMMU配置寄存器的模拟是直接的；模拟可以直接挂钩到HV现有的PCI配置空间访问
拦截设施上。

HV还必须安排拦截和模拟Guest对IOMMU的MMIO映射命令寄存器的访问。由于每次HV
拦截的开销很高，访问IOMMU的Guest操作系统在通过MMIO寄存器访问启动IOMMU命令
处理之前，将命令批量排入IOMMU在系统内存中的命令缓冲区时性能更好。

由于不受信任的Guest操作系统不能被允许写入真实的设备表，HV必须代表Guest在真
实表中维护影子条目。IOMMU架构要求软件在更新设备表项后向IOMMU发出无效条目命
令。HV可以拦截这些无效命令，查找Guest模拟设备表中的对应条目，并代表Guest在
真实设备表中创建影子条目。注意，Guest看到的DeviceID不需要与真实的DeviceID
相同，Guest使用的DomainID几乎肯定与HV在真实设备表中使用的DomainID不同。

此外，对于每个Guest VM的I/O页表，HV必须构建一个影子I/O页表。这个影子I/O页
表是给真实IOMMU使用的页表。不幸的是，由于不完整的I/O设备访问无法重新启动，
HV必须在Guest为该域启用分页后立即积极地构建每个Guest域的完整影子I/O页表。
HV必须对Guest写保护Guest I/O页表，以拦截所有Guest更新并将更新传播到影子I/O
页表。

Hypervisor（HV）也可以通过IOMMU扩展特性寄存器[MMIO偏移0030h]报告可选特性的
子集来实现该子集。可以使用上述相同技术实现额外特性的子集。

1.2.7 虚拟化用户模式设备访问

具有两级转换的IOMMU在强制系统保护策略的同时，允许虚拟化系统中的用户级进程正
确控制任意I/O设备。如第1.2.4节[用户模式设备访问]所述，由IOMMU转换内存访问
的I/O设备只能访问由HV和操作系统授权的关联I/O页表显式映射的页面。因此，当设
备提供PASID信息时，I/O设备的访问可以被限制在用户级进程合法有权访问的那些页
面内。这意味着I/O操作可以在没有Hypervisor或操作系统干预的情况下启动。

除地址转换外，启用用户级I/O设备访问还涉及其他设计考虑，如重映射中断。

系统软件必须评估是否需要将用户级进程控制的I/O设备可能访问的所有页面锁定在内
存中。使用ATS的外设可以在IOMMU实现外设页请求机制时使用该机制来避免需要锁定
所有页面。与GART模拟的情况一样，系统软件必须评估是否需要将用户级进程控制的
I/O设备可能访问的所有页面锁定在内存中。如果外设支持PCI Express ATS协议并实
现了IOTLB，并且还实现了对可选外设页请求协议（PRI/PPR）的支持，那么系统软件
（操作系统或Hypervisor）能够为用户级控制的I/O设备支持可分页内存访问。

1.3 IOMMU可选特性
--------------------------------------------------------------------------------

IOMMU的所有实现都提供一组基本功能。这种基本功能也称为IOMMU修订版1。

本规范的后续修订版添加了新的软件可观察特性。所有这些特性在技术上都是可选的，
尽管大多数IOMMU实现都包含了在其设计时已定义的所有特性。

但是，从架构上讲，软件必须确定对每个特性的支持，并且必须在使用每个特性之前启
用它。可选特性包括：
• Guest虚拟到Guest物理地址转换能力
• AMD64长模式页表兼容性
• PCI ATS支持
• PCI-SIG PRI和PASID TLP前缀ECN支持
• Guest虚拟APIC支持（例如AVIC）
• 增强的性能和错误日志记录特性
• Guest页表用户/特权访问权限检查
• Guest页表全局特权级访问保护
• Guest页表不可执行页保护
• 设备表分段
• PPR和事件日志双缓冲区（可选自动切换）
• PPR自动响应（带始终开启特性）
• PPR日志早期溢出警告
• 设备特定特性报告寄存器
• MMIO访问MSI设置和映射配置空间字段
• 内存访问路由和控制（MARC）
• 自动块停止标记消息处理
• Guest I/O保护
• x2APIC
• 硬件加速虚拟化IOMMU（vIOMMU）
• 安全ATS
• 安全嵌套分页

IOMMU的所有实现都支持基本功能，如设备虚拟到系统物理地址转换、中断重映射和访
问权限检查。要确定IOMMU的特定实现是否支持任何架构定义的可选特性，软件必须首
先检查IOMMU能力头[能力偏移00h]的EFRSup位（EFR）是否设置。如果EFRSup位已设
置，则支持IOMMU扩展特性寄存器[MMIO偏移0030h]。通过读取EFR，软件可以确定该
系统中特定IOMMU支持哪些可选特性。在大多数情况下，对特性的支持由EFR中相应字
段的非零值指示。

EFRSup位以及许多与I/O虚拟化相关的特性和能力报告字段在IOMMU ACPI表中被复制。
通过此方法报告的信息优先于通过扩展特性寄存器和杂项信息寄存器报告的信息。参见
第5章"I/O虚拟化ACPI表"。

软件实现注意事项：软件不应依赖IOMMU扩展特性寄存器传达的特性支持信息来判断
ACPI表中也报告的任何特性，因为系统固件可以覆盖IOMMU硬件报告的功能能力。

1.3.1 Guest和主机地址空间的两级转换

IOMMU添加了一个可选的Guest地址转换层，类似于处理器的两级嵌套分页功能。分层地
址转换可以视为如图2所示的嵌套地址空间。每个地址空间有一组地址转换表。IOMMU可
以提供由Hypervisor管理的Guest物理到系统物理地址转换（有时称为"L2转换"）。设
备表项被扩展以包含可选的地址转换信息，用于由Guest操作系统管理的Guest虚拟到
Guest物理地址转换（有时称为"L1转换"）。这使得虚拟化系统中的高级计算架构成为
可能，如计算卸载、用户级I/O和加速I/O设备。IOMMU通过MMIO偏移0030h[GTSup]
指示支持两级转换。当支持时，通过编程适当的设备表项来激活两级转换。

[图2: 嵌套地址空间]

Guest地址转换表可以支持最多1048576（2^20）个并发进程作为架构限制。但是，给定
的实现可能支持更少。MMIO偏移0030h[PASmax]字段的值可用于计算支持的最大PASID。
Guest地址转换表包含Guest物理地址，且这些表使用Guest虚拟地址索引。因此，这些
表由虚拟机内的Guest操作系统管理。HV管理嵌套转换表，IOMMU硬件提供机制以保持
表同步和处理异常情况。IOMMU根据系统软件设置的控制位自动遍历地址转换表。

IOMMU可以在三种操作模式下使用——传统单级转换、Guest和嵌套转换、以及使用处理
器兼容页表的单级Guest转换：
• 对于传统操作，软件清除设备表项（DTE）中的GV位。参见第59页"设备表项格式"。
• 对于Guest和嵌套两级转换，软件检查MMIO偏移0030h[GTSup]=1。然后软件能够为两
  级转换编程设备表项。
• 对于使用处理器兼容页表的单级转换，软件将IOMMU编程为Guest和嵌套转换，但将
  DTE[Mode]=000b设置为嵌套转换。

1.3.2 增强的处理器页表兼容性

在其基本功能中，当保留字段被编程为零时，IOMMU可以与AMD64处理器共享嵌套（主
机）页表。与处理器不同，当使用缓存信息检测到访问违规时，IOMMU不会重新遍历页
表。当IOMMU在嵌套事务中检测到访问违规时（无论是TLB命中还是页表遍历（TLB未
命中）），它会阻止访问或返回具有计算出的访问权限的ATS响应。当IOMMU确定存在
适当的访问权限时，它允许请求的访问或返回具有计算出的访问权限的ATS响应。

处理器页表兼容性特性。在AMD64模式下操作时，IOMMU与AMD64长模式页表的兼容性得
到增强。IOMMU可以直接与处理器共享AMD64长模式页表用于Guest地址转换。Guest页
转换表与AMD64长模式格式和语义严格兼容，包括IOMMU对已访问位和脏位的更新（参
见第2.2.6节[Guest转换的I/O页表]和第2.2.7.4节[更新Guest地址表中的已访问位和
脏位]）。当使用Guest转换时，IOMMU遵循AMD64长模式地址转换对Guest虚拟地址的要
求，因此软件在提升Guest访问权限时不需要发出无效命令；只有当软件降低Guest访问
权限或移除Guest页（"存在到不存在"）时，软件才必须发出无效命令。因此，由TLB
条目计算的Guest权限不足的ATS请求或DMA引用可能基于过时的信息。为了确定当前权
限，IOMMU重新遍历Guest页表以使用从内存读取的信息重新计算访问权限。嵌套页表
可能作为Guest表重新遍历的结果被读取。IOMMU根据新读取的页表信息确定访问结果。
重新遍历可能需要完整遍历Guest和嵌套两级转换。详情在第2.2.7节[Guest和嵌套地
址转换]中。AMD64长模式页表在页属性表（PAT）中包含内存类型信息；IOMMU可以向
外设提供内存类型信息，但不解释或验证该信息。

1.3.3 性能特性

IOMMU提供三个面向性能的特性：性能计数器、PREFETCH命令和FLUSH_ALL命令。

1.3.3.1 性能计数器

为了向系统软件提供一致的性能监控和评估机制，定义了一组可选的性能计数器。支持
由IOMMU扩展特性寄存器[MMIO偏移0030h]的PCSup位指示。实现可以在架构定义的计数
器之外提供额外的计数器。计数器独立于处理器活动运行。计数器组织在n个计数器库
中，每个库适合一个4KB页。HV可以私有控制所有计数器库或将一个或多个计数器库分
配给Guest操作系统。计数器和计数器库的数量向系统软件报告（参见第3.4节[IOMMU
MMIO寄存器]和第5.2.2.1节[I/O虚拟化硬件定义（IVHD）块]）。每个计数器库有过
滤感兴趣的设备和事件源的控制。每个事件计数器被编程为计数事件或事件持续时间，
每个计数器寄存器有一个用于阈值目的的可选信号（参见第2.5.11节
[EVENT_COUNTER_ZERO事件]）。

当支持性能计数器时（MMIO偏移0030h[PCSup]=1），软件必须为IOMMU MMIO寄存器分配
一个512KB的连续系统内存区域。该区域必须4KB对齐。如果IOMMU不支持性能计数器
（MMIO偏移0030h[PCSup]=0），分配需求降为16KB。

1.3.3.2 加载IOMMU TLB

可选的PREFETCH_IOMMU_PAGES命令使系统软件能够将相关的转换信息加载到IOMMU TLB
中（参见第2.4.6节[PREFETCH_IOMMU_PAGES]），特别是错误处理信息（第2.4.6.1
节[PREFETCH_IOMMU_PAGES的事件处理]）。

对预取特性的支持由MMIO偏移0030h[PreFSup]指示。如果PreFSup=0，
PREFETCH_IOMMU_PAGES命令会导致IOMMU创建一个错误事件（第2.5.6节
[ILLEGAL_COMMAND_ERROR事件]）。因为TLB是缓存结构，预取命令必须被视为建议性的。
即使IOMMU为每个预取命令获取地址转换信息，TLB条目也可能在被使用之前被其他转换
信息覆盖，尝试使用该转换信息将导致页表遍历。

PREFETCH_IOMMU_PAGES命令是对IOMMU的提示，表明关联的转换记录将在相对较短的
时间内需要，IOMMU应执行页表遍历以加载转换信息。根据内部状态和工作负载，IOMMU
可能将转换信息获取到TLB中。如果条目已在TLB中，IOMMU可以调整LRU或其他控制标
记以延长缓存驻留时间。

1.3.3.3 刷新IOMMU TLB

IOMMU的基本功能提供INVALIDATE_DEVTAB_ENTRY命令（按DeviceID）、
INVALIDATE_IOMMU_PAGES命令（按DomainID）和INVALIDATE_INTERRUPT_TABLE命令
（按DeviceID），软件可以使用它们来无效I/O TLB条目。

可选的INVALIDATE_IOMMU_ALL命令可以简化可信启动、错误恢复和从低功耗状态恢
复（参见第2.4.8节[INVALIDATE_IOMMU_ALL]）。在INVALIDATE_IOMMU_ALL命令完
成后，所有IOMMU TLB都被无效，包括设备表、Guest CR3表、页目录项、页表项和
中断重映射条目（包括Guest APIC表根指针）的缓存部分。第2.4.11节[IOMMU排序
规则]描述了如何处理未完成的操作。

IOMMU的操作状态不受INVALIDATE_IOMMU_ALL影响。转换、命令和事件处理、地址转
换请求和外设页请求处理正常继续。MMIO寄存器的内容不受影响，除了将命令缓冲区
头指针寄存器[MMIO偏移2000h]推进到INVALIDATE_IOMMU_ALL命令之后。IOMMU可以
在INVALIDATE_IOMMU_ALL命令完成后的任何时间开始用信息重新加载内部缓存。
INVALIDATE_IOMMU_ALL命令保证第2.4.11节[IOMMU排序规则]中描述的排序。

注意，INVALIDATE_IOMMU_ALL命令不会无效远程IOTLB。对于ATS，可以通过在每
个PCI设备/功能上禁用并重新启用ATS来实现无效。

对INVALIDATE_IOMMU_ALL命令的支持由IOMMU扩展特性寄存器[MMIO偏移0030h]的
IASup位指示。

1.3.4 Guest虚拟地址的地址转换服务

基本功能。地址转换服务可以被外设用来将GPA转换为SPA。要将GPA转换为SPA，
PCIe®连接的外设发出缺少IOMMU识别的PASID TLP前缀的ATS请求（参见第2.2.7.7
节[PCIe® TLP PASID前缀]）。IOMMU使用缓存信息评估访问权限，并在需要时遍历
页表。结果访问权限在ATS响应中返回。

可选增强。地址转换服务可以被外设用来将GVA或GPA转换为SPA。要将GVA转换为SPA，
外设发出ATS协议请求。请求包含有效的PASID、访问属性标志和（规范的）虚拟地址
（参见第1.3.6节[在内存事务中选择转换表]和第2.2.7.7节[PCIe® TLP PASID前缀]）。
集成外设可以使用ATS协议以外的方式（如线信号）来呈现标志和虚拟地址。IOMMU使
用缓存信息评估访问权限以提高效率，并在需要时遍历页表。为了匹配AMD64语义，
如果先前缓存的信息指示访问权限不足，IOMMU必须重新遍历Guest页表（参见第
2.2.7.1节[组合Guest和主机地址转换]和第111页的表32）。结果访问权限在ATS响应
中返回。为了携带Guest地址的附加信息，IOMMU使用包含有效PASID的PCIe TLP前缀。

IOMMU必须在处理ATS请求时更新GVA页表中的已访问位和脏位，就好像外设实际上已
经访问了内存一样（参见第2.2.7.4节[更新Guest地址表中的已访问位和脏位]）。为
了评估GVA已访问位和脏位，IOMMU必须使用ATS数据包中指示的访问级别。只读访问
的ATS请求确定已访问位的设置，读写访问的ATS请求确定脏位的设置（参见第111页
的表32）。当处理GPA时，IOMMU将页表视为只读。

软件注意事项：当软件将页表项中的A或D位从1更改为0时，必须发出无效命令。此
要求允许IOMMU将A和D位缓存在TLB中以获得更高性能。

软件发出INVALIDATE_IOTLB_PAGES命令以使IOMMU生成对外设的无效请求（参见第
2.4.4节[INVALIDATE_IOTLB_PAGES]）。当内容为GPA时，发送到设备的无效请求缺
少有效的PASID前缀。当内容为GVA且PASID在命令的PASID TLP前缀字段中时，发送
到下游设备的无效请求具有有效的PASID前缀。

具有IOTLB的外设必须无效导致权限不足检查的缓存转换条目并使用ATS获取新鲜转
换的条件在第2.1.4.8节[丢弃设备IOTLB信息以重新遍历页表]中描述。

1.3.5 与PCI-SIG PRI兼容的外设页请求支持

IOMMU可选地支持PCI-SIG PRI规范，作为PCI-SIG地址转换服务（ATS）规范的补充
（参见第2.1.1节[正常操作]）。IOMMU对PRI的支持称为外设页请求（PPR）日志记
录（参见第2.6节[外设页请求（PPR）日志记录]）。

操作系统通常需要固定用于I/O的内存页；固定的页面通常从容量有限的单独内存池中
分配。ATS和PRI可以一起使用，使外设能够使用未固定的页面进行I/O。当处理ATS请
求时，IOMMU在检测到访问权限不足或不存在的页面时不发出事件信号；而是返回从页
表计算的权限。外设检查响应以确定适当的操作（例如，使用PRI请求系统软件处理页
表项）。使用PRI/PPR允许外设请求操作系统更改页面的访问权限或请求将页面加载到
内存中（它不存在）。将ATS与PRI/PPR一起使用可以允许系统在可用内存不足的情况
下高效运行。

1.3.6 在内存事务中选择转换表

在IOMMU的基本功能中，PCIe数据包包含GPA，并且使用原始BDF选择GPA到SPA的转换
表。PCI-SIG TLP前缀不被IOMMU解释。

可选特性增加了对使用Guest操作系统编程的页表将Guest虚拟地址转换为系统物理地
址的支持。PCI-SIG定义了一种向事务添加信息的方法，称为TLP前缀。当PCIe事务具
有PASID TLP前缀时，数据包包含规范GVA，TLP前缀选择用于GVA到GPA转换的Guest
表；当PCIe事务没有TLP前缀时，数据包包含GPA。原始BDF用于选择GPA到SPA的转换
表。详情在第2.2.7.7节[PCIe® TLP PASID前缀]中。

1.3.7 AMD64中断虚拟化（Guest虚拟APIC中断控制器）

IOMMU可选地支持中断虚拟化。当中断被虚拟化时，设备中断可以直接传递给运行中的
Guest虚拟机而无需Hypervisor干预（参见MMIO偏移0030h[GASup]和MMIO偏移
0018h[GAEn]）。这可以减少Guest VM中断的传递延迟和开销。此特性需要处理器中兼
容的APIC虚拟化支持。处理器和IOMMU在传递中断时协调维护Guest虚拟APIC表中的中
断状态。中断重映射和中断虚拟化可以独立启用。详情参见第2.2.8节[用于中断虚拟化
的Guest虚拟APIC表]。

1.3.8 增强的访问位和脏位支持

页描述符中的访问位（A）指示描述符指向的物理页是否已被访问。页描述符中的脏位
（D）指示该描述符指向的页转换表或物理页是否已被外设写入。

Guest页表中的A位在描述符或物理页首次被读取或写入时由IOMMU设置为1。Guest页表
中的D位在设备首次写入物理页或为预期写入对应物理页而访问描述符时由IOMMU设置
为1。

可选地，IOMMU硬件可以在主机页表中设置A和D位（参见IOMMU扩展特性寄存器[MMIO
偏移0030h]的HASup和HDSup）。当启用时，IOMMU将以类似于第2.2.7.4节中描述的过
程在主机页表中设置A和D位。

注意IOMMU硬件从不清除A或D位。

系统软件可以使用A位帮助决定选择候选页面进行换出。D位可用于避免对存储的冗余
回写。

1.3.9 Guest I/O保护

可选地，IOMMU启用Guest I/O保护，即使对于不支持PASID的设备也是如此[参见第
2.2.9节]。当启用时，对于与启用的DTE关联的不带PASID的DMA请求，分配默认PASID
等于0。然后DMA请求像往常一样经过Guest和嵌套转换。操作系统可以使用此特性为可
能不支持PASID的设备使用Guest页表启用I/O保护缓冲区。

1.3.10 SMI过滤器

IOMMU可选地支持拦截系统固件或软件未预期的系统管理中断（SMI）。对系统运行和
完整性至关重要的SMI照常传递，但来自可疑来源的SMI可以被阻止或延迟以供后续处
理或分析。SMI过滤器操作的详情参见第2.1.5节[系统管理中断（SMI）控制]。

1.3.11 硬件错误寄存器

所有错误事件都在IOMMU事件日志中报告。可选地，通过将关键事件记录在硬件错误寄
存器中来增强错误报告。参见IOMMU硬件事件上部寄存器[MMIO偏移0040h]、IOMMU硬
件事件下部寄存器[MMIO偏移0048h]和IOMMU硬件事件状态寄存器[MMIO偏移0050h]。

1.3.12 硬件加速虚拟化IOMMU（vIOMMU）

IOMMU可以可选地支持vIOMMU特性，该特性在实现Guest IOMMU时提供部分硬件加速。
Guest IOMMU是暴露给VM的IOMMU。vIOMMU在VM访问其命令缓冲区、PPR日志和事件日
志时提供加速。这消除了支持HV拦截所需的CPU开销，并降低了这些操作的延迟。

HV继续模拟IOMMU配置空间寄存器和IOMMU MMIO寄存器的子集，这些寄存器主要在初
始化期间使用。此外，HV必须初始化vIOMMU特性，映射VM和IOMMU之间的MMIO资源，
管理内存中的额外支持数据结构，并为IOMMU分配后备存储内存。

先前定义的两级转换特性使IOMMU能够遍历VM创建的Guest页表数据结构。

在vIOMMU下，IOMMU暴露可用于实现每个VM的Guest IOMMU MMIO寄存器空间的一部分的
MMIO资源。当VM尝试访问偏移在8KB到12KB之间的Guest IOMMU MMIO寄存器（如命令缓
冲区、事件日志和PPR日志头尾指针寄存器）时，这由IOMMU直接处理。当IOMMU访问VM
命令缓冲区、PPR日志或COMPLETION_WAIT存储位置在内存中时，它直接访问VM Guest物
理内存。IOMMU根据需要在VM编码和对应的HV编码之间转换字段，如Device ID和
Domain ID。最后，IOMMU可以直接向VM IOMMU驱动程序发出中断，例如对于
COMPLETION_WAIT命令或由于写入PPR日志。

vIOMMU架构设计为最多支持64K个同时VM。HV可以在某些VM中启用vIOMMU，同时向其
他VM提供完全模拟的IOMMU。更多详情参见第2.10节[vIOMMU]。

1.3.13 安全嵌套分页（SEV-SNP）

SEV-SNP特性提供额外的内存安全检查，以更好地隔离Guest内存。它利用单独的反向
映射表（RMP）数据结构，该结构提供与每个SPA页关联的信息。

更多详情参见第188页"IOMMU安全嵌套分页（SEV-SNP）支持"。

有关SEV-SNP的更多详情，请参阅APM2和SNP-WP。


================================================================================
第2章  架构
================================================================================

本章主要从系统软件的角度描述IOMMU的架构。讨论从IOMMU设置完成后的正常稳态行
为开始，重点介绍IOMMU如何处理各种设备事务。随后的章节描述了用于控制IOMMU的内
存数据结构，以及软件正确更新这些（共享）数据结构所必须遵循的程序。最后，本章
以对系统启动时必须初始化的PCI资源的描述结束，以配置IOMMU。

--------------------------------------------------------------------------------
2.1 行为
--------------------------------------------------------------------------------

当IOMMU被禁用时，它只是简单地让所有总线流量通过而不做任何更改。

当IOMMU被启用时，它拦截从下游设备到达的请求（这些设备可能通过数据结构、PCIe
链路或其他方式连接到系统），对请求执行权限检查和地址转换，并将转换后的版本向
上游发送到系统内存。其他请求不经更改地通过（详情在第2.1.1节[正常操作]中）。
由单个IOMMU服务的PCI设备必须在同一PCI段组上（更多PCI段组详情参见PCI固件规范）。

IOMMU读取系统内存中的三个表来执行其权限检查、中断重映射和地址转换。为避免死
锁，IOMMU对设备表、页表和中断重映射表的内存访问使用等时虚拟通道，并且只能引
用系统内存中的地址。IOMMU对命令缓冲区、事件日志条目和可选请求队列条目的其他
内存读取使用普通虚拟通道。如果IOMMU对它处理的每个设备请求都执行完整的表查找
过程，系统性能可能会大幅降低。因此，IOMMU的实现预计维护IOMMU内存表内容的内
部缓存，IOMMU的正确操作要求系统软件在更新可能已被IOMMU缓存的表项时向IOMMU
发送适当的无效命令。

IOMMU使用普通虚拟通道写入系统内存中的事件日志。IOMMU可以可选地写入系统内存
中的外设页请求日志，这些写入使用普通虚拟通道。IOMMU可以可选地写入系统内存中
的Guest虚拟APIC日志，这些写入使用普通虚拟通道。

IOMMU使用标准PCI MSI或MSI-X中断来发出信号中断。

2.1.1 正常操作

通过IOMMU的典型请求流如下：
• IOMMU生成的读、写和中断事务不被IOMMU转换。
• 从上游到达的事务必须不经更改地传递到下游。
• 从下游到达的响应、栅栏或刷新命令的事务必须不经更改地传递到上游。
• 从下游到达的引用IOMMU排除范围内地址的事务必须不经更改地传递到上游。
• 来自下游的内存读写事务导致在设备表中进行表查找以获取请求I/O设备的DomainID
  并定位I/O页表。需要在I/O页表中进一步查找以执行地址转换和权限检查。执行权限
  检查和地址转换后，如果事务被允许从I/O设备发出，IOMMU将结果事务转发到上游。
• 来自下游的地址转换请求导致与内存读写事务相同的表查找。转换后的地址和访问权
  限信息返回给请求的外设。软件需要无效外设缓存的地址转换映射。
• 来自下游的外设页请求在不支持时导致事件日志条目，或导致写入系统内存的外设页
  请求日志条目。（可选的PPR自动响应修改此行为。）
• 中断地址永远不会被转换为系统内存地址，但为消息信号中断（MSI）保留的物理地
  址范围可以可选地被视为用于转换的内存地址（例如，ACPI地址范围、PCI配置空间
  映射）。
• 上游中断导致在设备表中查找，然后在中断重映射表中重映射中断。执行检查和中断
  重映射后，如果中断被允许从I/O设备发出，IOMMU将结果中断转发到上游。SMI请求
  可选地通过SMI过滤器，不使用中断重映射表。
• 来自下游设备的端口I/O空间事务导致设备表查找，以确定I/O设备是否被允许访问端
  口I/O空间。
• IOMMU在系统内存中维护一个事件日志，包含未正常完成的事务的详细信息。
• IOMMU维护一个可选的Guest虚拟APIC日志，包含在Guest未运行时到达的中断请求
  的详细信息。
• 如果I/O设备被标记为能够生成预转换地址，IOMMU不会进一步转换来自该设备的预
  转换内存读写请求。
• IOMMU处理来自命令队列的命令。

可选的MARC特性允许来自集成I/O设备（如GPU）的访问在访问系统内存的定义区域时
绕过IOMMU。

除了传递来自下游设备的事务外，IOMMU还插入自己的事务以执行对系统内存的读写
和发出信号中断。

IOMMU允许缓存页表和设备表内容以加速转换。定义了一个无效协议，以便软件在更
新表时可以保持缓存内容与内存一致。当软件启动不保留处理器或芯片组状态的挂起
操作时，存储在寄存器中的IOMMU状态将丢失，必须作为恢复序列的一部分进行恢复。

当系统软件处理PCI热插拔通知时，应检查ACPI表以确定将为外设提供服务的IOMMU，
然后适当地编程IOMMU。

IOMMU看到的事务以数据结构事务的形式定义。对系统定义的地址范围

2.1.3 IOMMU事件报告

IOMMU必须检测并可能报告由于异常硬件或软件行为而产生的几类事件。当IOMMU检测
到任何类型的事件且事件日志记录被启用时，它将适当的事件条目写入系统内存中的事
件日志。此外，它可以在事件日志被写入时可选地发出信号中断。

IOMMU检测的事件包括I/O缺页以及遍历I/O页表时检测到的硬件内存错误。检测到的
事件可能导致页表或中断重映射表遍历在到达最终内存转换或中断重映射条目之前终
止。当遍历提前终止时，报告的事件信息基于遍历已完成部分计算的结果，从设备表
项（DTE）开始。

软件注意事项：IOMMU对导致事件的条目的TLB缓存行为未定义；某些实现可能在验证
条目不会导致异常之前就将其插入TLB缓存。系统软件应无效导致事件的地址。

2.1.3.1 IOMMU事件响应

IOMMU对事件的响应取决于检测到的事件类型、导致事件的事务类型以及事件发生时
IOMMU的状态。

如果IOMMU未启用或不支持地址转换请求，IOMMU以主中止响应转换请求。

如果IOMMU已启用，它可以有三种事件响应之一：
• 对于被主中止或目标中止的上游事务，与IOMMU共置的PCI/主机桥是事务的完成者。
  被目标中止的事务以与接收事务的总线规范一致的方式设置传统Signaled Target
  Abort位（辅助端口）。这些中止的事务不应设置任何AER位（如果已实现且适用）。
• 在以IOMMU功能为目标的事务中检测到的异常不记录在IOMMU事件日志中。异常按照
  与IOMMU功能关联的主总线适用的总线规范规则发出信号。
• 在源自IOMMU功能的事务中检测到的异常按照与IOMMU关联的主总线适用的总线规范
  规则发出信号事件。此外，命令缓冲区和表遍历读取中的异常记录在IOMMU事件日
  志中。

尝试使用超出表末尾的设备表项的事务按表44处理。设备表的大小由设备表基地址寄
存器MMIO偏移0000h[Size]定义。

2.1.3.2 I/O缺页

IOMMU在处理外设请求时可能检测到缺页条件，IOMMU的响应取决于请求类型和IOMMU
控制设置。

外设的内存事务可能导致I/O缺页。这些缺页可能由多种原因产生，如I/O页表项缺
少足够的权限或内存页标记为不存在。在传统处理器虚拟内存实现中，缺页激活一个
异常处理程序，该处理程序可以选择纠正底层问题并重试出错的指令。IOMMU没有这
样的选项：底层数据结构和PCIe协议不提供IOMMU向设备发出信号以重试访问的方式。
因此，当IOMMU检测到I/O缺页时，它目标中止出错的请求。IOMMU设置传统PCI
Signaled Target Abort位（如果适用），并在事件日志记录启用时将I/O缺页信息
记录在其事件日志中。

对于地址转换请求，IOMMU返回转换结果而不发出信号故障（另见第2.1.4.5节[特殊
地址范围中的地址转换请求]）。外设可以检查转换响应以确定特定内存事务是否会导
致异常。外设可以按第2.6节[外设页请求（PPR）日志记录]中的描述请求缺页服务。

2.1.3.3 内存访问错误

IOMMU自身对其内存表的内存访问本身可能导致几类错误，包括：
• 访问不存在或非DRAM地址，因为IOMMU的等时虚拟通道仅限于DRAM地址。
• 不可纠正的ECC错误。
• 使用保留值，包括设备表项中无效或不支持的类型代码以及页表项中的保留位。

IOMMU在事件日志记录启用时将所有检测到的内存访问错误记录在其事件日志中。可
选地，硬件错误也可以存储在错误寄存器中（参见第2.5.16.2节[I/O硬件事件报告
寄存器]）。

2.1.4 特殊条件

本节定义了IOMMU在特定操作条件下的行为。

2.1.4.1 零字节读操作

在某些总线架构中，零字节读操作被定义为具有明确副作用的特殊操作。由于这些副
作用，当页面被标记为允许读或写访问时，IOMMU必须允许零字节读操作。此外，由于
零字节读操作在某些总线规范中返回未定义的数据，保护不可读内存位置的内容要求
IOMMU对零字节读操作混淆返回的数据。

实现注意事项：混淆零字节读操作中返回数据的方法包括返回常量、随机值或不基于
数据内容的可预测值（如地址）。

2.1.4.2 中断地址范围

对中断地址范围（表3）的访问被定义为通过IOMMU的中断重映射部分，而不通过地址
转换处理。因此，当事务作为中断重映射操作处理时，预转换或未转换的事务属性被
忽略。

软件注意事项：不应将IOMMU配置为使地址转换结果为特殊地址（如中断地址范围）。

2.1.4.3 缺少PDE的多页地址转换请求

对IOMMU的地址转换事务可以请求多个页面。页面大小（步幅）通常由level=0或
level=7的PDE确定。页面步幅始终是2的幂。对于没有相关PDE的情况（在IOMMU排除
范围内或DTE[Mode]=0时），IOMMU返回的结果是实现特定的。

2.1.4.4 IOMMU排除范围中的地址转换请求

I/O设备可以请求IOMMU排除范围中地址的地址转换，该范围由IOMMU排除基址寄存器/
完成存储基址寄存器[MMIO偏移0020h]和IOMMU排除范围限制寄存器/完成存储限制寄
存器[MMIO偏移0028h]定义，并可以缓存结果。当软件更改排除范围时，它必须无效
可能包含受影响转换条目的远程IOTLB。对排除范围的地址转换请求始终返回允许读
写的权限。

排除范围内GPA的地址转换请求返回实现定义的结果。

2.1.4.5 特殊地址范围中的地址转换请求

对表3中特殊地址范围内GPA的I/O设备地址转换请求受设备表项中SysMgt和IoCtl设
置的控制（参见第2.2.2.1节[设备表项格式]），可以返回转换或导致目标中止。

2.1.4.6 跨越内存和特殊地址范围的页转换条目

GPA的IOMMU地址转换条目可以构建为同时覆盖常规内存地址和特殊地址（参见表3）。
DTE[IoCtl]和DTE[SysMgt]字段控制IOMMU行为。要转换特殊地址范围中的GPA地址，
在DTE中设置对应的特殊地址范围控制以指示IOMMU将所需的特殊地址范围作为内存
地址转换。

2.1.4.7 丢弃IOMMU TLB信息以重新遍历页表

可选特性增加了IOMMU在某些条件下重新遍历页表的能力。当IOMMU基于缓存信息检
测到访问违规时，它丢弃IOMMU TLB中的信息并从内存重新加载转换信息。中断重映
射信息仅在TLB未命中时从内存加载。详情参见第1.3.4节[Guest虚拟地址的地址转
换服务]。

2.1.4.8 丢弃设备IOTLB信息以重新遍历页表

可选特性增加了IOMMU在某些条件下重新遍历页表的能力。外设可以使用来自IOTLB
或通过ATS获得的地址转换信息来确定嵌套（主机）访问的访问权限。作为AMD扩展，
当IOTLB条目中R=1或W=1用于Guest访问时，带有IOTLB的外设必须无效导致权限不
足失败的缓存条目。然后外设必须使用ATS请求Guest转换信息并重试访问。如果修
订后的权限对重试仍不足，外设必须采取适当行动以放弃访问或发出PCIe® PRI请
求以获得更高权限。

2.1.4.9 更新Guest页表中的已访问位和脏位

可选特性增加了IOMMU写入Guest页表的能力。IOMMU必须原子地更新Guest页表的已
访问位和脏位。IOMMU从不清除已访问位或脏位；软件负责清除这些位。IOMMU被允
许缓存这些位，因此软件在清除PTE中的这些位时必须发出无效命令。详情参见第
2.2.7.4节[更新Guest地址表中的已访问位和脏位]和第2.2.7.5节[清除已访问位和
脏位]。

2.1.4.10 DTE[Mode]=0时的地址转换响应

当DTE[Mode]=000b时，外设可以请求地址转换；返回的系统物理地址（SPA）等同于
Guest物理地址。

2.1.4.11 页分裂

当地址由不同页大小的Guest和嵌套页表项映射时，创建的IOMMU TLB条目匹配较小
页的大小（另见《AMD64技术，AMD64架构程序员手册，卷2：系统编程》，页分裂）。

2.1.4.12 原子操作需要读写权限

原子操作同时读写一个页面。IOMMU必须仅在页面被标记为允许读和写操作时才允许
来自外设的原子操作。

2.1.4.13 INVALIDATE_IOTLB_PAGES和外设复位

如果在IOMMU执行INVALIDATE_IOTLB_PAGES命令（第2.4.4节
[INVALIDATE_IOTLB_PAGES]）期间外设被复位，外设可能停止处理无效操作，软件
必须处理由此产生的任何IOTLB_INV_TIMEOUT事件（第2.5.8节
[IOTLB_INV_TIMEOUT事件]）。

2.1.5 系统管理中断（SMI）控制

为了确保传递给处理器的AMD64系统管理中断来自有效的外设来源（DeviceID），
IOMMU可选地支持SMI过滤器。当MMIO偏移0030h[SmiFSup]=00b或MMIO偏移
0018h[SmiFEn]=0b时，来自任何来源的SMI中断被传递给处理器；对于MMIO偏移
0030h[SmiFSup]的其他值，SMI中断按SMI过滤器寄存器（参见IOMMU SMI过滤器寄
存器[MMIO偏移00[60-D8]h]）中编程的值进行过滤。可用的SMI过滤器寄存器数量
在MMIO偏移0030h[SmiFRC]中。每个SMI过滤器寄存器包含一个DeviceID和控制位；
SMI过滤器寄存器集合可以被编程以定义将从中传递系统管理中断的设备集。

2.1.5.1 SMI过滤器操作

当SMI过滤被启用且IOMMU支持时，每个上游SMI都被检查是否与有效且启用的SMI过
滤器寄存器匹配，来自未能匹配任何SMI过滤器寄存器的DeviceID的上游SMI将被阻
止。MMIO偏移0018h[SmiFEn]和MMIO偏移0018h[SmiFLogEn]字段控制SMI过滤器的行
为。当SmiFEn=1b时，匹配任何有效SMI过滤器寄存器的上游SMI中断请求不经修改
地向上游传递。当SmiFEn=1b且SmiFLogEn=0b时，未能匹配任何有效SMI过滤器寄存
器的上游SMI中断请求被静默丢弃（即不向上游转发）。当SmiFEn=1b且SmiFLogEn=1b
时，未能匹配任何有效SMI过滤器寄存器的上游SMI中断请求被记录在IOMMU事件日
志中，上游SMI请求被丢弃。使用的事件日志条目格式是IO_PAGE_FAULT日志缓冲区
条目（第2.5.3节[IO_PAGE_FAULT事件]），Address[63:0]字段设置为SMI中断请求
寻址的值（参见表4）。软件必须检查事件日志条目的Address字段以确定记录的中
断请求是否为尝试的SMI中断。

2.1.5.3 SMI过滤器的推荐编程

SMI过滤器寄存器设计为可由任何软件组件编程。但是，当前可用的系统软件通常不
了解SMI请求或SMI请求的有效来源，因此它没有编程SMI过滤器所需的信息。在典型
的系统硬件设计中，所有SMI请求将通过单个来源（通常是称为基板管理控制器或BMC
的组件）路由，由固件（如BIOS或UEFI）处理。因此，当硬件支持SMI过滤器时（参
见MMIO偏移0030h[SmiFSup]），应由固件编程以处理单个SMI请求来源。

推荐的SMI过滤器编程方法是：
1. 检查SMI过滤器是否支持（参见MMIO偏移0030h[SmiFSup]），
2. 从可用集合中选择一个SMI过滤器寄存器（参见MMIO偏移0030h[SmiFRC]），
3. 将选定的SMI过滤器寄存器编程为BMC的DeviceID（参见MMIO偏移00[60-D8]h
   [SmiDID]），
4. 将选定的SMI过滤器寄存器编程为有效（参见MMIO偏移00[60-D8]h[SmiDV]），
5. 将选定的SMI过滤器寄存器编程为锁定（参见MMIO偏移00[60-D8]h[SmiFLock]），
6. 将任何剩余的SMI过滤器寄存器编程为无效并锁定，以及
7. 将MMIO偏移0018h[SmiFLogEn]编程为禁用SMI日志记录。

如果处理IOMMU事件日志的系统软件了解SMI过滤器，则应将MMIO偏移
0018h[SmiFLogEn]编程为启用SMI日志记录。

更一般的SMI过滤器寄存器编程情况在下一节中讨论。

2.1.5.4 SMI过滤器寄存器的通用编程

当IOMMU被复位时，软件必须编程寄存器以使SMI过滤器生效。当MMIO偏移
0030h[SmiFSup]=01b时实现可选SMI过滤器，当软件编程MMIO偏移
0018h[SmiFEn]=1b时启用。SMI过滤器寄存器作为集合工作，IOMMU实现的SMI过滤
器寄存器数量在MMIO偏移0030h[SmiFRC]中报告（另见表77）。每个SMI过滤器寄存
器包含三个字段：SmiDV、SmiDID和SmiFLock（参见IOMMU SMI过滤器寄存器[MMIO
偏移00[60-D8]h]）。

当SmiFLock=1b时，SmiDID和SmiDV的值为只读；当SmiFLock=0b时为读写。包含
SmiDV=0b的SMI过滤器寄存器处于非活动状态（从不匹配），当SmiFLock=0b时可
通过编程SmiDV=1b来激活；整个SMI过滤器寄存器可以在一次操作中编程（即软件
可以在同一操作中设置SmiDID、SmiDV=1b和SmiFLock=1b）。软件可以通过编程
SmiDV=0b和SmiFLock=1b将SMI过滤器寄存器锁定为非活动状态。软件可以通过将
SmiDID和SmiDV编程为所需值并设置SmiFLock=1b来锁定SMI过滤器寄存器中的值。
一旦锁定，SMI过滤器寄存器只能在系统复位设置SmiFLock=0b后更改。

描述被阻止的SMI中断请求的条目可选地记录在IOMMU事件日志中（参见MMIO偏移
0018h[SmiFLogEn]）。当日志记录启用时，软件可以监控日志条目以检测是否从意
外的来源设备（DeviceID）接收到过多的SMI中断。当检测到过多的SMI中断时，软
件可以关闭日志记录以减少处理开销。在软件限制或停止意外SMI中断的来源后，软
件可以编程MMIO偏移0018h[SmiFLogEn]以恢复为被SMI过滤器阻止的SMI中断创建
事件日志通知。DTE和IRTE字段SA、SE、IG和SupIOPF不影响SMI过滤器事件的日志
记录（参见表7和表20）。

要冻结特定的SMI过滤配置，在每个实现的SMI过滤器寄存器中编程SmiFLock=1b。

不了解SMI请求或SMI请求的有效来源的系统软件没有编程SMI过滤器所需的信息，
推荐的编程程序在第2.1.5.3节[SMI过滤器的推荐编程]中描述。

在更一般的系统设计中，SMI请求可能来自不限于基板管理控制器（BMC）的多个来
源。当硬件支持SMI过滤器时（参见MMIO偏移0030h[SmiFSup]），BIOS或UEFI可以
编程SMI过滤器以允许多个SMI请求来源。此情况的程序是：
1. 检查SMI过滤器是否支持（参见MMIO偏移0030h[SmiFSup]），
2. 对于每个预期的SMI请求来源：
   • 从可用集合中选择一个SMI过滤器寄存器（参见MMIO偏移0030h[SmiFRC]），
   • 将选定的SMI过滤器寄存器编程为外设的DeviceID（参见MMIO偏移00[60-D8]h
     [SmiDID]），
   • 将选定的SMI过滤器寄存器编程为有效（参见MMIO偏移00[60-D8]h[SmiDV]），
   • 将选定的SMI过滤器寄存器编程为锁定（参见MMIO偏移00[60-D8]h[SmiFLock]），
3. 保留一个或多个剩余的SMI过滤器寄存器，不编程且不锁定，供系统软件使用，
4. 将未保留给系统软件使用的剩余SMI过滤器寄存器编程为无效并锁定，
5. 将MMIO偏移0018h[SmiFLogEn]编程为启用SMI日志记录，以便系统软件了解被
   SMI过滤器阻止的SMI请求。

在此配置中，系统软件可以编程未锁定的SMI过滤器寄存器以允许来自其他外设的
SMI请求。软件应注意，一旦SMI过滤器寄存器被锁定，在系统复位之前无法重新
编程。系统软件和固件需要使用本文档范围之外的方法协调SMI过滤器寄存器的使
用。

2.2 数据结构
--------------------------------------------------------------------------------

主机软件必须维护最多八个内存数据结构供IOMMU使用。这些数据结构是：

1. 设备表（Device Table）是以DeviceID为索引的表。每个设备表项包含模式位、指
   向I/O页表的指针、指向中断重映射控制表的指针、一组控制位和一个16位
   DomainID。DomainID作为地址空间标识符，允许共享相同I/O页表的多个设备在
   IOMMU上共享相同的转换缓存资源。共享DomainID的所有设备的页表必须相同。

2. I/O页表：每个设备表项可以指定不同的I/O页表，或不同的设备表项可以共享相同
   的I/O页表。每次IOMMU处理设备对内存的访问时，它在其转换缓存和/或适当的I/O
   页表中查找设备虚拟地址（DVA），以确定设备是否有权限以及（如果允许的话）要
   访问的系统物理地址。

3. 命令缓冲区：IOMMU通过系统内存中的环形缓冲区接受处理器排队的命令。

4. 事件日志：IOMMU通过系统内存中的另一个环形缓冲区——事件日志——向处理器报
   告异常事件。

5. 中断重映射表：每个设备表项可以指定一个中断重映射表。每次IOMMU处理设备中
   断请求时，它查找IRTE以将中断重映射到具有转换后向量的目的地。（参见第87页
   图15和第87页表20）

6. 外设页请求日志：IOMMU可以接受来自PRI能力外设的页更改请求服务。这些请求
   在系统内存中的环形缓冲区——PPR日志中报告。

7. Guest虚拟APIC表：IOMMU可以更新Guest中断请求状态。

8. Guest虚拟APIC日志：IOMMU可以报告发送给未运行的Guest的Guest虚拟中断。

[图4: IOMMU数据结构]
图4说明了IOMMU数据结构之间的关系。

IOMMU的基本功能支持用于地址转换和中断重映射的单级转换表。事件日志是IOMMU
写入的系统内存中唯一的数据结构。虚拟地址（GPA）的最大大小在能力偏移
10h[VAsize]中定义，物理地址（SPA）的最大大小在能力偏移10h[PAsize]中定义。

IOMMU可选地支持单级和两级转换表（表5），以及Guest APIC虚拟化、硬件错误
寄存器、性能计数器寄存器、外设页请求服务、x2APIC和SMI过滤器。IOMMU可以
写入事件日志、外设页请求日志、Guest虚拟APIC表、Guest虚拟APIC日志和Guest
页表。Guest虚拟地址（GVA）的最大大小在能力偏移10h[GVAsize]中定义。

2.2.1 更新共享表

I/O页表结构的设计使得它们可以在处理器和IOMMU之间共享。表结构（中断重映射表、
设备表和主机I/O页表）可以在IOMMU之间共享。

Guest I/O页表结构与AMD64长模式页表格式直接兼容。IOMMU访问可以可选地更新这
些表，使其可以与处理器共享。

共享表对系统软件的正确更新有要求。

更新表项时，鼓励系统软件使用对齐的64位访问，尽管定义了控制位允许系统软件在
更新表时使用字节访问。软件应保持页表内容处于一致状态。

每个表的内容也可能被IOMMU或外设IOTLB缓存。因此，在更新可能被缓存的表项后，
系统软件必须向IOMMU发送适当的无效命令。外设IOTLB中的信息也必须被无效。

IOMMU可选地支持硬件更新Guest页表中的已访问位和脏位。IOMMU被允许缓存这些位，
因此软件在清除内存中的这些位时必须发出无效命令。

发起事务的I/O设备由16位DeviceID标识。DeviceID的派生取决于结构（fabric）；
例如，图5显示了PCIe和PCI-X® RequesterID如何映射到IOMMU DeviceID，图6显示
了数据结构UnitID如何映射到IOMMU DeviceID。

阅平台文档。

[图5: 从外设RequesterID派生的DeviceID示例]

分配给Bus、Device和Function字段的位数根据PCI配置中的设置而变化。显示的分

设备表表示为256位条目的数组，DeviceID用作数组的索引。由于有65536（64K）个

设备表基地址寄存器[MMIO偏移0000h]控制设备表的系统物理地址和大小。设备表必
须在系统内存中的4KB边界上对齐，长度必须是4KB的倍数。

IOMMU在两个128位事务（由有效性指示器的范围定义）或单个256位事务中读取整个
设备表项。

当IOMMU启用时，DeviceID超出设备表末尾的任何I/O设备被拒绝I/O权限（IOMMU目
标中止该访问），当事件日志记录启用时，此类I/O设备的所有尝试访问都被记录。
当支持PPR日志记录时，PRI请求不使用设备表进行验证，因此当启用页请求时（参见
MMIO偏移0018h[PPREn]），IOMMU可能为DeviceID超出设备表末尾的I/O设备创建PPR
日志条目，因此软件必须在PPR处理中验证DeviceID。

如果I/O设备使用PCI幻象功能，软件必须复制设备表项，使得索引计算为I/O设备使
用的任何幻象功能检索到正确的条目。

设备表分段是第2.2.2.3节[设备表分段]（第74页）中描述的可选特性。

2.2.2.2 修改设备表项

本节包含更改IOMMU表的软件信息。在对表进行某些类型的更改后，软件应发出无效
命令，并注意I/O设备访问既不会被IOMMU排队也不会被限流。软件可以独立于设备
表项中的地址转换信息更改中断重映射信息。这些操作序列是通用的，系统条件可
能允许优化。在无效命令发出后，设备表项更改可能影响Guest操作系统I/O行为。

软件可以通过单次64位写入更改设备表项中的中断重映射信息。当更改前IV=1b或V=1b
时，更改后必须跟随INVALIDATE_DEVTAB_ENTRY命令。如果无法使用64位操作，软件
可以按照更改前相关设备表项中IV的值，以以下方式更改设备表项中的中断重映射信息：

• 如果更改前IV=0b，可以按任意顺序进行更改，只要最后的更改是设置IV=1b；当更
  改前V=1b时需要INVALIDATE_DEVTAB_ENTRY命令。
• 如果更改前IV=1b，可以按以下步骤更改固定和仲裁中断的中断重映射信息：
  - 在设备表项中设置IntCtl=00b以阻止中断；任何设备发起的域中断被目标中止，
    并在启用时记录到事件日志。
  - 更新中断表根指针、IG和IntTabLen。
  - 如果中断表根指针或IntTabLen被更改，则无效中断表（参见第2.4.5节
    [INVALIDATE_INTERRUPT_TABLE]）。
  - 更改IntCtl以停止阻止来自设备的中断（设置IntCtl=01b或10b）。
  - 无效设备表项（参见第2.4.2节[INVALIDATE_DEVTAB_ENTRY]）。

软件可以通过单次128位写操作更改设备表项中的地址转换信息，当更改前IV=1b或V=1b
时紧跟INVALIDATE_DEVTAB_ENTRY命令。如果无法使用128位操作，软件可以根据更改
前V和TV的值以以下方式更改地址转换信息：

• 如果更改前V=0b，可以按任意顺序进行地址转换更改，只要最后的更改是设置V=1b。
  如果更改前IV=1b，则需要INVALIDATE_DEVTAB_ENTRY命令。
• 如果更改前V=1b，软件可以使用以下步骤将IOMMU设置为不经转换地传递地址，访
  问由IR和IW控制。

IOMMU可选地支持硬件更新页表中的已访问位和脏位。IOMMU被允许缓存这些位，因此
软件在清除内存中的A或D位时必须发出无效命令。

2.2.2.3 设备表分段

设备表分段是一个可选特性，允许将设备表分为2、4或8个可独立定位在系统物理地址
空间中的较小表。将表分为较小分配块的能力使得更容易将表放入系统内存中，并且
在DeviceID空间稀疏填充的情况下可以减少设备表所需的总内存空间。

IOMMU扩展特性寄存器[MMIO偏移0030h]的两位字段DevTblSegSup指示给定实现支持
的段数。当DevTblSegSup=00b时，不支持设备表分段，设备表基地址寄存器[MMIO偏
移0000h]控制单个统一设备表的位置和大小。

当DevTblSegSup > 00b时，设备表基地址寄存器[MMIO偏移0000h]定义第一个（n=0）
段的位置和大小，并支持额外的设备表基地址寄存器。架构定义了七个设备表段n基地
址寄存器，位于MMIO偏移100-130h处。

启用的段数由IOMMU控制寄存器[MMIO偏移0018h]的3位字段DevTblSegEn控制。

表13: 用于分配设备表段的寄存器

DevTblSegEn    活动段数    段号    使用的设备表基地址寄存器（MMIO偏移）
─────────────────────────────────────────────────────────────────────────
000b           1           0       设备表基地址寄存器（0000h）
001b           2           0       设备表基地址寄存器（0000h）
                           1       设备表段1基地址寄存器（0100h）
010b           4           0-3     0000h, 0100h, 0108h, 0110h
011b           8           0-7     0000h, 0100h-0130h

DeviceID的最高1、2或3位选择用于DTE查找的表段，对应表分为2、4或8段的情况。
当启用两个段时，DeviceID的最高有效位（DeviceID[15]）选择表段，位[14:0]提供
段中的索引以找到设备的正确DTE。当启用四个段时，DeviceID[15:14]选择表段，位
[13:0]提供索引。当启用八个段时，DeviceID[15:13]选择表段，位[12:0]提供索引。

为提供DeviceID空间的完全覆盖，当启用两个段时，每个表段必须为1MB长；当启用
四个段时，每个表段必须为512KB；当启用八个段时，每个表段必须为256KB。如果
DeviceID空间稀疏填充，段可以按4KB增量缩小。

DevTblSegEn不应编程为大于DevTblSegSup值的值。

2.2.3 主机转换的I/O页表

IOMMU使用一种页表结构，设计为支持完整的64位DVA空间，同时在许多常见情况下
允许更快的转换。IOMMU页表的格式是AMD64架构长模式页表的泛化，同时保持与它
们的兼容性。

IOMMU页表层次结构最多支持6级，从底部标记为Level-1到顶部的Level-6。每级有
多达512个8字节条目。将其与AMD64长模式4KB页转换进行比较，AMD64长模式定义了
4级页表层次结构，其级别从底部标记为Level-1到顶部的Level-4。

设备表项中的Mode字段控制为每个设备使用的页表级别数。Mode字段由表8定义。对
于给定设备，Mode字段中表级别数的选择取决于设备可以生成的最大DVA以及页表中
页的大小。

IOMMU支持两种类型的I/O页表项：
• 页表项（PTE）——终止转换并定义映射的物理页面。
• 页目录项（PDE）——指向下一级页表的非叶节点。

当页不存在时（PR=0），I/O页表项格式如图8所示。
当页存在时（PR=1），I/O页表项（PTE）如图9所示。
当页存在时（PR=1），I/O页目录项（PDE）如图10所示。

页表项中的主要字段包括：
• PR（存在位）：指示页面是否存在于内存中。
• FC（页缓存/窥探控制）：控制是否需要窥探。
• IR/IW（读/写权限）：定义页面的读写访问权限。
• U/S（用户/特权）：定义访问特权级别。
• 物理页基地址：转换后的物理地址。
• NextLevel：指向下一级页表（仅PDE）。

IOMMU支持跳过级别优化。软件可以配置页表以跳过中间级别，允许更大的页面大小。
例如，跳过级别的设置可以实现2MB或1GB大页的映射。

图11和图12展示了跳过级别和非默认页大小的地址转换示例。

2.2.3.1 主机访问支持

当MMIO偏移0030h[HADSup]指示支持时，IOMMU可以可选地在主机页表项中设置已访
问位（A位）。当DTE[HAD]=1时，IOMMU在遍历主机页表时为该设备更新PTE和PDE中
的A位。

2.2.3.2 主机脏支持

当MMIO偏移0030h[HDSup]指示支持时，IOMMU可以可选地在主机页表项中设置脏位
（D位）。当DTE[HAD]=1且EFR[HDSup]=1时，IOMMU在该设备的DMA写操作期间更新
PTE中的D位。

2.2.4 共享AMD64处理器和IOMMU页表——GPA到SPA

在设计上，IOMMU主机I/O页表兼容AMD64处理器长模式页表。这意味着系统软件可以
配置一组页表同时供IOMMU和AMD64处理器使用，当映射GPA到SPA时。

要共享页表，软件必须确保：
• 页表项中的保留字段设置为零。
• IOMMU和处理器使用相同数量的页表级别。
• 使用兼容的页大小。

当共享页表时，对页表的任何更改都必须同时无效IOMMU和处理器的TLB。

2.2.5 中断重映射表

中断重映射表用于重映射和控制来自I/O设备的中断。每个设备表项可以包含指向中断
重映射表的指针。中断重映射表中的条目称为中断重映射表项（IRTE）。

中断重映射提供以下功能：
• 将设备中断向量重映射到不同的向量。
• 控制中断目的地。
• 允许或阻止特定的中断类型。
• 支持可选的中断虚拟化。

[图14: 固定和仲裁中断的中断重映射表查找]

中断重映射表查找过程：
1. IOMMU从设备表项中获取中断重映射表的基地址。
2. 使用中断向量作为索引查找IRTE。
3. IRTE中的字段定义重映射后的中断属性。

[图15: 中断重映射表项——基本格式]

IRTE的基本格式是128位。主要字段包括：
• RemapEn：启用中断重映射。
• SupIOPF：抑制IO_PAGE_FAULT日志。
• IntType：中断类型。
• RqEoi：请求EOI。
• DM：目的地模式。
• Destination：中断目的地。
• Vector：重映射后的中断向量。

2.2.5.1 中断重映射操作

当中断重映射启用时（DTE[IntCtl]=10b），IOMMU拦截来自设备的中断消息，使用
DeviceID查找设备表项，然后使用设备表项中的中断表根指针和中断向量查找IRTE。
IOMMU根据IRTE中的信息重映射中断并向上游转发。

2.2.5.2 Guest虚拟APIC支持（中断虚拟化）

当支持Guest虚拟APIC（MMIO偏移0030h[GASup]=1且MMIO偏移0018h[GAEn]=1）时，
IRTE可以配置为使用中断虚拟化模式。在此模式下，IRTE包含额外的字段用于直接
将中断传递给Guest虚拟机。

[图17: 具有Guest虚拟APIC的IRTE字段，IRTE[GuestMode]=0]
[图18: 具有Guest虚拟APIC的IRTE字段，IRTE[GuestMode]=1]

当GuestMode=1时，IRTE中的附加字段包括：
• GA_LogIntr：如果Guest未运行是否记录中断。
• GA_Destination：Guest APIC目的地。
• GA_Tag：Guest APIC标记。
• GA_Root_Pointer：Guest虚拟APIC后备页的指针。

2.2.5.3 IOMMU x2APIC支持

当MMIO偏移0030h[XTSup]=1时，IOMMU支持x2APIC扩展模式。在x2APIC模式下，
IRTE被扩展以支持更大的APIC ID（32位而非8位）。

2.2.6 Guest转换的I/O页表

IOMMU可选地支持Guest地址转换，使设备能够使用Guest虚拟地址（GVA）。Guest
转换使用与AMD64长模式兼容的页表格式。

Guest转换通过两级地址转换工作：
1. GVA到GPA（Guest转换）：使用Guest页表。
2. GPA到SPA（嵌套/主机转换）：使用主机页表。

2.2.6.1 AMD64 Guest页表NX字段支持

可选地，IOMMU可以执行Guest页表中的不可执行（NX）位。当EFR[NXSup]=1时，
IOMMU检查Guest PTE中的NX位。

2.2.6.2 AMD64 Guest页表访问保护

可选地，IOMMU支持Guest页表中的用户/特权（U/S）访问保护。当EFR[USSup]=1时，
IOMMU检查Guest PTE中的U/S位以提供特权级别的访问控制。

2.2.6.3 Guest CR3表

Guest CR3表将PASID映射到Guest页表根指针。IOMMU使用设备表项中的GCR3表根指
针查找Guest CR3表，然后使用TLP PASID前缀中的PASID值索引到表中以获取该进
程的Guest页表根指针。

Guest CR3表支持1级或2级结构：
• 1级Guest CR3表：4KB，最多512个条目，支持最多9位PASID。
• 2级Guest CR3表：支持最多20位PASID（最多1048576个条目）。

2.2.7 Guest和嵌套地址转换

本节描述IOMMU执行两级（Guest和嵌套）地址转换的过程。

2.2.7.1 组合Guest和主机地址转换

两级转换的完整流程如下：
1. 使用DeviceID查找设备表项。
2. 使用PASID从Guest CR3表获取Guest页表根指针。
3. 使用GVA遍历Guest页表，将GVA转换为GPA。
4. 在Guest页表遍历的每一步中，Guest页表项本身的地址（GPA）需要通过主机页
   表转换为SPA以进行读取。
5. 最终的GPA通过主机页表转换为SPA。

[图37: 完整的GVA到SPA地址转换]

2.2.7.2 Guest到嵌套权限组合

最终的访问权限是Guest和嵌套转换的权限的组合（取交集）。即，只有当Guest和
嵌套页表都授权访问时，才允许访问。

2.2.7.3 重新计算读写访问权限

当缓存的权限表示访问被拒绝但页表可能已更新时，IOMMU可能需要重新遍历页表
以获取当前权限。

2.2.7.4 更新Guest地址表中的已访问位和脏位

当Guest地址转换启用时，IOMMU在遍历Guest页表时更新已访问位和脏位：
• 当IOMMU首次读取或写入页面时，设置A位。
• 当IOMMU首次写入页面时，设置D位。

这些更新必须是原子的。IOMMU从不清除A或D位。

2.2.7.5 清除已访问位和脏位

软件清除A或D位时必须：
1. 清除内存中的位。
2. 向IOMMU发出适当的无效命令。
3. 如果适用，无效远程IOTLB。

2.2.7.6 Guest地址转换的缺页

在Guest地址转换期间发生缺页时，IOMMU根据请求类型和IOMMU控制设置做出响应。

2.2.7.7 PCIe® TLP PASID前缀

PCIe TLP PASID前缀携带PASID值和附加的访问属性信息。IOMMU使用PASID TLP前
缀中的信息选择适当的Guest转换表。

[图38: PCIe® TLP PASID前缀载荷格式]

PASID TLP前缀中的字段包括：
• PASID[19:0]：20位进程地址空间ID。
• Execute Requested：请求执行权限。
• Privileged Mode Requested：请求特权模式。

2.2.8 用于中断虚拟化的Guest虚拟APIC表

Guest虚拟APIC表用于支持将中断直接传递给Guest虚拟机。当中断虚拟化启用时，
IOMMU使用Guest虚拟APIC后备页来管理Guest的中断状态。

2.2.9 Guest I/O保护

Guest I/O保护允许为不支持PASID的设备提供Guest转换。当DTE[GIoV]=1时，为不
带PASID的DMA请求分配默认PASID=0，然后请求经过正常的Guest和嵌套转换。

--------------------------------------------------------------------------------
2.3 启动IOMMU
--------------------------------------------------------------------------------

启动IOMMU涉及初始化其数据结构和MMIO寄存器。

2.3.1 数据结构初始化

在启用IOMMU之前，系统软件必须初始化以下数据结构：
1. 设备表——分配内存，初始化所有条目。对于不使用的设备，将V=0和IV=0设置
   以阻止所有访问。
2. I/O页表——为需要地址转换的设备分配和设置页表。
3. 中断重映射表——为需要中断重映射的设备分配和设置中断重映射表。
4. 命令缓冲区——分配一个4KB对齐的内存区域作为命令环形缓冲区。
5. 事件日志——分配一个4KB对齐的内存区域作为事件日志环形缓冲区。
6. PPR日志——（可选）分配一个4KB对齐的内存区域作为PPR日志。
7. Guest虚拟APIC日志——（可选）分配一个4KB对齐的内存区域。

初始化MMIO寄存器的步骤：
1. 编程设备表基地址寄存器[MMIO偏移0000h]。
2. 编程命令缓冲区基地址寄存器[MMIO偏移0008h]。
3. 编程事件日志基地址寄存器[MMIO偏移0010h]。
4. 设置命令缓冲区头指针和尾指针为零。
5. 设置事件日志头指针和尾指针为零。
6. 编程IOMMU控制寄存器[MMIO偏移0018h]以启用IOMMU。

2.3.2 修改Guest中断虚拟化

更改Guest中断虚拟化设置需要特定的操作顺序以确保一致性。软件必须按照正确的
序列修改DTE和IRTE字段，并发出适当的无效命令。

--------------------------------------------------------------------------------
2.4 命令
--------------------------------------------------------------------------------

IOMMU通过命令缓冲区接受处理器的命令。命令缓冲区是系统内存中的环形缓冲区，
由MMIO寄存器中的头指针和尾指针管理。

软件通过以下步骤发出命令：
1. 将命令写入命令缓冲区的尾指针位置。
2. 推进命令缓冲区尾指针寄存器[MMIO偏移2004h]。
3. IOMMU从头指针位置开始处理命令，直到追上尾指针。

命令缓冲区中的每个条目为128位（16字节）。

[图39: 系统内存中的命令缓冲区]
[图40: 通用命令缓冲区条目格式]

IOMMU定义了以下命令：

2.4.1 COMPLETION_WAIT（完成等待）

COMPLETION_WAIT命令在之前的所有命令完成后发出完成信号。它可选地将数据存储
到指定的系统物理地址，并/或设置完成等待中断标志。

[图41: COMPLETION_WAIT命令格式]

字段说明：
• s：存储标志。1=将StoreData写入StoreAddress。
• i：中断标志。1=在完成时设置完成等待中断位。
• f：刷新标志。1=刷新所有命令之前的写操作。
• StoreAddress：存储数据的系统物理地址。
• StoreData：要存储的64位数据。

2.4.2 INVALIDATE_DEVTAB_ENTRY（无效设备表项）

此命令使IOMMU无效其内部缓存中指定DeviceID的设备表项。

[图42: INVALIDATE_DEVTAB_ENTRY命令格式]

字段说明：
• DeviceID：要无效的设备的16位DeviceID。

软件在更改设备表项后必须发出此命令以确保IOMMU使用更新后的信息。

2.4.3 INVALIDATE_IOMMU_PAGES（无效IOMMU页）

此命令使IOMMU无效其TLB中指定域的地址转换条目。

[图43: INVALIDATE_IOMMU_PAGES命令格式]

字段说明：
• DomainID：16位域标识符。
• S：大小位。1=无效所有匹配的页面。
• PDE：页目录条目位。1=也无效页目录条目。
• Address：要无效的地址（与S和PDE一起确定无效范围）。

2.4.4 INVALIDATE_IOTLB_PAGES（无效IOTLB页）

此命令使IOMMU生成一个设备IOTLB无效请求，发送到指定的外设以无效其IOTLB中
的条目。

[图44: INVALIDATE_IOTLB_PAGES命令格式]

字段说明：
• DeviceID：目标外设的DeviceID。
• Maxpend：外设支持的最大待处理无效请求数。
• QueueID：目标队列ID。
• S：大小位。
• Address：要无效的地址。

当外设未能在超时时间内响应无效请求时，IOMMU会生成IOTLB_INV_TIMEOUT事件。

2.4.5 INVALIDATE_INTERRUPT_TABLE（无效中断表）

此命令使IOMMU无效其缓存中指定设备的中断重映射信息。

[图45: INVALIDATE_INTERRUPT_TABLE命令格式]

字段说明：
• DeviceID：要无效中断重映射信息的设备的DeviceID。

2.4.6 PREFETCH_IOMMU_PAGES（预取IOMMU页）

此可选命令（需EFR[PreFSup]支持）提示IOMMU预取指定的地址转换信息到其TLB中。

[图46: PREFETCH_IOMMU_PAGES命令格式]

字段说明：
• DeviceID：设备的DeviceID。
• Address：要预取的地址。
• PFCount：要预取的页面数量。
• Inval：无效标志。

2.4.6.1 PREFETCH_IOMMU_PAGES的事件处理

如果预取命令遇到页表错误（如不存在的页面或权限不足），IOMMU将生成相应的
事件日志条目。

2.4.7 COMPLETE_PPR_REQUEST（完成PPR请求）

此命令允许软件响应外设页请求。软件处理PPR日志中的请求后，使用此命令将结
果发送回外设。

[图47: COMPLETE_PPR_REQUEST命令格式]

字段说明：
• DeviceID：外设的DeviceID。
• PASID：进程地址空间ID。
• CompletionTag：完成标记。
• Status：完成状态。

2.4.8 INVALIDATE_IOMMU_ALL（全部无效）

此可选命令（需EFR[IASup]支持）无效IOMMU中的所有缓存条目，包括设备表、
页表、中断重映射和Guest CR3表缓存。

此命令在以下场景中特别有用：
• 可信启动
• 错误恢复
• 从低功耗状态恢复

INVALIDATE_IOMMU_ALL不影响IOMMU的操作状态或MMIO寄存器内容（除了推进命令
缓冲区头指针）。此命令也不无效远程IOTLB。

2.4.9 INSERT_GUEST_EVENT命令

此命令使IOMMU在事件日志中插入一个Guest事件条目。当vIOMMU启用时，此命令
用于将Guest事件传播到主机事件日志。

2.4.10 RESET_VMMIO命令

此命令重置指定Guest的vIOMMU MMIO状态。当HV需要重新初始化Guest的vIOMMU
状态时使用。

2.4.11 IOMMU排序规则

IOMMU命令的处理遵循严格的排序规则以确保一致性。

2.4.11.1 无效命令排序要求

无效命令按照它们在命令缓冲区中的顺序执行。后续的无效命令不会在前面的无效
命令完成之前生效。COMPLETION_WAIT命令可用于确定之前的所有命令是否已完成。

2.4.11.2 无效命令交互要求

不同类型的无效命令之间的交互必须遵循特定规则：
• INVALIDATE_DEVTAB_ENTRY在INVALIDATE_IOMMU_PAGES之前完成。
• INVALIDATE_IOMMU_PAGES在INVALIDATE_IOTLB_PAGES之前完成。
• INVALIDATE_INTERRUPT_TABLE在COMPLETION_WAIT之前完成。

软件在更改页表后的典型命令序列是：
1. 更新页表条目。
2. 发出INVALIDATE_IOMMU_PAGES命令。
3. 如果有ATS设备，发出INVALIDATE_IOTLB_PAGES命令。
4. 发出COMPLETION_WAIT命令以确认完成。

2.5 事件日志
--------------------------------------------------------------------------------

IOMMU事件日志是系统内存中的环形缓冲区，IOMMU向其中写入事件条目以报告异常
情况。事件日志由MMIO寄存器中的头指针和尾指针管理。

事件日志中的每个条目为128位（16字节）。

当事件日志被写入时，IOMMU可选地向处理器发送MSI/MSI-X中断以通知软件检查
事件日志。

事件日志溢出处理：
• 当事件日志已满时（尾指针追上头指针），IOMMU设置EventOverflow状态位。
• 当EventOverflow置位时，后续事件被丢弃直到软件清除溢出条件。

表42: 事件类型摘要

事件类型                            编码      描述
────────────────────────────────────────────────────────────────────
ILLEGAL_DEV_TABLE_ENTRY             1h       非法设备表项
IO_PAGE_FAULT                       2h       I/O缺页
DEV_TAB_HARDWARE_ERROR              3h       设备表硬件错误
PAGE_TAB_HARDWARE_ERROR             4h       页表硬件错误
ILLEGAL_COMMAND_ERROR               5h       非法命令错误
COMMAND_HARDWARE_ERROR              6h       命令硬件错误
IOTLB_INV_TIMEOUT                   7h       IOTLB无效超时
INVALID_DEVICE_REQUEST              8h       无效设备请求
INVALID_PPR_REQUEST                 9h       无效PPR请求
EVENT_COUNTER_ZERO                  Ah       事件计数器归零

RMP_PAGE_FAULT                      Dh       RMP缺页
RMP_HARDWARE_ERROR                  Eh       RMP硬件错误

2.5.1 事件日志重启过程

当软件检测到事件日志溢出或其他错误时，可以使用以下过程重启事件日志：
1. 禁用事件日志记录。
2. 重置事件日志头指针和尾指针。
3. 清除溢出状态位。
4. 重新启用事件日志记录。

2.5.2 ILLEGAL_DEV_TABLE_ENTRY事件

当IOMMU在读取设备表项时发现非法值（如保留字段非零、不支持的模式值等），
它将生成此事件。

事件日志条目包含：
• DeviceID：触发事件的设备的DeviceID。
• 事件类型编码：01h。
• 地址：导致事件的事务地址。
• DTE相关信息。

2.5.3 IO_PAGE_FAULT事件

I/O缺页事件在以下情况下产生：
• 页表项标记为不存在（PR=0）。
• 权限不足（例如，写入只读页面）。
• 使用了保留的页表项编码。

事件日志条目包含：
• DeviceID：触发事件的设备。
• DomainID：设备所属域的标识符。
• 地址：导致缺页的地址。
• 标志：指示故障类型（读/写、Guest/嵌套等）。

2.5.4 DEV_TAB_HARDWARE_ERROR事件

当IOMMU在读取设备表时遇到硬件错误（如ECC错误、无效地址）时产生。

2.5.5 PAGE_TAB_HARDWARE_ERROR事件

当IOMMU在遍历页表时遇到硬件错误时产生。

2.5.6 ILLEGAL_COMMAND_ERROR事件

当IOMMU在命令缓冲区中遇到无法识别或非法的命令时产生。此事件导致IOMMU
停止处理命令缓冲区中的命令。软件必须更正错误并重启命令处理。

2.5.7 COMMAND_HARDWARE_ERROR事件

当IOMMU在读取命令缓冲区时遇到硬件错误时产生。

2.5.8 IOTLB_INV_TIMEOUT事件

当外设未能在超时时间内响应INVALIDATE_IOTLB_PAGES命令时产生。

2.5.9 INVALID_DEVICE_REQUEST事件

当来自设备的请求无效时产生，包括：
• 访问保留的中断地址空间。
• 端口I/O访问被阻止。
• 系统管理地址访问被阻止。
• 预转换地址请求来自未标记IOTLB的设备。

2.5.10 INVALID_PPR_REQUEST事件

当IOMMU收到无效的外设页请求时产生。

2.5.11 EVENT_COUNTER_ZERO事件

当性能计数器达到零（下溢）时产生。

2.5.12 GUEST_EVENT_FAULT事件

当vIOMMU处理Guest命令时检测到错误时产生。

2.5.13 VIOMMU_HARDWARE_ERROR事件

当vIOMMU在访问内存数据结构时遇到硬件错误时产生。

2.5.14 RMP_PAGE_FAULT事件

当安全嵌套分页（SEV-SNP）的RMP检查失败时产生。

2.5.15 RMP_HARDWARE_ERROR事件

当IOMMU在访问RMP表时遇到硬件错误时产生。

2.5.16 IOMMU事件报告

本节描述IOMMU报告事件的通用框架。

2.5.16.1 IOMMU数据验证序列

IOMMU在处理设备请求时按特定顺序执行数据验证：
1. 读取设备表项。
2. 验证DTE字段。
3. 遍历页表（如果启用）。
4. 验证页表项。
5. 执行权限检查。

在任何步骤中发现错误都会终止处理并生成相应的事件。

2.5.16.2 I/O硬件事件报告寄存器

当MMIO偏移0030h[HESup]=1时，关键硬件错误也可以记录在专用的硬件事件寄存器
中，以便更快地检测和处理严重错误。

相关寄存器包括：
• IOMMU硬件事件上部寄存器[MMIO偏移0040h]
• IOMMU硬件事件下部寄存器[MMIO偏移0048h]
• IOMMU硬件事件状态寄存器[MMIO偏移0050h]

2.5.17 事件日志双缓冲

当MMIO偏移0030h[DualEventLogSup]=1时，IOMMU支持两个事件日志缓冲区。双缓
冲机制允许软件安全地处理一个缓冲区中的事件，同时IOMMU将新事件写入另一个
缓冲区。

双缓冲可以手动或自动切换：
• 手动：软件显式切换活动缓冲区。
• 自动（autoswap）：当一个缓冲区满时IOMMU自动切换。

2.6 外设页请求（PPR）日志记录
--------------------------------------------------------------------------------

IOMMU可选地支持外设页请求日志记录，用于记录来自PRI能力外设的页请求。PPR日
志是系统内存中的环形缓冲区。

当外设需要访问当前不在内存中或权限不足的页面时，它可以通过PRI协议发出页请
求。IOMMU接收这些请求并将其记录在PPR日志中供系统软件处理。

PPR日志由以下MMIO寄存器管理：
• PPR日志基地址寄存器[MMIO偏移0038h]
• PPR日志头指针寄存器[MMIO偏移2030h]
• PPR日志尾指针寄存器[MMIO偏移2038h]

2.6.1 PPR日志双缓冲

当MMIO偏移0030h[DualPprLogSup]=1时，IOMMU支持两个PPR日志缓冲区。双缓冲
的工作方式与事件日志双缓冲类似。

2.6.2 外设页请求日志重启过程

当PPR日志溢出或出现错误时，软件可以使用以下过程重启PPR日志：
1. 禁用PPR日志记录。
2. 重置PPR日志头指针和尾指针。
3. 清除溢出状态位。
4. 重新启用PPR日志记录。

2.6.3 外设页请求条目

PPR日志条目为128位，包含以下信息：
• DeviceID：发出请求的外设的DeviceID。
• PASID：进程地址空间ID。
• Tag：请求标记，用于将响应与请求匹配。
• 地址：请求的虚拟地址。
• 标志：页请求类型（读/写、最后一个请求等）。

2.6.4 PPR日志溢出保护

IOMMU提供多种机制来处理PPR日志溢出：

2.6.4.1 PPR自动响应

当PPR自动响应启用时（EFR[PprAutoRspSup]=1且控制位设置），IOMMU可以自动
向外设发送响应而无需软件干预。这在PPR日志即将溢出时特别有用。

2.6.4.3 PPR日志溢出早期指示

当EFR[PprOvrflwEarlySup]=1时，IOMMU可以在PPR日志接近满时提前发出警告，
给软件更多时间处理待处理的请求。

2.6.4.5 块停止标记消息

块停止标记机制允许IOMMU在PPR日志溢出时阻止外设发送更多请求，直到日志
有空间为止。

--------------------------------------------------------------------------------
2.7 Guest虚拟APIC（GA）日志记录
--------------------------------------------------------------------------------

Guest虚拟APIC日志记录用于在Guest虚拟机未运行时记录中断请求信息。当IOMMU
需要将中断传递给不在运行的Guest时，它将中断信息写入GA日志。

2.7.1 Guest vAPIC虚拟中断请求日志

当IOMMU接收到目标为非运行Guest的中断时：
1. IOMMU查找IRTE以获取Guest虚拟APIC信息。
2. 如果Guest正在运行，直接传递中断。
3. 如果Guest未运行，将中断信息写入GA日志。
4. 可选地向Hypervisor发送中断以通知。

GA日志也是系统内存中的环形缓冲区。

2.7.2 Guest虚拟APIC日志条目（通用）

GA日志条目为128位，包含中断信息和Guest标识。

2.7.3 Guest虚拟APIC请求条目（GA_GUEST_NR）

当Guest未运行时，此类型的GA日志条目记录需要在Guest恢复运行时传递的中断。

2.7.4 Guest虚拟APIC日志重启过程

与事件日志和PPR日志类似，GA日志在溢出或错误后可以重启。

--------------------------------------------------------------------------------
2.8 IOMMU中断支持
--------------------------------------------------------------------------------

IOMMU使用PCI MSI或MSI-X中断向处理器发送通知。IOMMU可以为以下事件发送
中断：
• 事件日志写入
• PPR日志写入
• GA日志写入
• COMPLETION_WAIT完成
• 性能计数器事件

IOMMU MSI/MSI-X配置通过PCI配置空间或MMIO寄存器访问。

2.9 内存地址路由和控制（MARC）
--------------------------------------------------------------------------------

MARC是一个可选特性（EFR[MarcSup]），为需要低延迟系统内存访问的Guest分配
设备提供专用的物理内存访问路径。

MARC定义了最多4个孔径（aperture），每个孔径指定一个系统物理地址范围。来
自集成I/O设备（如GPU显示扫描输出引擎）的访问如果落在MARC孔径内，可以绕
过IOMMU的转换和检查，直接访问系统内存。

每个MARC孔径由一对寄存器定义：
• MARC孔径基地址寄存器
• MARC孔径重定位和长度寄存器

MARC孔径由Hypervisor配置和管理，用于满足实时设备的低延迟需求。

2.10 vIOMMU
--------------------------------------------------------------------------------

vIOMMU是一个可选特性（EFR[vIOMMUSup]），提供硬件加速的虚拟化IOMMU功能。

2.10.1 vIOMMU私有地址空间

vIOMMU为每个启用vIOMMU的VM分配私有的MMIO地址空间。每个VM的vIOMMU MMIO
空间为4KB，映射到IOMMU暴露的MMIO资源中。

2.10.1.1 vIOMMU后备存储内存

IOMMU为vIOMMU功能使用系统内存中的后备存储。每个启用vIOMMU的VM需要后备
存储来保存vIOMMU状态。

2.10.2 vIOMMU MMIO资源

vIOMMU暴露一组MMIO寄存器子集给每个VM，包括：
• 命令缓冲区头指针和尾指针
• 事件日志头指针和尾指针
• PPR日志头指针和尾指针

当VM访问这些寄存器时，IOMMU直接处理而无需HV干预。

2.10.3 vIOMMU事件日志记录

vIOMMU支持将Guest IOMMU事件直接写入VM的事件日志。IOMMU在必要时进行
DeviceID和DomainID的转换（从Guest编码到主机编码）。

2.10.4 vIOMMU扩展中断重映射

vIOMMU支持将中断重映射功能扩展到虚拟化环境，允许Guest IOMMU参与中断
重映射。

2.10.5 vIOMMU和EOI总线周期

描述vIOMMU如何处理来自Guest的中断结束（EOI）信号。

--------------------------------------------------------------------------------
2.11 安全ATS支持
--------------------------------------------------------------------------------

当EFR[SATSSup]=1时，IOMMU支持安全ATS请求处理。安全ATS在SEV-SNP环境
中提供额外的安全检查，确保ATS响应中的转换信息受到RMP表的保护。

当DTE[SATS]=1且DTE[I]=1时，来自外设的ATS请求被视为安全ATS请求，IOMMU
在处理转换时执行额外的RMP检查。

--------------------------------------------------------------------------------
2.12 IOMMU安全嵌套分页（SEV-SNP）支持
--------------------------------------------------------------------------------

IOMMU的SEV-SNP支持为虚拟机内存提供额外的完整性保护，利用反向映射表（RMP）
数据结构。

2.12.1 SEV-SNP RMP访问检查

当SEV-SNP启用时，IOMMU在转换过程中执行额外的RMP检查：
1. 对转换后的SPA进行RMP查找。
2. 验证RMP条目中的所有权和权限信息。
3. 如果RMP检查失败，生成RMP_PAGE_FAULT或RMP_HARDWARE_ERROR事件。

2.12.2 SEV-SNP限制

SEV-SNP在IOMMU操作上施加了一些限制。

2.12.2.1 SEV-SNP寄存器锁定

当SNP启用时，某些IOMMU寄存器被锁定以防止恶意修改。这些锁定在SNP启用后
生效，直到系统复位。

2.12.2.2 SEV-SNP COMPLETION_WAIT存储限制

在SNP启用的系统中，COMPLETION_WAIT命令的存储地址必须受到SNP的保护限制。

2.12.2.3 SEV-SNP排除范围限制

在SNP启用的系统中，IOMMU排除范围的使用受到限制以防止安全绕过。

2.12.2.4 SEV-SNP页模式限制

SNP启用时对页表模式有特定限制，以确保转换的安全性。

2.12.3 SEV-SNP Guest虚拟APIC支持

描述SEV-SNP环境中Guest虚拟APIC的特殊处理要求和限制。


================================================================================
第3章 寄存器（Registers）
================================================================================

IOMMU通过两组寄存器进行配置和控制——一组位于PCI配置空间，另一组映射在系统地址空间中。由于IOMMU表现为一个PCI功能（Function），因此它在PCI配置空间中有一个能力块（Capability Block）。

本章未定义的IOMMU寄存器均为保留（Reserved）。软件不应写入保留的寄存器位置，且不得依赖于硬件在读取保留寄存器时返回的值。可写寄存器中被标记为Reserved的位范围应始终写入全零。读取时，软件不得依赖该位范围返回的值。

3.1 PCI资源

IOMMU实现为一个独立的PCI功能。任何包含IOMMU能力块的PCI功能不能用于除包含IOMMU以外的任何其他用途。包含IOMMU能力块的PCI功能不包括PCI BAR寄存器。IOMMU的配置和状态信息通过PCI能力块映射到PCI配置空间。

一个外设可以在单个Function中实现多个IOMMU。一个PCIe Function中可以实现一个或多个IOMMU能力块。

如果在一个PCI Function中实现了单个IOMMU能力块，则IOMMU可以支持MSI或MSI-X或两者。如果在一个Function中实现了多个IOMMU能力块，则IOMMU必须支持生成MSI-X中断。该Function必须为每个IOMMU能力块可以生成的每个中断分配不同的中断向量。

IOMMU的基本功能支持一个中断，用于报告事件日志异常和CompletionWait完成。如果启用了此中断，则Capability Offset 10h [MsiNum]字段必须写入适当的MSI/MSI-X向量。如果支持PPR日志和GA日志，则提供另外两个中断源来通知PPR和GA日志异常。如果要启用PPR日志中断，必须先在Capability Offset 10h[MsiNumPPR]中设置向量。如果要启用GA日志中断，必须先在Capability Offset 10h [MsiNumGA]中设置向量。

PCI类为System Base Peripheral（08h），子类为IOMMU（06h），编程接口代码为00h，由PCI-SIG颁发。

IOMMU生成请求时使用的HyperTransport UnitID不得用于任何其他流量。一个HyperTransport UnitID可以由物理组件内的多个IOMMU共享。

3.1.1 访问MSI能力块寄存器

可选地，MSI能力寄存器的特定字段由IOMMU硬件映射到MMIO空间。当支持此功能时，这些寄存器可以通过配置空间读写或内存读写来访问。但在某些情况下，字段写入能力在配置空间和MMIO空间访问之间有所不同。详见第3.4.8节[MMIO访问MSI能力块寄存器]。

3.2 IOMMU基础能力块寄存器（IOMMU Base Capability Block Registers）

PCI功能中存在IOMMU能力块表示存在IOMMU。IOMMU能力块包含各种寄存器来控制IOMMU和配置IOMMU的MMIO寄存器位置。

当Capability Offset 04h [Enable]被写入1b时，本节中定义的所有RW能力寄存器将被锁定直到下一次系统复位。这意味着寄存器变为只读，写入尝试将被忽略。

■ Capability Offset 00h — IOMMU能力头（IOMMU Capability Header）

位域：[31:29] Reserved | [28] CapExt | [27] EFRSup | [26] NpCache | [25] HtTunnel | [24] IotlbSup | [23:19] CapRev | [18:16] CapType | [15:8] CapPtr | [7:0] CapId

这是IOMMU能力块的第一个寄存器，提供IOMMU支持的能力信息。

- Bits[31:29] Reserved。
- Bit[28] CapExt：1=支持IOMMU杂项信息寄存器1[Capability Offset 14h]。0=不支持。
- Bit[27] EFRSup：IOMMU扩展特性寄存器支持。RO。复位Xb。1=支持IOMMU扩展特性寄存器[MMIO Offset 0030h]。0=MMIO Offset 0030h为保留。
- Bit[26] NpCache：缓存标记为不存在的页表项。RO。复位Xb。1=IOMMU缓存标记为不存在的页表项，软件在任何PDE或PTE更改后必须发出无效化命令。0=IOMMU仅缓存标记为存在的页表项。实现说明：对于IOMMU硬件实现，此位必须为0b。
- Bit[25] HtTunnel：HyperTransport隧道转换支持。RO。复位Xb。表示设备包含支持HyperTransport接口上地址转换的隧道。
- Bit[24] IotlbSup：IOTLB支持。RO。复位Xb。表示IOMMU支持PCI ATS 1.0或更高版本中定义的ATS转换请求消息。
- Bits[23:19] CapRev：能力修订版。RO。复位0_0001b。指定IOMMU接口修订版本。
- Bits[18:16] CapType：IOMMU能力块类型。RO。复位011b。
- Bits[15:8] CapPtr：能力指针。RO。复位XXh。指示下一个能力块的位置，最后一个则为00h。
- Bits[7:0] CapId：能力ID。RO。复位0Fh。标识安全设备能力块。

■ Capability Offset 04h — IOMMU基地址低位寄存器（IOMMU Base Address Low Register）

指定IOMMU控制寄存器基地址（SPA）的位[31:14]。当Enable被写入1b时锁定。

当MMIO Offset 0030h [PCSup]=1时，MMIO区域须512KB大小且512KB对齐，BaseAddress[18:14]须为0_0000b。当PCSup=0时，MMIO区域须16KB大小且16KB对齐。

- Bits[31:19] BaseAddress[31:19]：Enable=0时RW，Enable=1时RO。复位0。
- Bits[18:14] BaseAddress[18:14]：Enable=0时RW，Enable=1时RO。复位0_0000b。须设为0_0000b。
- Bits[13:1] Reserved。
- Bit[0] Enable：RW1S。复位0b。1=IOMMU接受对基地址的内存访问，同时锁定所有RW能力寄存器。注意：BaseAddress可在设置Enable=1b的同一写操作中更改并锁定。

■ Capability Offset 08h — IOMMU基地址高位寄存器

指定IOMMU控制寄存器基地址的高32位。Enable被写入1b时锁定。

- Bits[31:0] BaseAddress[63:32]：Enable=0时RW，Enable=1时RO。复位0000_0000h。

■ Capability Offset 0Ch — IOMMU范围寄存器（IOMMU Range Register）

指示与IOMMU关联的第一个和最后一个设备的设备号和功能号。Enable被写入1时锁定。

根端口设备（设备号在FirstDevice和LastDevice之间含）由IOMMU支持并提供完整源标识。非根端口设备（在同一范围内）是与IOMMU集成的设备，支持使用IOMMU的地址转换。关联的集成设备须位于同一逻辑总线。

- Bits[31:24] LastDevice：最后一个设备。Enable=0时RW，=1时RO。复位XXh。
- Bits[23:16] FirstDevice：第一个设备。Enable=0时RW，=1时RO。复位XXh。
- Bits[15:8] BusNumber：设备范围总线号。Enable=0时RW，=1时RO。复位XXh。
- Bit[7] RngValid：范围有效。Enable=0时RW，=1时RO。复位Xb。1b=BusNumber/FirstDevice/LastDevice字段有效（但鼓励使用第5章I/O拓扑信息）。0b=须使用I/O拓扑信息。
- Bits[6:5] Reserved。
- Bits[4:0] UnitID：IOMMU HyperTransport UnitID。Enable=0时RW，=1时RO。复位X_XXXXb。注意：此字段已弃用，可设为0_0000b。

■ Capability Offset 10h — IOMMU杂项信息寄存器0（Miscellaneous Information Register 0）

返回IOMMU支持的虚拟和物理地址大小及MSI/MSI-X消息编号。

- Bits[31:27] MsiNumPPR：外设页请求MSI消息编号。RO。PPRSup=1时指示PPR日志中断的MSI/MSI-X向量。PPRSup=0时须为零。PPR中断不支持INTx。
- Bits[26:23] Reserved。
- Bit[22] HtAtsResv：ATS响应地址范围保留。Enable=0时RW，=1时RO。复位0b。1=ATS响应地址范围被保留不可由IOMMU转换。
- Bits[21:15] VAsize：虚拟地址大小。RO。最大虚拟地址的二进制对数。允许值：32/40/48/64位。若支持Guest转换，定义GPA大小。
- Bits[14:8] PAsize：物理地址大小。RO。最大SPA宽度的二进制对数。允许值：40/48/52位。
- Bits[7:5] GVAsize：Guest虚拟地址大小。RO。允许大小为48位。
- Bits[4:0] MsiNum：MSI消息编号。RO。指示事件日志中断的MSI/MSI-X向量。支持INTx仅为向后兼容，不鼓励使用。

■ Capability Offset 14h — IOMMU杂项信息寄存器1（Miscellaneous Information Register 1）

当Capability Offset 00h [ExtCap]=1时内容有效，=0时保留。

- Bits[31:5] Reserved。
- Bits[4:0] MsiNumGA：GA日志关联的MSI/MSI-X消息编号。RO。当GASup=1时有效。

3.3 IOMMU厂商特定能力块寄存器

IOMMU厂商特定能力是PCI 3.0兼容配置空间中的一个能力结构，包含控制vIOMMU的寄存器。

■ Vendor Specific Capability Offset 00h — 厂商特定能力头

- Bits[31:24] CapType：RO。复位01h。
- Bits[23:16] CapLen：能力长度（字节数）。RO。复位XXh。
- Bits[15:8] CapPtr：能力指针。RO。复位XXh。
- Bits[7:0] CapID：能力ID。RO。复位09h（厂商特定能力块）。

■ Vendor Specific Capability Offset 04h — 厂商特定能力信息

- Bits[31:24] Reserved。
- Bits[23:16] VSC_Rev：厂商特定能力修订版。RO。复位0h。
- Bits[15:0] VSC_ID：厂商特定能力ID。RO。复位00h。

■ Vendor Specific Capability Offset 08h — 虚拟功能基地址低位寄存器

IOMMU虚拟功能MMIO区域基地址（SPA）。须256M对齐。

- Bits[31:28] VF_BaseAddress[31:28]：VF_Enable=0时RW，=1时RO。复位0000b。
- Bits[27:1] Reserved。
- Bit[0] VF_Enable：RW1S。复位0b。1=接受VF_BaseAddress内存访问，锁定VF_BaseAddress。

■ Vendor Specific Capability Offset 0Ch — 虚拟功能基地址高位寄存器

- Bits[31:0] VF_BaseAddress[63:32]：VF_Enable=0时RW，=1时RO。复位0000_0000h。

■ Vendor Specific Capability Offset 10h — 虚拟功能控制基地址低位寄存器

IOMMU虚拟功能控制MMIO区域基地址（SPA）。须4M对齐。

- Bits[31:22] VFCntl_BaseAddress[31:22]：VFCntl_Enable=0时RW，=1时RO。复位00_0000_0000b。
- Bits[21:1] Reserved。
- Bit[0] VFCntl_Enable：RW1S。复位0b。1=接受VFCntl_BaseAddress内存访问，锁定VFCntl_BaseAddress。

■ Vendor Specific Capability Offset 14h — 虚拟功能控制基地址高位寄存器

- Bits[31:0] VFCntl_BaseAddress[63:32]：VFCntl_Enable=0时RW，=1时RO。复位0000_0000h。

3.4 IOMMU MMIO寄存器

IOMMU控制寄存器使用IOMMU基地址低位寄存器[Capability Offset 04h]和基地址高位寄存器[Capability Offset 08h]指定的地址进行映射。软件对IOMMU寄存器的访问不得大于64位。访问须按大小对齐，大小（字节）须为2的幂。软件可使用最小1字节的访问。

3.4.1 控制和状态寄存器

■ MMIO Offset 0000h — 设备表基地址寄存器（Device Table Base Address Register）

不支持或未启用设备表分段时，此寄存器建立单一统一设备表的基地址。启用分段时，作为段0的基地址寄存器。

- Bits[63:52] Reserved。
- Bits[51:12] DevTabBase：设备表基地址。RW。复位00_0000_0000h。指定第一级设备表4KB对齐基地址的位[51:12]。
- Bits[11:9] Reserved。
- Bits[8:0] Size：设备表大小。RW。复位000h。无符号值n，大小=(n+1)*4KB。值0=4KB，1FFh（最大值）=2MB。

■ MMIO Offset 0008h — 命令缓冲区基地址寄存器（Command Buffer Base Address Register）

指定命令缓冲区的系统物理地址和长度。

- Bits[63:60] Reserved。
- Bits[59:56] ComLen：命令缓冲区长度。RW。复位1000b。2的幂增量。最小256项（4KB），<1000b保留。1000b=256项，...，1111b=32768项（512KB）。
- Bits[55:52] Reserved。
- Bits[51:12] ComBase：命令缓冲区基地址。RW。复位00_0000_0000h。须4KB对齐。
- Bits[11:0] Reserved。

■ MMIO Offset 0010h — 事件日志基地址寄存器（Event Log Base Address Register）

指定事件日志的系统物理地址和长度。软件说明：EventLogRun=1时更改EventLen或EventBase则IOMMU行为未定义。

- Bits[63:60] Reserved。
- Bits[59:56] EventLen：事件日志长度。RW。复位1000b。最小256项（4KB）。
- Bits[55:52] Reserved。
- Bits[51:12] EventBase：事件日志基地址。RW。复位00_0000_0000h。须4KB对齐。
- Bits[11:0] Reserved。

■ MMIO Offset 0018h — IOMMU控制寄存器（IOMMU Control Register）

此64位寄存器控制IOMMU的行为，包含大量控制位。

高32位（Bits[63:32]）：
- Bits[63:61] SNPAVICEn：SNP启用时控制Guest虚拟APIC模式。000b=第2.2.5.2节定义的中断重映射行为。001b-111b=保留。
- Bit[60] Reserved。
- Bit[59] IRTCacheDis：1=禁用IRTE缓存。缓存禁用后软件可直接修改内存IRTE而无需发出INVALIDATE_INTERRUPT_TABLE命令。写1不会使已缓存IRTE无效，须用无效化命令。
- Bits[58:55] Reserved。
- Bit[54] GAUpdateDis：禁用Guest页表访问位的硬件更新。
- Bit[53] vIommuEn：启用虚拟化IOMMU。
- Bit[52] vCmdEn：启用虚拟化命令缓冲区处理。
- Bit[51] IntCapXTEn：启用IOMMU x2APIC中断生成。XTSup=0时保留。
- Bit[50] XTEn：启用X2APIC。XTSup=0时保留。
- Bit[49] Reserved。
- Bit[48] GDUpdateDis：禁用Guest页表脏位的硬件更新。
- Bits[47:46] HADUpdate：主机页表中的访问和脏位更新。
- Bit[45] EPHEn：增强型外设页请求处理。
- Bits[44:43] Reserved。
- Bit[42] PprAutoRspAon：PPR自动响应-始终开启启用。RW。复位0b。启用需同时设置此位和PprAutoRspEn。PprAutoRespSup=0时保留。
- Bit[41] BlkStopMrkEn：阻止StopMark消息启用。RW。复位1b（默认启用）。BlkStopMrkSup=0时保留。
- Bit[40] MarcEn：MARC功能启用。RW。复位0b。0=禁用，1=启用。MarcSup=00b时保留。
- Bit[39] PprAutoRspEn：PPR自动响应启用。RW。复位0b。PprAutoRespSup=0时保留。
- Bits[38:37] PrivAbrtEn：特权中止启用。RW。复位00b。00b=PMR位指示非特权时中止Guest特权页访问。01b=无论PMR位如何都中止。10b/11b=保留。USSup=0时保留。
- Bits[36:34] DevTblSegEn：设备表分段启用。RW。复位000b。000b=不分段，001b=2段，010b=4段，011b=8段，100b-111b=保留。DevTblSegSup=00b时保留。
- Bits[33:32] DualEventLogEn：双事件日志启用。RW。复位00b。00b=默认(A)缓冲区，01b=备用(B)，10b=满时自动交换，11b=保留。DualEventLogSup=00b时保留。

低32位（Bits[31:0]）：
- Bits[31:30] DualPprLogEn：双PPR日志启用。RW。复位00b。00b=默认(A)日志，01b=备用(B)，10b=满时自动交换，11b=保留。若自动响应已启用则不得设为10b。DualPPRLogSup=00b时保留。
- Bit[29] GAIntEn：Guest虚拟APIC中断启用。RW。复位0b。1=GALogInt从0变1时发中断。GASup=0时忽略写入。
- Bit[28] GALogEn：GA日志启用。RW。复位0b。1=启用GA日志。GASup=0时忽略写入。
- Bits[27:25] GAMEn：Guest虚拟APIC模式启用。RW。复位000b。指定高级中断行为和IRTE大小。000b=中断重映射，001b=Guest虚拟APIC虚拟中断行为，010b-111b=保留。GASup=0时忽略写入。
- Bit[24] SmiFLogEn：SMI过滤器日志启用。RW。复位0b。0=不记录被过滤SMI，1=在事件日志中报告。SmiFSup=0b时忽略。
- Bit[23] SlfWBdis：自写回禁用。
- Bit[22] SmiFEn：SMI过滤器启用。0=SMI始终传递上游，1=按SMI过滤器寄存器控制阻止。SmiFSup=0b时忽略。
- Bits[21:18] CRW：RO。复位0h。保留，未来使用。
- Bit[17] GAEn：Guest虚拟APIC启用。RW。复位0b。1=使用DTE中的Guest虚拟APIC表根指针更新设备中断并发送到处理器。GASup=0时忽略写入。
- Bit[16] GTEn：Guest转换启用。RW。复位0b。1=可通过DTE[GV]为外设启用Guest转换。GTSup=0时忽略写入。
- Bit[15] PPREn：外设页请求处理启用。RW。复位0b。1=处理PPR请求。0=PPR请求视为无效设备请求。PPRSup=0时忽略写入。
- Bit[14] PprIntEn：PPR中断启用。RW。复位0b。1=PPRLogInt=1时发中断。PPRSup=0时忽略。
- Bit[13] PPRLogEn：PPR日志启用。RW。复位0b。1=PPR日志已配置，IommuEn也设置时写入PPR日志。0=PPR日志未启用，请求被丢弃。启用时IOMMU清除PPRLogOverflow并设置PPRLogRun。
- Bit[12] CmdBufEn：命令缓冲区启用。RW。复位0b。1=启动/重启命令处理。CmdBufEn=1且IommuEn=1时开始获取命令并设CmdBufRun=1。0=停止获取新命令。
- Bit[11] Isoc：同步。RW。复位0b。控制HyperTransport读请求中的同步位。1=使用同步通道。平台固件应为支持同步通道的处理器设为1b。
- Bit[10] Coherent：一致性。RW。复位1b。控制HyperTransport读请求中的一致性位。1=设备表请求由处理器嗅探。
- Bit[9] ResPassPW：响应越过已发送写入。RW。复位0b。
- Bit[8] PassPW：越过已发送写入。RW。复位0b。
- Bits[7:5] InvTimeOut：无效化超时。RW。复位000b。000b=无超时，001b=1ms，010b=10ms，011b=100ms，100b=1秒，101b=10秒，110b=100秒，111b=保留。
- Bit[4] ComWaitIntEn：完成等待中断启用。RW。复位0b。1=ComWaitInt=1时发中断。
- Bit[3] EventIntEn：事件日志中断启用。RW。复位0b。1=EventLogInt=1或EventOverflow=1时发中断。
- Bit[2] EventLogEn：事件日志启用。RW。复位0b。1=已配置事件日志，IommuEn设置时记录所有事件。启用时清除EventOverflow并设EventLogRun。
- Bit[1] HtTunEn：HyperTransport隧道转换启用。RW。复位0b。1=隧道上游流量由IOMMU转换。IommuEn=0时忽略。
- Bit[0] IommuEn：IOMMU启用。RW。复位0b。1=IOMMU启用，处理所有上游事务。设此位前须先配置设备表基地址寄存器。0=IOMMU禁用，不转换/重映射事务。

■ MMIO Offset 0020h — IOMMU排除基地址寄存器/完成存储基地址寄存器

指定IOMMU排除范围的基DVA。设备表中EX位或Allow位设置时，排除范围内的设备访问不被转换也不进行访问检查。排除范围测试不适用于呈现有效PASID TLP前缀的事务。

软件说明：使用远程IOTLB的外设可能缓存排除范围转换结果，更改排除范围后须发INVALIDATE_IOTLB_PAGES命令。

- Bits[63:52] Reserved。
- Bits[51:12] 排除范围基地址：RW。复位00_0000_0000h。4KB对齐基地址的位[51:12]。SNP启用时此字段被锁定并重用为完成存储基地址。
- Bits[11:2] Reserved。
- Bit[1] Allow：允许所有设备。RW。复位0b。1=排除范围内的所有访问不经转换转发。0=由DTE中EX位控制。SNP启用前后须为0。
- Bit[0] ExEn：排除范围启用。RW。复位0b。1=启用。0=禁用。SNP启用前后须为0。

■ MMIO Offset 0028h — IOMMU排除范围限制寄存器/完成存储限制寄存器

指定排除范围上限。限制的低12位视为FFFh。当基地址=限制地址时排除范围为4KB。

- Bits[63:52] Reserved。
- Bits[51:12] 排除范围限制：RW。复位00_0000_0000h。SNP启用时锁定并重用为完成存储限制。
- Bits[11:0] Reserved。

■ MMIO Offset 0030h — IOMMU扩展特性寄存器（Extended Feature Register）

指定IOMMU支持的扩展特性。当EFRSup=0b时保留。

高32位（Bits[63:32]）：
- Bit[63] SNPSup：安全嵌套分页支持。1=支持SEV-SNP（须CPU和IOMMU同时支持）。
- Bit[62] ForcePhyDestSup：强制物理目标模式用于重映射中断。1=支持。
- Bit[61] GAUpdateDisSup：支持禁用Guest页表访问位硬件更新。
- Bits[60:56] Reserved。
- Bit[55] vIommuSup：虚拟化IOMMU支持。
- Bit[54] InvIotlbTypeSup：无效化IOTLB类型支持。
- Bit[53] Reserved。
- Bit[52] HDSup：主机脏位支持。须同时支持主机访问功能。
- Bit[51] AttrFWSup：属性转发支持。
- Bit[50] EPHSup：增强PPR处理支持。
- Bit[49] HASup：主机访问支持。
- Bit[48] GIoSup：Guest I/O保护支持。
- Bit[47] Reserved。
- Bit[46] MsiCapMmioSup：MSI能力寄存器MMIO访问支持。
- Bit[45] PerfOptSup：IOMMU性能优化功能。
- Bit[44] BlkStopMrkSup：阻止StopMark消息支持。
- Bits[43:42] MarcSup：MARC功能支持。00b=不支持，01b=4个三元组，10b=8个三元组，11b=保留。
- Bit[41] PPRAutoRspSup：PPR自动响应支持。
- Bit[40] PprOvrflwEarlySup：PPR日志溢出早期警告支持。
- Bits[39:38] DevTblSegSup：分段设备表支持。段数=2^DevTblSegSup。
- Bit[37] USSup：用户/特权页保护支持。
- Bits[36:32] PASmax[4:0]：最大PASID。最大值=2^PASmax-1。GTSup=0时无意义。

低32位（Bits[31:0]）：
- Bit[31] SATSSup：安全ATS支持。1=支持按DTE[SATS]控制安全/非安全ATS请求。
- Bit[30] Reserved。
- Bits[29:28] DualEventLogSup：双事件日志支持。00b=不支持，01b=支持无自动交换，10b=支持自动交换。
- Bits[27:26] Reserved。
- Bits[25:24] DualPprLogSup：双PPR日志支持。00b=不支持，01b=支持无自动交换，10b=支持自动交换。
- Bits[23:21] GAMSup[2:0]：Guest虚拟APIC模式支持。000b=仅中断重映射，001b=支持Guest虚拟APIC。
- Bits[20:18] SmiFRC[2:0]：SMI过滤器寄存器计数。000b=1个，001b=2个，010b=4个，011b=8个，100b=16个。SmiFSup=00b时无意义。
- Bits[17:16] SmiFSup[1:0]：SMI过滤器支持。00b=始终通过，01b=按SmiFEn和过滤器寄存器控制。

3.4.12 内存访问和路由（MARC）寄存器

内存访问路由和控制（MARC）功能提供快速静态转换机制，为I/O设备域到系统物理地址转换提供简单的只读或读/写访问权限。当I/O设备发出的DVA落在已编程并启用的MARC孔径内且访问权限匹配时，应用转换并绕过其余转换过程。

MARC功能支持由MMIO Offset 0030h [MarcSup]=1指示。

如果I/O设备域地址不落在任何已启用的MARC孔径内，或访问权限不匹配，则地址传递给其余转换过程。此功能允许受信任的系统软件为虚拟化系统中的低延迟图形处理分配多个连续物理范围。

每个孔径及其对应的转换地址和访问权限由软件通过编程一组三个MARC孔径寄存器来设置：MARC孔径基地址寄存器、长度寄存器和重定位寄存器。支持时MARC提供四个或八个这样的寄存器三元组。

实现说明：第一个修订版2.6实现提供四组（四个三元组）这些寄存器。

孔径0：基=200h，重定位=208h，长度=210h
孔径1：基=218h，重定位=220h，长度=228h
孔径2：基=230h，重定位=238h，长度=240h
孔径3：基=248h，重定位=250h，长度=258h

- Bits[11:0] Reserved。

■ MMIO Offset 02[08,20,38,50]h — MARC孔径[0-3]重定位寄存器

- Bits[63:52] Reserved。
- Bits[51:12] MarcRelocAddr：MARC孔径重定位地址。RW。复位00_0000_0000h。对应MARC基地址和长度寄存器定义的孔径的4KB对齐系统物理地址（SPA）位[51:12]。

4.1 转换缓存设计（Translation Cache Design）

当用户进程直接控制单个设备时，设备访问的总内存占用可能只是进程自身内存占用的适度部分。此外，用户进程直接了解特定设备，因此设备的访问模式很有可能被控制以获得良好的局部性。在这种情况下，实现性能的主要考虑是确保IOMMU转换缓存足够大。

相比之下，虚拟机Guest设备的潜在内存占用是Guest的全部内存。访问模式通常难以控制，由Guest操作系统的工作负载决定（HV可能对此没有具体了解），而且由不同Guest设备驱动程序和子系统控制的多种设备交互组成，具有多样的内存分配策略。对于非半虚拟化Guest，HV改善性能的策略可能是使用可用的最大页大小设置I/O页表，并假设IOMMU可以在多个设备之间共享相同的转换缓存条目。正是因为这个原因，IOMMU表结构包含可以为多个DeviceID共享的DomainID：由于IOMMU使用{DomainID, I/O虚拟地址}标记的转换缓存条目，它自动在分配到同一域的多个设备之间共享转换。

基于这些考虑，设计者应考虑IOMMU转换缓存的两级组织：

• 第一级应将DeviceID映射到{DomainID, I/O页表基地址}。大多数系统只有少数不同的DeviceID，因此第一级的容量可以很小。一个复杂之处是DeviceID不太随机且倾向于集群，因此为避免冲突，这一级应该是高关联度的或使用良好的DeviceID哈希函数。

• 第二级应将{DomainID, DVA}映射到{系统物理地址, 保护}。这一级应有（至少）数百个条目。这一级应在集合索引哈希中明确包含DomainID（而不仅仅将DomainID用作标签），以便具有相似内存布局的不同域不会争用相同的转换缓存条目。（服务器整合环境可能会创建许多具有非常相似内存布局的域。）

此外，由于IOMMU访问系统内存的延迟可能很高，实现者应考虑页目录缓存（PDC）来加速转换缓存未命中的处理。此缓存应将{DomainID, DVA}映射到页目录条目（PDE），以便IOMMU可以快速计算解决转换缓存未命中所需的最终PTE地址。这样，大多数转换缓存未命中可以通过IOMMU的单次内存访问来解决，而不需要完整的多级表遍历。页目录缓存还可以兼作大页转换缓存，因为对于大页，PDE也是PTE。

4.2 IOMMU拓扑

IOMMU的架构设计用于适应各种系统总线和拓扑结构。可以有多个IOMMU，位于系统总线中的不同位置。某些请求者ID信息可能在总线之间或总线类型之间的桥接器处丢失，因此将IOMMU放置在桥接器中是有利的。总线请求者ID到IOMMU DeviceID的映射取决于总线类型以及IOMMU在系统总线中的位置。在大多数其他方面，IOMMU的行为是与总线无关的。

IOMMU的最基本实现采用HyperTransport隧道的形式。

[图80：隧道中的IOMMU]

此方法的优点是可以轻松改装到现有系统设计中。主要限制是HyperTransport规范仅定义5位UnitID信息来标识请求发起者，因此IOMMU最多可以为31个下游设备提供不同的转换。如果下游节点包含任何桥接器，IOMMU无法区分桥接器之后的不同设备，因为桥接请求使用桥接器的UnitID。

一个可能的解决方案是在每个下游总线上包含单独的IOMMU；每个IOMMU可以被编程为不重写其UnitID证明已经通过另一个IOMMU的事务。软件必须了解系统拓扑以正确协调多个IOMMU。如果下游HyperTransport设备是PCIe根复合体或PCI-X主桥，该设备可以实现RequesterID映射能力来为PCIe或PCI-X设备分配特定的UnitID。

4.3 HyperTransport架构特有问题

本节讨论连接到HyperTransport链路的IOMMU的实现注意事项。

HyperTransport规范要求设备（特别是隧道和桥接器）以确保正确性和维持性能的方式与其他设备互操作。在其他要求中，HyperTransport设备必须提供某些事务排序保证并确保无死锁运行。

HyperTransport规范中的一个关键要求是已发送请求（posted requests）必须能够越过非已发送请求（non-posted requests）。然而，IOMMU的引入意味着已发送请求（例如内存写入）可能产生非已发送请求（I/O页表遍历），这些请求必须在已发送请求被允许进一步进行之前完成。

为避免死锁，IOMMU要求其I/O页表遍历请求使用专用虚拟通道。这确保IOMMU代表已发送请求进行的页表遍历可以完成，无论总线中其他非已发送流量的完成状态如何。IOMMU还要求主桥处理其请求时不产生对其他设备的请求。换言之，IOMMU的表结构必须仅位于系统内存中。

IOMMU可以与其他流量共享其虚拟通道，只要其他流量也保证能够向前推进。实际上，这意味着共享IOMMU页遍历通道的任何其他设备也必须将其非已发送流量限制为仅访问系统内存。

为了允许IOMMU支持具有不同同步能力的不同AMD处理器，IOMMU控制寄存器包含控制IOMMU发出的HyperTransport链路读请求中PassPW位、一致性位和同步位状态的位。

4.4 芯片组特定实现问题

同时实现IOMMU和旧版PCI或AGP桥的芯片组必须提供源标识，以唯一标识来自PCI或AGP总线的DMA流量。为提供此标识，IOMMU必须使用PCI或AGP桥的requesterID来执行来自旧版总线的DMA事务转换。设备旧版ID信息必须通过适当的ACPI或设备树表条目报告。详见第5.2.2节[I/O虚拟化定义块]。

4.5 软件和平台固件实现问题

由于IOMMU的灵活架构，任何单一系统软件实现不太可能使用所有功能、拓扑或选项。强烈建议以下约束：

• IOMMU应是根复合体设备（即直接出现在PCI树层次结构顶部的总线上）。
• 某些系统软件可能禁止IOMMU出现在PCI-to-PCI桥下。
• 为确保IOMMU被正确识别和配置，平台固件应在将控制权移交给系统软件时对IOMMU执行初始配置，使其对系统软件可访问。
• 平台固件应在第5章"I/O虚拟化ACPI表"中定义的ACPI表中描述IOMMU。该表必须包含识别、配置和访问IOMMU所需的所有信息。建议系统软件按以下优先级从高到低获取IOMMU硬件信息：Type11h表，如果不可用则使用Type10h IVHD表，最后如果没有其他属性可用或可访问，则评估IOMMU MMIO特性寄存器。
• 系统固件必须确保在电源管理状态变化中保留或恢复IOMMU配置。

5.1 IOMMU控制流

IOMMU启动过程经历多个阶段。通常，IOMMU PCI配置空间最初由平台固件配置，随后主要操作由（特权）系统软件完成。某些设置在设计时编程到IOMMU硬件中（如虚拟地址大小、物理地址大小、MSI中断）。某些硬件功能可由平台设计者覆盖，因此在ACPI设置中定义（如排除范围、远程IOTLB支持），例如当某些功能未包含在平台验证测试中或保留供平台固件使用时。

在系统复位时，IOMMU设置为默认状态。系统复位后，平台固件能够将必要的平台特定信息编程到IOMMU中，主要通过PCI配置空间寄存器（例如MMIO基地址）。一些额外设置由平台固件在MMIO空间中完成（例如隧道启用），而其他设置由系统软件在MMIO空间中完成（例如一致性位）。最后，系统软件必须初始化和管理分配在内存中的IOMMU控制和操作表（例如设备表）。其中一些控制和操作设置必须根据平台固件确定的策略进行配置，因此通过ACPI表传达给系统软件（包括选定的中断控制和系统管理控制）。

配置完成后，系统软件启用IOMMU，IOMMU开始处理来自外设的事务。从此时起，IOMMU在系统软件的控制之下。参见第114页"启动IOMMU"获取更多信息。

软件说明：虽然本规范允许将IOMMU放置在根复合体外部，但建议当前平台实现避免此类设计。本架构目前未定义热插拔包含IOMMU的外设控制器所需的ACPI方法或数据结构。

5.2 I/O虚拟化报告结构（IVRS）

I/O虚拟化报告结构（IVRS）是用于报告平台中每个IOMMU配置和能力的ACPI兼容数据结构。系统中包含一个或多个IOMMU时有单一的IVRS。

表82展示了IVRS格式的顶层视图。字节0-47提供头信息字段。Length字段给出包括头长度在内的整个数据结构长度。

在48字节头之后，IVRS包含一个或多个IVDB。IVDB在IVRS数据结构中紧密排列，不允许填充。定义了两种类型的IVDB：IVHD和IVMD。

定义了三种类型的IVHD块：
• Type 10h和11h：仅支持固定DeviceID的设备条目。
• Type 40h：增加对ACPI HID设备条目的支持。

Type 10h IVHD块为向后兼容而定义，支持向旧版系统软件报告旧版IOMMU属性。为使用IOMMU的全部功能集，系统软件必须支持Type 11h和40h IVHD的解码。IVHD Type11h包含所有相关IOMMU功能信息。建议系统软件从IVHD Type11h结构信息中的字段检测IOMMU功能，优先于Type10h块和MMIO寄存器中的信息。

定义了三种类型的IVMD：IVMD类型20h、21h和22h。IVRS中包含IVMD是可选的。每个IVMD（如果存在）提供与紧接其前的IVHD相关的内存范围信息。IVMD可用于固定DeviceID和ACPI HID命名的设备。

表82：I/O虚拟化报告结构（IVRS）格式
字节偏移00：Signature("IVRS") | Length(字节)
字节偏移08：Revision | Check | OEM ID
字节偏移16：OEM Table ID
字节偏移24：OEM Revision | Creator ID
字节偏移32：Creator Revision | IVinfo
字节偏移40：Reserved(0000_0000_0000_0000h)
字节偏移48：I/O虚拟化定义块（IVDB）

5.2.1 IVRS头字段

表83 IVRS字段定义：
- Signature(偏移00，4字节)："IVRS"，I/O虚拟化报告结构签名（ASCII）。
- Length(偏移04，4字节)：整个IVRS的字节长度，包括IVDB。
- Revision(偏移08，1字节)：01h或02h，IVRS格式修订号。
- Check(偏移09，1字节)：整个结构的校验和必须等于零。
- OEM ID(偏移10，6字节)：标识平台OEM。
- OEM Table ID(偏移16，8字节)：由OEM指定。
- OEM Revision(偏移24，4字节)：由OEM指定。
- Creator ID(偏移28，4字节)：创建该表的工具的供应商ID。
- Creator Revision(偏移32，4字节)：创建该表的工具的修订版。
- IVinfo(偏移36，4字节)：系统中所有IOMMU单元通用的I/O虚拟化信息（参见表85）。
- Reserved(偏移40，8字节)：保留，须为零。
- IVDB(偏移48+)：I/O虚拟化定义块。

注意：
1. IVHD和IVMD是IVDB的子类。
2. Revision=01h的IVRS仅允许type 10h和11h IVHD。
3. Revision=02h的IVRS允许包含type 10h、11h和40h IVHD。

表84 IVRS修订字段：
- 01h=Fixed格式，仅支持预分配DeviceID。支持Type 10h和11h IVHD块及所有IVMD块类型。
- 02h=Mixed格式，支持预分配DeviceID和ACPI HID设备命名。支持所有IVHD和IVMD块类型。
- 03h-FFh=保留。

表85 IVRS IVinfo字段：
- Bits[31:23] Reserved：须为零。
- Bit[22] HtAtsResv：ATS响应地址转换范围保留。参见Capability Offset 10h [HtAtsResv]定义。
- Bits[21:15] VAsize：虚拟地址大小。若支持Guest转换，定义GPA宽度。
- Bits[14:8] PAsize：系统物理地址宽度。
- Bits[7:5] GVAsize：Guest虚拟地址宽度。
- Bits[4:2] Reserved：须为零。
- Bit[1] DMA remap support：此标志指示OS/HV该IOMMU用于Pre-Boot DMA保护，OS加载后应重映射设备访问的内存。

5.2.2.2 I/O虚拟化内存定义（IVMD）块

平台固件可能需要根据其需求或硬件特性向系统软件传达内存使用要求。平台固件可使用IVMD块通知系统软件内存使用限制或要求。

IVMD可以是按设备的（指定DeviceID），也可以适用于所有设备（DeviceID被忽略）。IVMD块可引用IVHD条目中提供的别名范围、特殊设备和ACPI HID命名设备的DeviceID。

表108 IVMD格式：
偏移0：Type | Flags | Length | DeviceID | Auxiliary data
偏移8：Reserved(0)
偏移16：IVMD start address
偏移24：IVMD memory block length

表109 IVMD字段定义：
- Type(偏移0，1字节)：20h=所有外设，21h=指定外设，22h=外设范围
- Flags(偏移1，1字节)：内存块标志（参见表110）
- Length(偏移2，2字节)=32：IVMD块字节长度
- DeviceID(偏移4，2字节)：Type 20h保留，Type 21h为DeviceID，Type 22h为范围起始DeviceID
- Auxiliary data(偏移6，2字节)：Type 20h/21h保留，Type 22h为范围结束DeviceID（含）
- Reserved(偏移8，8字节)=0
- Start address(偏移16，8字节)：内存块起始系统物理地址
- Memory block length(偏移24，8字节)：内存块字节长度（软件可向上取整到4KB边界）

表110 IVMD标志定义：
- Bits[7:4] Reserved：须为零。
- Bit[3] ExclusionRange：排除范围。1b=包含在排除范围中。ExclusionRange=1b时忽略IR/IW/Unity。
- Bit[2] IW：写权限。1b=可写，0b=不可写。
- Bit[1] IR：读权限。1b=可读，0b=不可读。
- Bit[0] Unity：统一地址映射。1b=虚拟地址必须与物理地址相同。0b=可使用任何虚拟地址转换。

如果IR=0b且IW=0b，则内存范围不映射到外设设备地址空间，Unity字段须为0b。要永远阻止内存范围映射到任何外设地址空间，使用IR=0b、IW=0b和IVMD type=20h（所有设备）。

5.3 I/O虚拟化设备树

I/O虚拟化设备树（IVDT）是上述IVRS数据结构的替代方案。当不要求或不需要ACPI兼容性时（例如为Linux构建的x86平台），I/O设备信息应由平台固件以设备树格式提供。

5.3.1 I/O虚拟化设备树数据结构

以下伪代码定义了I/O虚拟化设备树（IVDT）结构：

每个IOMMU一个条目：
iommuX : iommu@0x###### {
    compatible = "amd,iommu-v2","amd,iommu-v3";
    ivrs,revision = <0x2>;
    ivrs,ivinfo = <#>;
    ivhd {
        compatible = "amd,ivrs-ivhd";
        type = <#>;
        flags = <#>;
        device_entries = <#>;
        iommu_dev_id = <#>;
        capabilities_offset = <#>;
        iommu_base_address = <# #>;
        pci_segment_group = <#>;
        iommu_info = <#>;
        iommu_attributes = <#>;
        iommu_ext_features = <# #>;
        device_entryX {
            compatible = "amd,ivrs-device-entry";
            entry = [ # # # # ... ];  /* 条目可以是4、8或22+字节 */
        };
    };
    /* 可选 */
    ivmdX {
        compatible = "amd,ivrs-ivmd";
        type = <#>;
        flags = <#>;
        devid_start = <#>;
        devid_end = <#>;
        start_address = <# #>;
        block_length = <# #>;
    };
};
