第九章：传输层

9.1 概述

每个 IBA 数据包包含一个传输头。传输头包含端节点完成指定操作所需的信息，例如将数据有效载荷传递给端节点内的适当实体（如线程或 IO 控制器）。本章定义了 IBA 使用的传输服务。

IBA 通道适配器的客户端通过操作由发送工作队列和接收工作队列组成的"队列对"（QP）与传输层通信。对于主机平台，传输层的客户端是 Verbs 软件层。客户端向这些队列发布缓冲区或命令，硬件从缓冲区传输数据或将数据传入缓冲区。

在本章中，发起操作（即向 fabric 注入消息）的 QP 称为请求者，接收消息的 QP 称为响应者。

创建 QP 时，它与四种 IBA 传输服务类型之一或非 IBA 协议封装服务关联。传输服务描述了可靠性程度以及 QP 传输数据的对象和方式。

四种 IBA 传输服务类型是：
1) 可靠连接（Reliable Connection, RC）
2) 可靠数据报（Reliable Datagram, RD）
3) 不可靠数据报（Unreliable Datagram, UD）
4) 不可靠连接（Unreliable Connection, UC）

非 IBA 协议封装服务是：
1) 原始 IPv6 数据报
2) 原始 Ethertype 数据报

第 1025 页的表 314"通道适配器属性"列出了主机通道适配器和目标通道适配器所需的服务。下面的表 34 比较了这五种传输服务类型的几个关键属性。

可靠传输服务使用序列号和确认消息（ACK/NAK）的组合来验证数据包传递顺序、防止重复数据包和乱序数据包被处理，以及检测丢失的数据包。在检测到错误（例如丢失数据包）后，请求者将重传丢失的数据包以及所有后续数据包。IBA 不支持选择性数据包重传，也不支持乱序接收数据包。

IBA 操作定义为包括一个请求消息以及（对于可靠服务）其对应的响应。因此，请求消息由请求者生成，响应（如果存在）由响应者生成。

请求消息由一个或多个 IBA 数据包组成。请求消息的数据包称为请求数据包。响应（RDMA READ 响应除外）恰好由一个数据包组成。响应也称为确认。响应数据包确认接收一个或多个数据包。响应可以确认接收从请求消息的一部分到多个请求消息的数据包。

不可靠传输服务不使用确认消息。但它们会生成序列号。这允许响应者检测乱序或丢失的数据包，并执行本地恢复处理。不可靠数据报的任何恢复处理的具体细节超出了 IBA 规范的范围。

9.2.2 保留传输功能操作码

为了未来扩展其传输层，IBA 提供了保留和制造商定义的 BTH OpCode。指定了两组未定义的 OpCode：一组用于 IBA 的未来修订，一组用于制造商特定功能。在设备被明确识别为支持这些操作码之前，不应在设备之间使用制造商定义的操作码。

9.2.3 请求事件（SE）- 1 位

请求者将此位设置为 1，以指示响应者应调用 CQ 事件处理程序。附加操作指南：
• SE 位应仅在 SEND、SEND with Immediate 或 RDMA WRITE with Immediate 的最后或唯一数据包中设置。
• 有关影响 HCA 的附加操作指南，参见第 11.4.2.2 节"请求完成通知"（第 627 页）。

SE 位不被视为数据包头验证的一部分，即接收到设置了此位但不满足调用要求的数据包不会导致生成 NAK。

C9-3：对于 HCA，如果入站请求数据包的 BTH 中请求事件位设置为 1，且附加 SE 操作指南有效，则应调用 CQ 事件处理程序。

o9-1：对于支持请求事件的 TCA，如果入站请求数据包的 BTH 中请求事件位设置为 1，且附加 SE 操作指南有效，则应调用 CQ 事件处理程序。

C9-4：响应者不应将 BTH 中的 SE 位视为数据包头验证的一部分。

除了在 SEND、SEND with Immediate 或 RDMA Write with Immediate 操作中使用外，SE 位还可以与 SEND with Invalidate 操作一起设置。在这种情况下，SE 位应仅在 SEND with Invalidate 的最后或唯一数据包中设置。在所有其他方面，SE 位的使用遵循与正常 SEND 操作定义的相同规则。

9.2.4 MigReq (M) - 1 位

用于通信迁移状态。设置为 1 表示连接或 EE 上下文已迁移；设置为 0 表示当前迁移状态没有变化。参见第 17 章"通道适配器"中的自动路径迁移（第 1016 页）。

9.2.5 填充计数（PadCnt）- 2 位

数据包有效载荷作为 4 字节数量的倍数发送。填充计数指示附加到数据包有效载荷的填充字节数——0 到 3。填充用于将有效载荷（有效载荷可以是零个或多个字节长度）"拉伸"为 4 字节的倍数。

9.2.6 传输头版本（TVer）- 4 位

指定此数据包使用的 IBA 传输版本。此版本适用于所有传输字段，包括 BTH、扩展头和不变 CRC——此字段设置为 0x0。如果接收方不支持指定的传输版本，则丢弃该数据包。

C9-5：使用 IBA 传输的请求者和响应者应生成 BTH:TVer = 0x0 的 IBA 传输数据包。

9.2.7 分区键（P_Key）- 16 位

P_Key 标识目标 QP（RC、UC、UD）或 EE 上下文（RD）所属的分区。

9.2.8 目标 QP（DestQP）- 24 位

此字段指定目标队列对（QP）标识符。

9.2.9 保留 8（Resv8）- 8 位

保留（可变）- 8 位。发送时为 0，接收时忽略。此字段不包含在不变 CRC 中。

C9-6：生成数据包时，发送方应将 Resv8 字段设置为零。接收方应忽略此字段。

9.2.10 AckReq (A) - 1 位

请求响应者在关联的 QP 上调度确认。

9.2.11 保留 7（Resv7）- 7 位

发送时为 0，接收时忽略。此字段包含在不变 CRC 中。

C9-7：生成数据包时，发送方应将 Resv7 字段设置为零。接收方应忽略此字段。

9.2.12 数据包序列号（PSN）- 24 位

此字段用于标识数据包在数据包序列中的位置。所有 IBA 请求者在发起数据包时应生成单调递增（模 2^24）的 PSN。根据传输服务类型和/或实现要求，响应者可以验证 PSN 以检测丢失的数据包。

9.3.3.1 虚拟地址（VA）- 64 位

缓冲区的起始地址。RDMA VA 可以从任何字节边界开始。

9.3.3.2 R_Key - 32 位

R_Key 具有以下属性：
• R_Key 充当访问指定内存地址和范围的保护键，即它是确保对目标内存正确访问的保护机制。响应者将 R_Key 与本地保护机制关联以验证请求者的访问权限。
• R_Key 必须导出给请求者——此过程（还包括导出起始虚拟地址和内存大小，即长度）超出本节范围。
• 可为 RDMA READ、RDMA WRITE 和 ATOMIC 的任意组合（包括无和全部）授予访问权限。
• 每个内存区域或窗口在任何给定时刻都有一个有效的 R_Key。虚拟连续的内存位置范围可以同时关联多个区域或窗口，每个都有关联的 R_Key。
• R_Key 可以导出到多个远程响应者。
• R_Key 仅用于 RDMA 和 ATOMIC 操作。R_Key 包含在数据包头中。

支持 RDMA 和/或 ATOMIC 操作的响应者应验证 R_Key、关联的访问权限和指定的虚拟地址。响应者还必须执行边界检查（即验证引用的数据长度不超过关联的内存起始和结束地址）。任何违规必须导致数据包被丢弃，对于可靠服务，还必须生成 NAK。

9.3.3.3 DMA 长度（DMALen）- 32 位

此字段指示远程 DMA 操作的长度（以字节为单位）。

C9-9：对于执行 RDMA 操作的 HCA，DMALen 字段中指定的最小长度为 0；最大长度为 2^31。

o9-3：如果 TCA 实现 RDMA 功能，则 DMALen 字段中指定的最小长度为 0；最大长度为 2^31。

9.4.1 SEND 操作

SEND 操作有时被称为推送操作或具有通道语义。两个术语都是指传输服务的 SW 客户端如何看待数据移动。通过 SEND 操作，数据传输的发起者将数据推送到远程 QP。发起者不知道数据在远程节点上将被放置在哪里。远程节点的通道适配器将数据放入该 QP 的下一个可用接收缓冲区中。在 HCA 上，接收缓冲区由 QP 接收队列头部的 WQE 指向。

SEND 操作被称为具有通道语义，因为它移动数据的方式类似于大型机 IO 通道——数据被标记有鉴别器（对于 IBA，鉴别器是目标 LID 和 QP 号），目标根据鉴别器选择放置数据的位置。

SEND 操作移动单条消息。对于 RC、RD 和 UC 传输服务，此消息可能长于单个数据包。消息大小范围从零字节到 2^31 字节。

C9-10：请求者生成的 SEND 操作大小应在零到 2^31 字节（含）之间。

C9-11：对于 HCA 中的 RC 和 UC 传输服务，长于 PMTU 的请求消息应分段为 PMTU 大小的段，通过多个数据包传输。类似地，HCA 响应者应将此类数据包重新组装为单条消息。

o9-4：对于 HCA 中的 RD 传输服务，长于 PMTU 的请求消息应分段为 PMTU 大小的段，通过多个数据包传输。类似地，HCA 响应者应将此类数据包重新组装为单条消息。

o9-5：对于 TCA 中的 RC、UC 和 RD 传输服务，长于 PMTU 的请求消息应分段为 PMTU 大小的段，通过多个数据包传输。类似地，TCA 响应者应将此类数据包重新组装为单条消息。

C9-12：对于不可靠数据报传输服务，SEND 操作应仅由单数据包消息组成（即消息数据有效载荷限制为请求者和响应者之间 PMTU 的最大值，即 256、512、1024、2048 或 4096 字节）。

SEND 操作可以（由客户端决定）在每条发送消息中包含 4 字节即时数据。如果包含，即时数据包含在 SEND 操作最后一个数据包的附加头字段（即时数据扩展传输头或 ImmDt）中。

图 75 SEND 操作示例

SEND 操作的注意事项：
• BTH OpCode 字段确定 SEND 消息的开始和结束。
• 如果 SEND 消息小于或等于 PMTU，则使用 BTH OpCode "SEND Only" 或 "SEND Only with Immediate"。
• 如果 SEND 消息长度为零，则使用 BTH OpCode "SEND Only" 或 "SEND Only with Immediate"。此时没有数据有效载荷字段，但所有其他字段如图所示。
• 如果 SEND 消息大于 PMTU，则第一个数据包的 BTH OpCode 是 "SEND First"，最后一个数据包的 BTH OpCode 是 "SEND Last" 或 "SEND Last with Immediate"。
• 如果 SEND 消息大于两倍 PMTU，则第一个和最后一个之间的数据包使用 BTH OpCode "SEND Middle"。
• 消息中不具有 SEND Only、SEND Only with Immediate、SEND Last 或 SEND Last with Immediate 操作码的每个数据包应具有 PMTU 长度的数据字段。
• 响应者节点在具有 "SEND Last" 或 "SEND Last with Immediate" OpCode 的最后一个数据包到达之前不知道 SEND 消息的最终长度。
• 数据包序列号字段由响应者用于检测乱序或丢失的数据包。
• 如果整个消息不是 PMTU 的倍数，则消息的初始数据包携带完整 PMTU 字节数，最后一个数据包携带余数作为部分有效载荷。
• 对于给定请求节点的 QP，一旦开始多数据包 SEND 操作，在 "SEND Last" 或 "SEND Last with Immediate" 数据包之前不得生成其他请求数据包。

C9-13：多数据包消息不应与同一发送队列上的其他操作交错。

• 并非所有 SEND 消息都携带即时数据。如果携带，消息的最后或唯一数据包中包含一个特殊头。头的存在由 BTH 中的特殊 "SEND Last with Immediate Data" 或 "SEND Only with Immediate Data" OpCode 指示。
• 对于 HCA，SEND 消息的源或目标缓冲区没有对齐要求。对于 TCA 中的缓冲区，任何对齐要求是实现特定的。

C9-14：生成 SEND 操作的数据包时，请求者应在请求的每个数据包中至少包含这些头和字段：LRH、BTH、数据有效载荷、ICRC、VCRC。

C9-15：生成 SEND 操作的响应时，响应者应在响应中至少包含这些头和字段：LRH、BTH、AETH、ICRC、VCRC。

9.4.1.1 SEND with Invalidate

在大多数方面，SEND with Invalidate 的操作与正常 SEND 操作完全相同。重大变化是消息的最后（或唯一）数据包在 IETH（无效化扩展传输头）中携带一个 R_KEY。这是请求响应者无效化的 R_KEY。实际上，无效化功能搭载在正常 SEND 操作上。

关于 SEND with Invalidate 与正常 SEND 操作的注意事项：
• 如果搭载无效化操作的 SEND 消息小于或等于 PMTU，则使用 BTH OpCode "SEND Only with Invalidate"。
• 如果搭载无效化操作的 SEND 消息长度为零，则使用 BTH OpCode "SEND Only with Invalidate"。
• 如果搭载无效化操作的 SEND 消息大于 PMTU，则第一个数据包的 BTH OpCode 是 "SEND First"，最后一个数据包的 BTH OpCode 是 "SEND Last with Invalidate"。
• 响应者节点在具有 "SEND Last with Invalidate" OpCode 的最后一个数据包到达之前不知道消息的最终长度。

远程无效化操作用于禁止通过 IETH 中包含的 R_Key 访问内存。但是，无效化操作维护与被无效化的 R_Key 关联的内存转换和保护资源。SEND with Invalidate 操作不禁止对 R_Key 引用的内存的所有访问，它只是意味着被无效化的 R_Key 不能再用于访问那些内存区域或内存窗口。引用同一内存的其他 R_Key 仍然有效。

其目的是无效化操作由通道接口执行，这可能包括硬件和传输层硬件之上的软件驱动程序。因此，IETH 不作为正常传输数据包头验证的一部分被验证。无效化操作不改变引用内存的状态。

9.4.1.1.1 无效化操作排序

o9-5.2.1：对于支持 SEND with Invalidate 的任何 HCA，在接收到 IETH 后，无效化操作不得在正常传输头验证检查成功完成之前发生。

如果包含 IETH 的数据包不是正常数据包验证过程描述的有效数据包，则不得发生无效化。但是，由于无效化操作不由传输层执行，无效化操作可以在传输级确认生成之前或之后发生，但无论如何，传输级确认（ACK/NAK）不反映无效化操作的成功或失败。

SEND with Invalidate 操作的排序规则：
1) 任何 SEND with Invalidate 操作与任何后续操作之间没有排序保证。
2) 正常排序规则适用于 RDMA WRITE、SEND 或 ATOMIC 操作后跟 SEND with Invalidate 操作。
3) SEND with Invalidate 操作可能影响先前的 RDMA READ 操作。请求者应使用 Fence 属性确保先前未完成的 RDMA READ 操作在发起后续 SEND with Invalidate 操作之前已完成。
4) 确认始终由响应者按序返回，请求者的 WQE 始终按序完成。

9.4.1.1.2 响应者 - R_Key 验证

传输层不在传输级数据包头验证期间验证 IETH 的 R_Key 字段。通道接口保留在执行无效化之前验证 IETH 的 R_Key 字段的责任。

9.4.1.1.3 远程内存无效化的 R_Key 验证

o9-5.2.2：对于支持远程无效化的 CA，在执行远程无效化操作之前，必须验证 IETH 中包含的 R_Key。要被视为有效，以下所有条件必须为真：
1) R_Key 不得处于无效状态。
2) 如果实现支持 L_Key 和 R_Key 的单一键空间，则 IETH 中的 R_Key 不得与保留 L_Key 相同。
3) 对于 HCA，R_Key 不得引用通过注册内存区域或重新注册内存区域创建的内存区域。
4) 对于 HCA，R_Key 不得引用共享内存区域。
5) 对于 HCA，R_Key 不得引用与接收 IETH 的 QP 关联的不同保护域的内存区域。
6) 对于 HCA，R_Key 不得引用类型 1 内存窗口。
7) 对于 HCA，R_Key 不得同时处于有效状态且引用与接收 IETH 的 QP 不同的 QP 关联的类型 2A 内存窗口。
8) 对于 HCA，R_Key 不得同时处于有效状态且引用与接收 IETH 的 QP 不同的保护域或 QP 关联的类型 2B 内存窗口。
9) 对于 HCA，R_Key 不得同时处于空闲状态且引用与接收 IETH 的 QP 的保护域不同的保护域关联的类型 2 内存窗口。

注意：如果 L_Key 及其伴随的 R_Key 处于空闲状态，则远程无效化操作不会改变 L_Key 及其伴随的 R_Key 状态。

9.4.2 RESYNC 操作

RESYNC 操作仅支持可靠数据报传输服务。RESYNC 本质上与零长度可靠数据报 SEND-only 请求相同，但具有几个独特属性：
1) RESYNC 由请求者用于强制响应者将其预期 PSN 重置为请求者定义的值。
2) RESYNC 请求携带零长度数据有效载荷。
3) 响应者必须接受 RESYNC 请求，即使当前执行的请求尚未完成。
4) RESYNC 请求本身不直接消耗请求者侧的发送 WQE，也不消耗响应者侧的接收 WQE。

C9-15.a1：RESYNC 请求应携带零长度数据有效载荷。

9.4.3 RDMA WRITE 操作

RDMA WRITE 操作由请求节点用于写入目标节点的虚拟地址空间。消息大小在零到 2^31 字节（含）之间，写入目标 QP 虚拟地址空间的连续范围（不一定是物理内存的连续范围）。

C9-16：对于执行 RDMA WRITE 操作的 HCA 请求者，RDMA WRITE 消息的长度（如 RETH:DMALen 字段所反映）应在零到 2^31 字节（含）之间。

在允许传入 RDMA WRITE 之前，目标节点首先为目标 QP（或 QP 组）的访问分配内存范围。目标的通道适配器将 32 位 R_Key 与此内存区域或窗口关联。对于 HCA，verbs 层将此称为注册内存区域——参见第 468 页 10.6"内存管理"。

目标将虚拟地址、长度和 R_Key 通信给希望访问该内存区域的任何其他主机。地址和 R_Key 的通信由客户端上层协议完成——交换超出 IBA 范围。

C9-17：与 SEND 操作一样，HCA 请求者应将大于 PMTU 的 RDMA WRITE 消息分段为多个数据包。

如果由 verbs 层指定，即时数据包含在 RDMA WRITE 消息的最后一个数据包中。即时数据不写入目标虚拟地址范围，而是在成功处理最后一个 RDMA WRITE 数据包后传递给客户端。

图 76 RDMA WRITE 操作示例

C9-18：对于 HCA RDMA WRITE 请求，多数据包消息不应与同一发送队列上的其他操作交错。

C9-19：生成 RDMA WRITE 请求时，HCA 请求者应在每个请求数据包中至少包含以下头和字段：LRH、BTH、数据有效载荷、ICRC、VCRC。请求的第一个（或唯一）数据包还应包含 RETH。

C9-20：生成 RDMA WRITE 响应时，HCA 响应者应在每个响应数据包中至少包含以下头和字段：LRH、BTH、AETH、ICRC、VCRC。

9.4.4 RDMA READ 操作

RDMA READ 操作与 RDMA WRITE 操作类似。它们允许请求节点读取远程节点上虚拟连续的内存块。与 RDMA WRITE 一样，响应节点首先允许请求节点访问其内存。响应者将虚拟地址、长度和 R_Key 传递给请求者用于 RDMA READ 请求数据包。

单个 RDMA READ 请求可以读取零到 2^31 字节（含）的数据。

C9-21：对于响应 RDMA READ 请求的 HCA，如果请求的数据大小大于 PMTU，响应者应将数据分段为 PMTU 大小的数据段，作为多个 RDMA READ 响应数据包传输。数据在请求节点的内存中重新组装。

C9-22：对于使用 RDMA 操作的 HCA 请求者，请求的 RDMA READ 数据长度（如 RETH:DMALen 字段所反映）应在零到 2^31 字节（含）之间。

图 77 RDMA READ 操作示例

RDMA READ 操作注意事项：
• 如果读取长度大于 PMTU，单个请求数据包将导致多个读取响应数据包。
• BTH OpCode 字段确定 RDMA READ 确认消息的开始和结束。
• 如果 RDMA READ 请求消息请求零字节传输，则使用 BTH OpCode "RDMA READ Response Only"。
• 如果 RDMA READ 确认消息小于或等于 PMTU，则使用 BTH OpCode "RDMA READ Response Only"。
• 如果 RDMA READ 消息大于 PMTU，则第一个数据包的 BTH OpCode 是 "RDMA READ Response First"，最后一个数据包的是 "RDMA READ Response Last"。
• 发起 RDMA READ 请求数据包后，请求节点可以发送额外的请求数据包而无需等待响应数据包返回。
• 特定 QP 在任何时刻可以未完成的 RDMA READ 请求的最大数量在连接建立时协商。
• RDMA READ 数据包从不携带即时数据。

RDMA READ 请求在请求者未收到正确响应时会被重试。
• 重试的 RDMA READ 请求不需要从与原始 RDMA READ 相同的地址开始，也不需要具有相同的长度。
• 响应者验证重试请求的 R_Key 和 RDMA READ 虚拟地址。
• 重试的 RDMA READ 的 PSN 必须在重复 PSN 区域内。

C9-23：生成 RDMA READ 请求时，HCA 请求者应在其请求数据包中至少包含以下头和字段：LRH、BTH、RETH、ICRC、VCRC。

C9-24：生成 RDMA READ 响应时，HCA 响应者应在每个响应数据包中至少包含以下头和字段：LRH、BTH、数据有效载荷、ICRC、VCRC。如果响应数据包 BTH:Opcode 是 "RDMA READ Response First"、"RDMA READ Response Last" 或 "RDMA READ Response Only"，数据包还应包含 AETH。如果响应数据包 BTH:Opcode 是 "RDMA READ Response Middle"，则不应包含 AETH。

9.4.5 ATOMIC 操作

ATOMIC 操作在远程节点的指定地址执行 64 位操作。这些操作以原子方式读取、修改和写入目标地址，并保证同一 CA 上其他 QP 对此地址的操作不会在读取和写入之间发生。原子性保证的范围可以可选地扩展到其他 CPU 和 HCA。

ATOMIC 操作使用与 RDMA READ 和 WRITE 相同的远程内存寻址机制。请求数据包中指定的虚拟地址在 ATOMIC 操作目标的远程 QP 的地址空间中。

ATOMIC 操作由两种数据包类型组成："ATOMIC Command" 请求数据包和 "ATOMIC Acknowledge" 响应数据包。

1) ATOMIC 操作仅受可靠连接和可靠数据报传输服务支持。
2) ATOMIC 操作不支持即时数据。
3) 强烈建议 ATOMIC 操作支持严格在硬件中提供。
4) ATOMIC Command 请求数据包中的虚拟地址应自然对齐到 8 字节边界。响应 CA 检查此项，如果未自然对齐则返回 Invalid Request NAK。

IBA 定义了以下 ATOMIC 操作：

• FetchAdd（取回并加）
FetchAdd ATOMIC 操作告诉响应者在响应者内存中自然对齐的虚拟地址处读取 64 位缓冲区值，使用 AtomicETH 中的 64 位 Add Data 字段执行无符号加法，并将结果写回同一虚拟地址。响应者的操作应是原子的。

FetchAdd 操作以目标内存的字节序格式执行。原始远程数据从目标内存的字节序格式转换后返回。字段在线路上以大端格式传输。

请求者指定：远程数据地址和 R_Key、加数据。
确认数据包返回：原始远程数据。

操作后，响应者指定虚拟地址处的内存包含原始值和 AtomicETH 头中 Add 字段的无符号和。

• CmpSwap（比较并交换）
CmpSwap ATOMIC 操作告诉响应者在响应者内存中自然对齐的虚拟地址处读取 64 位值，将其与 AtomicETH 头中的 Compare Data 字段进行比较，如果相等，则将 AtomicETH 头中的 Swap Data 字段写入同一虚拟地址。如果不相等，响应者内存的内容不变。在任何情况下，从虚拟地址读取的原始值都返回给请求者。

请求者指定：远程数据地址和 R_Key、写入（交换）数据、比较数据。
确认数据包返回：原始远程数据。

o9-15：生成 ATOMIC 操作请求时，请求者应至少包含 LRH、BTH、AtomicETH、ICRC 和 VCRC。

o9-16：响应 ATOMIC 操作请求时，响应者应在其响应数据包中至少包含 LRH、BTH、AETH、AtomicAckETH、ICRC 和 VCRC。

9.4.5.1 原子性保证

o9-17：ATOMIC 操作在响应者节点上的读取/修改/写入的原子性应在同一 CA 上其他 QP 的并发原子访问存在时得到保证。

o9-18：CA 可以可选地在来自其他 CA、IO 设备和 CPU 的并发内存访问存在时保证 ATOMIC 操作的原子性。对于 HCA，Verbs 层应报告是否支持此增强的原子性保证。

9.4.5.2 ATOMIC 确认生成和排序规则

1) 对于请求者，ATOMIC 操作在响应数据包返回时被视为完成。
2) 如果 RDMA READ 工作请求在 ATOMIC 操作工作请求之前发布，则原子操作可能在先前的 RDMA READ 读取其数据之前执行其远程内存操作。可以通过在 ATOMIC 操作工作请求上设置 fence 修饰符来确保严格排序。
3) 当请求序列到达 QP 时，ATOMIC 操作仅在先前（非 RDMA READ）请求访问内存之后和后续请求访问内存之前访问内存。
4) 每个 ATOMIC 操作请求需要显式响应和确认消息。

9.4.5.3 错误行为

响应者利用供应商特定的资源和设施来实现 ATOMIC 操作和 RDMA READ，以及促进重试的 ATOMIC 请求。请求者有责任确保所有未确认的 ATOMIC 操作和 RDMA READ 合计不超过接收者资源。

响应节点保存回复数据、PSN 和指示存储数据来自 ATOMIC 操作的标志。此保存的数据用于为重试的 ATOMIC 操作生成响应。

ATOMIC 操作保证最多执行一次。如果 ATOMIC 操作未在目标上执行，会通过适当的 NAK syndrome 响应报告给发送者。

与所有操作的重试一样，如果响应 CA 实际上已执行了请求，它将仅再次确认请求，而不会重新执行 ATOMIC 操作。这是必要的，因为 ATOMIC 操作不是幂等的。

9.4.6 保留和制造商定义的传输功能操作码

IBA 有两种机制用于未来扩展其传输层：
• 保留和制造商定义的 BTH OpCode
可以通过定义新的 BTH OpCode 来扩展 IBA 传输层功能。指定了两组未定义的 OpCode。一组用于 IBA 的未来修订，一组用于制造商特定功能。

9.5 事务排序

本节定义了给定 QP 传输、执行和完成事务的排序规则：

C9-25：请求者应按工作队列元素（WQE）发布的顺序传输请求消息。

C9-26：对于分段为 PMTU 大小数据包的消息，数据有效载荷应使用与 WQE 定义的数据段相同的顺序。

来自给定源 QP 到给定目标 QP 的数据包在 fabric 中沿相同路径传输，并按注入的顺序接收。

C9-27：对于 HCA 上的可靠服务，所有确认数据包应是强排序的，即所有先前的 RDMA READ 响应和 ATOMIC 响应应在后续 SEND、RDMA WRITE 响应、RDMA READ 响应或 ATOMIC 操作响应之前注入 fabric。

C9-28：响应者应按接收到消息的顺序执行 SEND 请求、RDMA WRITE 请求和 ATOMIC 操作请求。

• 应用程序不应依赖于消息内数据写入内存的顺序。

C9-29：接收方的完成按发送顺序（仅适用于 SEND 和 RDMA WRITE with Immediate），并不意味着先前的 RDMA READ 已完成，除非被请求者设置了 fence。

C9-30：请求者应按 WQE 传输的顺序完成 WQE。

C9-31：设置了 fence 属性的工作请求应阻塞，直到所有先前的 RDMA Read 和 Atomic WR 完成。

C9-32：所有 WQE 应按发布顺序完成，与其执行顺序无关。

o9-20：应用程序不应依赖于响应者处 RDMA WRITE 缓冲区的内容，直到以下情况之一发生：
• 最后一个 RDMA WRITE 请求数据包的到达和完成（当与即时数据一起使用时）。
• 后续 SEND 消息的到达和完成。
• 后续 ATOMIC 操作对内存元素的更新。

o9-21：应用程序不应依赖于请求者处 RDMA READ 目标缓冲区的内容，直到对应 WQE 完成。

o9-21.a1：应用程序不应依赖于响应者处 SEND 缓冲区的内容，直到其完成。

C9-33：应用程序不应依赖于接收队列缓冲区的内容，直到对应的接收 WQE 完成。

9.6 数据包传输头验证

数据包传输头验证在从较低 IBA 层传递到传输层的每个数据包上进行。目的是确定入站数据包是否可以与特定队列对关联。如果不能，则静默丢弃数据包。数据包传输头验证仅适用于使用 IBA 传输的数据包。

C9-34：传输层应根据本节的要求验证使用 IBA 传输的所有数据包的数据包头。如果 LRH:LNH 字段的 msb 设置为 1，则认为数据包使用 IBA 传输。如果 LRH:LNH 字段的 msb 设置为零，则数据包是原始数据包。

C9-35：对于每个使用 IBA 传输的入站数据包，CA 应根据图 81 所示的状态图验证数据包。

如果数据包可以与给定队列对关联，则通过将数据包的某些特征与存储在队列对（或 EE 上下文，对于可靠数据报服务）中的上下文信息进行比较来进行进一步验证。此级别的验证在第 9.7 节"可靠服务"和第 9.8 节"不可靠服务"中描述。

在本节中，短语"静默丢弃数据包"的使用意味着：
• 不返回确认消息。
• 响应者不消耗接收 WQE。
• 不执行错误请求数据包。
• 在错误请求之前接收到的任何请求数据包正常执行和完成。
• 响应者不更新其预期 PSN。
• 响应者继续等待有效的入站请求数据包。

对于请求者：
• 没有发送 WQE 因静默丢弃的数据包而完成。
• 不因静默丢弃的数据包采取直接操作，但错误计数器可能递增。
• 静默丢弃的数据包不计入满足传输定时器的目的。

C9-47.1.1：如果目标 QP 配置为 RC、UC 或 RD 传输服务，则应在传输层验证 SGID 和 DGID。未通过这些有效性检查的数据包必须静默丢弃。

9.7 可靠服务

可靠服务提供一种保证，确保消息从请求者传递到响应者最多一次、按序且不损坏。可靠服务的关键要素包括：用于检测损坏数据的保护方案（CRC）、允许请求者确认消息已成功传递的确认机制、用于检测丢失数据包并允许请求者将响应与请求关联的数据包编号机制，以及用于检测丢弃或丢失确认消息的定时器。

本节讨论确认和数据包序列编号机制。用于检测数据包损坏的CRC机制不在此处讨论。注意，CRC在较低协议层进行检查，可能导致数据包在传递到传输层之前被丢弃。这些丢弃的数据包最终可能在传输层被检测为序列错误。

• 可靠服务的特征
  • 消息最多传递一次，按序且在没有不可恢复错误的情况下不损坏。
  • 每条消息都被显式或隐式确认。

• 可靠服务的类型
  • 可靠连接
  • 可靠数据报

• 可靠性机制
  • ACK/NAK 协议
  • 数据包序列号（PSN）

• 响应者在以下情况下认为操作完成：
  • 在 BTH 中接收到有效的"Last"或"Only" OpCode，
  • 按正确的 PSN 顺序接收到组成消息的所有数据包，
  • 有效载荷已提交到本地故障区域（SEND 或 RDMA WRITE），
  • 对于 RDMA READ 或 ATOMIC 操作，响应已提交到线路，
  • 请求最后一个数据包的确认数据包已提交到线路（包括 RDMA READ 响应的适当字段）

• 请求者在以下情况下认为操作完成：
  • 已接收并提交到本地内存的所有响应数据包（对于 RDMA READ 或 ATOMIC 操作），
  • 已接收并验证确认消息。

C9-60：在请求者可以认为 WQE 完成之前，请求者必须等待必要的响应到达。如果请求者需要显式响应以完成给定的 WQE，则请求者应负责采取必要步骤确保所需的响应即将到来。

有几种机制可用于实现此目的，例如：
1) 在每条消息的最后一个数据包上设置 AckReq 位，从而保证响应者将生成所需的显式响应，
2) 在需要显式响应的消息的最后一个数据包上设置 AckReq 位，
3) 如果未为需要显式响应的请求设置 AckReq 位，请求者可以重试请求（设置 AckReq），从而要求响应者返回响应，
4) 如果未为需要显式响应的请求设置 AckReq 位，请求者可以发送 NOP 命令（例如长度为零的 RDMA WRITE 请求）并设置 AckReq 位。此策略仅在响应者支持 RDMA WRITE 时有效。

使用这些或其他策略的选择取决于实现。

9.7.1 数据包序列号（PSN）

PSN 在基本传输头（BTH）中为所有数据包传输。它们用于检测丢失或乱序的数据包，对于可靠服务，还用于将响应数据包与给定的请求数据包关联。每个 IBA QP 由一个发送队列和一个接收队列组成；同样，EE 上下文有一个发送侧和一个接收侧。请求者的发送队列上的 PSN 与响应者对应接收队列上的 PSN 之间存在关系。因此，QP（或 EE 上下文）的每一半维护一个独立的 PSN；给定队列对的发送队列和接收队列上使用的 PSN 之间或不同连接之间没有关系。

在响应者端必须检测和解决两种异常情况以确保可靠操作。这两种情况是重复数据包和无效数据包。

1) 重复数据包。如果请求者将请求数据包多次注入 fabric，响应者可能会识别到重复数据包。当请求者检测到规定的恢复机制是重试操作的条件时，就会发生这种情况。有两个主要原因可能导致超时条件，使请求者将给定的请求数据包多次注入 fabric：
   • 响应在到达请求者时延迟，原因可能是响应数据包在 fabric 中丢失或延迟，或者响应者在生成响应时经历了延迟，或者
   • 请求数据包在到达响应者时可能丢失或延迟。
   无论原因如何，响应者必须能够确定入站请求是否是先前已执行（或未执行）的重复请求，并做出适当响应。

   重复数据包也可能由于"幽灵"请求数据包而被响应者检测到。当请求数据包在 fabric 中延迟足够长的时间以导致请求者发生超时时，就会发生这种情况。请求者重新发送原始请求数据包，响应者对其生成正确的确认消息。稍后，原始（延迟的）数据包到达响应者，响应者将迟到的数据包解释为重复请求。例如，这可能在自动路径迁移期间发生。

2) 无效请求数据包序列。当响应者认为一个或多个请求数据包已在 fabric 中丢失时，就会发生此情况。

无效数据包和重复数据包之间的区别很重要，因为请求者的操作和响应者的操作对两种情况是不同的。

这两种情况必须由响应者（对于请求数据包）或请求者（对于可靠服务上的响应数据包）检测到。

除了重复数据包和无效数据包外，还有第三种情况，称为过时数据包（"TIME WAIT 数据包"）。如果与响应者的连接被拆除并在数据包仍在传输中时建立新连接，来自旧（过时）连接的数据包可能会到达响应者。响应者反过来可能将此过时的入站数据包解释为有效数据包，而实际上它是先前连接的残留。没有传输层机制来防范此情况；连接管理有责任避免在过时数据包仍有可能到达响应者之前重用 QP。这是通过将请求者和响应者 QP 置于"Time Wait"状态足够长的时间来确保 fabric 中任何过时数据包在重用这些 QP 之前已过期来实现的。

重复数据包通过基本传输头中携带的 24 位 PSN 字段与无效数据包区分，该字段允许唯一命名多达 16,777,216 个数据包。

C9-61：为了使响应者能够区分重复数据包和乱序数据包，给定的发送队列在任何给定时间应有不超过 8,388,608 个未完成的 PSN 系列。因此，发送队列在任何给定时间应有不超过 8,388,608 个未完成的数据包。这包括所有 SEND 请求数据包加上所有 RDMA WRITE 请求数据包加上所有 ATOMIC 操作请求数据包加上所有预期的 RDMA READ 响应数据包的总和。

因此，PSN 空间（由 16,777,216 个 PSN 的范围组成）被分为两个区域，每个区域占据 8,388,608 个 PSN 的范围，称为有效区域和无效区域。

响应者进一步将有效区域细分为预期 PSN 和重复区域。响应者的预期 PSN（ePSN）在第 9.7.4.1.2 节"响应者 - PSN 验证"中定义，简单描述为响应者期望在下一个要接收的新请求数据包中找到的 PSN。因此，重复区域被定义为整个有效区域，除了单个预期 PSN。简单来说，重复 PSN 是响应者先前已看到并执行的、落在有效区域内的 PSN。

9.7.1.1 可靠服务的 PSN 模型

C9-62：对于使用可靠连接服务的 HCA 请求者，请求者应在其生成的每个请求的每个数据包中插入 PSN。当响应请求时，响应者应在其生成的每个响应的每个数据包中插入 PSN。

o9-29：如果 TCA 实现可靠连接服务，或者如果 CA 请求者实现可靠数据报服务，请求者应在其生成的每个请求的每个数据包中插入 PSN。当响应请求时，响应者应在其生成的每个响应的每个数据包中插入 PSN。

除了 RDMA READ 响应的特殊情况外，请求数据包中的 PSN 与对应响应数据包中的 PSN 之间存在 1:1 的关系。

在一般 PSN 模型中，请求者计算要生成的下一个请求数据包的 PSN。这个计算的 PSN 称为下一个 PSN。在请求者生成新请求数据包时，"下一个 PSN"被复制到 BTH 中，从而成为当前 PSN。然后请求者计算新的"下一个 PSN"。

为了检测丢失或乱序的数据包，响应者也计算它期望在下一个入站请求数据包中找到的 PSN。这称为预期 PSN。

相反，当生成响应时，响应者计算响应 PSN 以将响应与给定请求关联。然而，由于第 9.7.5.1.2 节"合并确认消息"中描述的确认合并，请求者不一定能预测下一个响应数据包中可能出现的 PSN 范围。因此，请求者必须准备好接受一系列响应 PSN 中的任何一个。该范围由最旧的未确认请求数据包的 PSN 和最近发送的请求数据包的预期响应 PSN 限定。请求者评估入站响应数据包的 PSN 以确保它落在这两个极端之间。

9.7.2 ACK/NAK 协议

ACK/NAK 协议连同数据包序列号是可靠服务的基本组件，适用于可靠连接服务和可靠数据报服务。本节及后续各节描述该协议，提供一组管理 ACK 和 NAK 响应生成的规则，指定 ACK 和 NAK 代码，并指定请求者在接收到 ACK 或 NAK 响应时的所需响应。

ACK/NAK 协议的目的是允许请求者确定性地确认响应者是否正确接收了请求数据包。还提供了机制以确保完整消息被正确接收。这通过数据包序列号和数据包 OpCode（首包/中间包/末包/仅有包指示）的组合来实现。

由于响应数据包可能在 fabric 中丢失，ACK/NAK 协议要求请求者实现定时器以检测丢失的响应数据包。传输定时器也在本节中描述。

"确认"一词在本节中始终用于表示否定（NAK）或肯定（ACK）确认。通用术语"响应"用于描述响应者返回给请求者的确认。响应通过一个或多个确认数据包携带，可能包括（取决于原始请求消息）ACK 数据包、NAK 数据包、RDMA READ 响应或 ATOMIC 操作响应。

以下是管理 ACK/NAK 协议的规则摘要：
• 在可靠服务上接收的每个请求数据包都应被确认。
• 每个 RDMA READ 请求需要显式响应。带有正确格式的 ACK 扩展传输头（AETH）的 RDMA READ 响应被视为有效响应。ACK 扩展传输头出现在 RDMA READ 响应的第一个数据包和最后一个数据包（或唯一数据包）中。
• 每个 ATOMIC 请求需要显式响应。带有正确格式的 ACK 扩展传输头（AETH）和 ATOMIC ACK 扩展传输头（AtomicAckETH）的确认数据包被视为有效响应。
• 确认可以合并；即，单个确认数据包可以作为一个或多个先前请求数据包的确认。
• 确认数据包应按接收原始请求数据包的 PSN 顺序返回，包括 RDMA READ 响应。
• RDMA READ 响应由一个或多个数据包组成；所有其他响应恰好由一个数据包组成。

C9-63：对于使用可靠连接服务的 HCA 响应者，响应者应表现如下。响应者应确认接收到的每个请求数据包。响应者应为接收到的每个 RDMA READ 请求生成显式响应。响应者应为接收到的每个 ATOMIC 请求生成显式响应。响应者应按接收原始请求数据包的 PSN 顺序生成响应数据包，包括 RDMA READ 响应。

o9-30：如果 TCA 响应者实现可靠连接服务，或者如果 CA 响应者实现可靠数据报服务，响应者应表现如下。响应者应确认接收到的每个请求数据包。响应者应为接收到的每个 RDMA READ 请求生成显式响应。响应者应为接收到的每个 ATOMIC 请求生成显式响应。响应者应按接收原始请求数据包的 PSN 顺序生成响应数据包，包括 RDMA READ 响应。

9.7.3 请求者：生成请求数据包

本节指定请求者在生成请求数据包时的要求。

9.7.3.1 请求者侧 - 生成 PSN

C9-64：对于 HCA 中的可靠连接服务，请求者必须在每个请求数据包的 BTH:PSN 字段中放置一个称为当前 PSN 的值。

o9-31：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，则请求者必须在每个请求数据包的 BTH:PSN 字段中放置一个称为当前 PSN 的值。

在连接建立期间，传输层的客户端将下一个 PSN 编程为零到 16,777,215 之间的任何值。为了正确操作，响应者侧的初始预期 PSN 值必须加载相同的值。

C9-65：对于 HCA 中的可靠连接服务，由传输层客户端编程的初始 PSN 是请求者生成的第一个请求数据包中应出现的 PSN。

o9-32：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，则由传输层客户端编程的初始 PSN 是请求者生成的第一个请求数据包中应出现的 PSN。

此后，请求者计算下一个 PSN。计算取决于正在执行的操作（SEND、RDMA READ 等）和数据有效载荷的大小。

除一个例外外，请求者应为其生成的每个请求数据包将当前 PSN 值递增 1。唯一的例外是紧随 RDMA READ 请求消息之后的任何请求数据包。在这种情况下，紧随 RDMA READ 请求消息之后的请求数据包应具有比最后一个预期 RDMA READ 响应数据包的 PSN 大 1 的 PSN。通过这种方式，请求者在请求数据包的 PSN 序列中留下一个"空洞"，使得所有响应数据包将具有单调递增的 PSN。

因此，对于 RDMA READ 请求：
设 curr_PSN = RDMA READ 请求的 PSN
设 next_PSN = RDMA READ 请求之后的请求的 PSN
设 n = 预期的 RDMA READ 响应数据包数量
则 next_PSN = (curr_PSN + n) 模 2^24

由于请求者知道请求的 RDMA READ 数据的总长度和请求者与响应者之间的 PMTU，并且要求每个响应数据包（末包或仅有包除外）填充到完整的 PMTU 大小，请求者可以计算预期响应数据包的总数，从而计算紧随 RDMA READ 请求之后的请求的 PSN。

C9-66：对于使用可靠连接服务的 HCA 请求者，请求者应表现如下。对于除紧随 RDMA READ 请求消息之后的数据包之外的每个请求数据包，请求者应将下一个 PSN 值递增 1（模 2^24）。对于紧随 RDMA READ 请求消息之后的任何请求数据包，该数据包应具有比最后一个预期 RDMA READ 响应数据包的 PSN 大 1（模 2^24）的 PSN。

o9-33：如果 TCA 请求者实现可靠连接服务，或者如果 CA 请求者实现可靠数据报服务，请求者应表现如下。对于除紧随 RDMA READ 请求消息之后的数据包之外的每个请求数据包，请求者应将下一个 PSN 值递增 1（模 2^24）。对于紧随 RDMA READ 请求消息之后的任何请求数据包，该数据包应具有比最后一个预期 RDMA READ 响应数据包的 PSN 大 1（模 2^24）的 PSN。

表 39 请求者的下一个 PSN 计算

9.7.3.2 请求者 - 可靠数据报的特殊规则

9.7.3.2.1 RDD 检查

对于可靠数据报服务，任何给定的发送队列通过可靠数据报域（RDD）与 EE 上下文关联。每个发送队列和 EE 上下文有一个与其关联的 RDD。在发送请求之前，EE 上下文必须检查当前活动发送队列的 RDD。如果发送队列的 RDD 与 EE 上下文的 RDD 不匹配，则当前消息传输终止，并向发送队列指示超时条件。

o9-34：对于每个请求，请求者必须确认当前活动发送队列的 RDD 与所选 EE 上下文的 RDD 匹配。

9.7.3.2.2 RESYNC 生成

在某些条件下，请求者的 EE 上下文需要生成一种称为 RESYNC 请求的特殊形式的请求数据包。当请求者 EE 上下文选择中止当前请求消息时，就会发生这种情况。中止当前请求消息并生成后续 RESYNC 请求的过程在第 9.7.8 节"可靠数据报"中处理 QP 错误的子节中描述。

RESYNC 请求具有特殊属性，它强制响应者将其预期 PSN 重新初始化为请求者侧 EE 上下文定义的值。

以下段落规定了请求者在中止当前消息并生成 RESYNC 请求时需要使用的 PSN。它还规定了请求者在响应 RESYNC 请求时应期望的确认中的 PSN。

o9-35.a1：对于支持可靠数据报服务的 CA，当中止当前请求消息并生成 RESYNC 请求时，请求者侧 EE 上下文应将 RESYNC 请求的 PSN 设置为比传输当前请求中使用的最大 PSN（模 2^24）大 1 的递增值。此外，请求者应将其在响应中期望的 PSN 重置为相同的值。"传输当前请求中使用的最大 PSN"是指：
1) 分配给请求消息的任何数据包的（逻辑上）最大 PSN（模 2^24），包括请求消息的任何重试，或
2) 如果请求是 RDMA READ 请求，则为最后预期 RDMA READ 响应的 PSN。

例如，如果正在传输的当前请求由三个 PSN 编号为 4、5 和 6 的数据包组成，则 RESYNC 请求的 PSN 将设置为 7。

或者，考虑请求者正在发送由许多数据包组成的大型请求消息的情况。即使响应者在传输早期返回 NAK 但请求者已发送后续数据包，RESYNC 请求的 PSN 必须比发送的任何请求数据包的 PSN 大 1（模 2^24），无论 NAK 数据包何时生成或其 PSN 是什么。

在另一个示例中，如果正在传输的当前请求是 PSN 为 24 的 RDMA READ 请求，请求者预期接收 5 个响应数据包，则 RESYNC 请求的 PSN 将设置为 29，即比最后预期 RDMA READ 响应数据包的 PSN 大 1。

如果请求者由于超时或其他原因执行一次或多次重试，RESYNC 请求的 PSN 必须比所有先前尝试中发送的任何请求数据包或预期的响应数据包的 PSN 大 1（模 2^24）。

o9-35.a2：对于支持可靠数据报服务的 CA，请求者应在 RESYNC 请求中插入与当前请求（正在中止的请求）相同的源和目标 QP 编号。

9.7.4.1.1 响应者 - 可靠数据报检查的特殊规则

o9-38：对于 HCA 内的 RD，当入站数据包到达时，接收队列必须将其自身的 RDD 值与入站数据包到达的 EE 上下文的 RDD 值进行测试。如果它们不匹配，接收队列必须丢弃数据包并调度 NAK-Invalid RD Request。用于返回 NAK 的 P_Key 和 PSN 应由 EE 上下文提供。

o9-38.a1：如果在 CA 中实现可靠数据报服务，当入站数据包到达时，响应者应执行以下操作：
1) 如果响应者接收到具有"中间"或"末"opcode 的新请求数据包（即 PSN = 预期 PSN）（即响应者先前已接收到具有"首"opcode 的新请求数据包且尚未接收到具有"末"opcode 的新请求数据包），响应者应验证新请求数据包中包含的源和目标 QP 编号与最近接收到的"首"数据包的完全匹配。如果新"中间"或"末"请求数据包的源和目标 QP 编号与最近接收到的"首"请求的不匹配，响应者应忽略该数据包。
2) 如果响应者接收到重复请求数据包（即 PSN 在重复区域中），响应者应验证重复请求数据包中的源和目标 QP 编号与最近接收到的新请求的完全匹配。如果源和目标 QP 编号与最近接收到的新请求的不匹配，响应者应忽略该数据包。

o9-38.a2：如果在 CA 中实现可靠数据报服务，当入站 RESYNC 请求到达时，响应者应执行以下操作：
1) 如果响应者接收到 RESYNC 请求，它应接受该请求，而不考虑 RESYNC 请求中源和目标 QP 编号的值。
2) 如果响应者接收到 RESYNC 请求，它应接受该请求，而不考虑当前正在执行的请求的 opcode 序列状态（"首"、"中间"等）。
3) 如果 RESYNC 请求的 PSN 等于或逻辑上大于响应者的预期 PSN（即 RESYNC 请求是新请求），响应者应：
   a) 将其预期 PSN 设置为等于 RESYNC 请求的 PSN 加 1（模 2^24），
   b) 使用 RESYNC 请求的 PSN 作为响应的 PSN，
   c) 如果当前正在执行的请求不完整（即尚未到达具有"末"或"仅有"opcode 的请求数据包），则中止与当前正在执行的请求相关的任何处理。如果此类当前正在执行的请求是带有立即数据的 SEND RDMA WRITE（因此将消耗接收 WQE），则部分消耗的接收 WQE 应以错误方式完成。
4) 如果 RESYNC 请求的 PSN 逻辑上小于响应者的预期 PSN，响应者应将 RESYNC 请求视为重复请求，因此不应更改其预期 PSN。响应者应使用重复 RESYNC 请求的 PSN 作为响应的 PSN。

9.7.4.1.2 响应者 - PSN 验证

C9-74：对于 HCA 响应者中的可靠连接服务，在执行入站请求之前，响应者应通过将入站 BTH:PSN 与响应者的预期 PSN 进行比较来检查 PSN。PSN 应由响应者的接收队列检查。

o9-39：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，在执行入站请求之前，响应者应通过将入站 BTH:PSN 与响应者的预期 PSN 进行比较来检查 PSN。PSN 应由响应者的接收队列检查。

对于可靠数据报服务，PSN 由响应者的 EE 上下文检查。

在很大程度上，响应者响应请求时的行为基于对传入 PSN 的解释。

逻辑上，接收队列或 EE 上下文维护一个预期 PSN（ePSN）。这是响应者期望在其接收到的下一个新请求数据包的 BTH 中找到的 PSN。响应者用于计算其下一个预期 PSN 的规则与请求者在计算其下一个请求数据包中要插入的 PSN 值时使用的规则相同。

C9-75：对于 HCA 响应者中的可靠连接服务，响应者应使用第 9.7.3.1 节"请求者侧 - 生成 PSN"中给出的规则来计算其预期 PSN。

o9-40：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，响应者应使用第 9.7.3.1 节"请求者侧 - 生成 PSN"中给出的规则来计算其预期 PSN。

响应者的预期 PSN 在连接建立时由通信管理器初始化为零到 16,777,215 之间的任何值。为了正确操作，此初始值必须与请求者上加载的初始下一个 PSN 值匹配。

初始预期 PSN 只能由客户端在队列处于初始化状态时设置。客户端在任何其他状态下设置 PSN 的尝试可能被传输层忽略。

C9-76：对于 HCA 响应者中的可靠连接服务，HCA 应仅在接收队列处于正确配置以接收请求数据包的状态时更新其预期 PSN。例如，当队列对处于初始化状态时，传输不会修改预期 PSN。

o9-41：如果在 TCA 中实现可靠连接服务，响应者应仅在接收队列处于正确配置以接收请求数据包的状态时更新其预期 PSN。

o9-42：如果在 CA 中实现可靠数据报服务，响应者应仅在 EE 上下文处于正确配置以接收请求数据包的状态时更新其预期 PSN。

与其预期 PSN 相比，入站请求消息的实际 PSN 可能落入三个区域之一；它可能恰好等于响应者的预期 PSN，它可能逻辑上"小于"响应者的预期 PSN 从而落入图 86 所示的重复区域，或者它可能落在有效区域和预期 PSN "区域"之外，因此无效。

预期（新）请求：接收到的 PSN 恰好匹配响应者预期 PSN 的入站请求数据包是新请求数据包。

C9-77：对于 HCA 响应者中的可靠连接服务，新请求数据包应正常验证并根据执行顺序规则执行。一旦请求被执行，应按第 9.7.5 节"响应者：生成响应"中的规定调度响应。

o9-43：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，新请求数据包应正常验证并根据执行顺序规则执行。一旦请求被执行，应按第 9.7.5 节"响应者：生成响应"中的规定调度响应。

注意，不要求为每个入站请求数据包返回离散的确认数据包。

一旦接收到具有有效预期 PSN 的数据包，响应者通过计算新的预期 PSN 来推进其预期 PSN，并滑动有效区域窗口以反映新的有效 PSN 范围。

有效重复请求：落在有效区域内但不是预期 PSN 的 PSN 是有效重复请求数据包。

C9-78：对于 HCA 响应者中的可靠连接服务，响应者应按第 9.7.5.1.4 节"确认重复请求"中的规定响应有效重复请求。

o9-44：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，则响应者应按第 9.7.5.1.4 节"确认重复请求"中的规定响应有效重复请求。

表 41 总结了这些操作。

表 41 摘要：响应者对重复 PSN 的操作
重复请求消息 | 响应者操作
SEND 或 RDMA WRITE 或 RESYNC | 调度确认数据包
RDMA READ | 重新执行请求，调度响应
ATOMIC 操作 | 不重新执行请求，在验证请求后返回引用的 ATOMIC 操作请求的保存结果。

无效请求：实际接收的 PSN 在有效区域之外且不在预期"区域"中的数据包是无效请求。无效的 PSN 值通常表示一个或多个请求数据包已在 fabric 中丢失。

响应者对无效请求的详细行为如下：
• 错误的请求数据包不被执行。
• 在发出 NAK-Sequence Error 之前，必须执行并完成在错误请求之前接收到的任何请求数据包，因为它对先前未完成的 SEND 或 RDMA WRITE 请求充当隐式 ACK，对未完成的 RDMA READ 或 ATOMIC 操作请求充当隐式 NAK。
• 向请求者返回 NAK-Sequence Error。
• 响应者不更新其预期 PSN。

C9-79：对于 HCA 响应者中的可靠连接服务，当入站请求消息的实际 PSN 在有效区域之外（无效请求）时，响应者应返回 NAK-Sequence Error。在发出 NAK-Sequence Error 之前，必须执行并完成在错误请求之前接收到的任何请求数据包。

o9-45：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，且入站请求消息的实际 PSN 在有效区域之外（无效请求），则响应者应返回 NAK-Sequence Error。在发出 NAK-Sequence Error 之前，必须执行并完成在错误请求之前接收到的任何请求数据包。

响应者恢复等待具有等于其预期 PSN 或在其有效区域内的 PSN 的有效入站请求数据包。如果在等待有效新请求时，响应者接收到任何后续无效请求数据包，这些数据包将被静默丢弃；不返回任何 NAK。

C9-80：对于 HCA 响应者中的可靠连接服务，在生成 NAK-Sequence Error 后，响应者不应生成 ACK 或 NAK，直到它接收到有效的新请求或有效的重复请求。

o9-46：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，则在生成 NAK-Sequence Error 后，响应者不应生成 ACK 或 NAK，直到它接收到有效的新请求或有效的重复请求。

不要求队列停止或对于连接传输服务拆除连接。

9.7.4.1.3 响应者 - OpCode 序列检查

请求数据包必须符合有效 OpCode 序列调度表。对于可靠连接和可靠数据报服务，响应者应按如下方式检查组成请求消息的数据包 OpCode 序列：
1) 如果这是建立连接后的第一个数据包，则数据包 OpCode 必须指示"首"或"仅有"。
2) 如果最后一个有效数据包的 OpCode 指示"首"，则当前 OpCode 必须指示"中间"或"末"。它还必须匹配最后一个有效数据包中指定的操作类型（Send、RDMA、ATOMIC 操作）。如果当前 OpCode 指示"首"或"仅有"则为错误，因为这意味着前一条消息的最后一个数据包被遗漏。
3) 如果最后一个有效数据包的 OpCode 指示"中间"，则当前 OpCode 必须指示"中间"或"末"。它还必须匹配最后一个有效数据包中指定的操作类型。如果当前 OpCode 指示"首"或"仅有"数据包则为错误，因为这意味着前一条消息的最后一个数据包被遗漏。
4) 如果最后一个有效数据包的 OpCode 指示"末"，则当前 OpCode 必须指示"首"或"仅有"。如果当前 OpCode 指示"中间"或"末"则为错误，因为这意味着消息的第一个数据包被遗漏。
5) 如果最后一个有效数据包的 OpCode 指示"仅有"，则当前 OpCode 必须指示"首"或"仅有"。如果当前 OpCode 指示中间数据包或末数据包则为错误，因为这意味着消息的第一个数据包被遗漏。

C9-81：对于使用可靠连接服务的 HCA 响应者，响应者应检查组成请求消息的数据包 OpCode 序列是否符合表 42"有效 OpCode 序列调度表"中所示的调度表。如果响应者检测到无效的 opcode 序列，它应向请求者返回 NAK-Invalid Request。

o9-47：如果 TCA 响应者实现可靠连接服务，响应者应检查组成请求消息的数据包 OpCode 序列是否符合表 42 所示的调度表。如果响应者检测到无效的 opcode 序列，它应向请求者返回 NAK-Invalid Request。

o9-48：如果 CA 响应者实现可靠数据报服务，响应者应检查组成请求消息的数据包 OpCode 序列是否符合表 42 所示的调度表。如果响应者检测到无效的 opcode 序列，它应向请求者返回 NAK-Invalid Request。

在存在无效 OpCode 序列时的详细行为在第 9.9 节"错误检测和处理"中指定。

o9-49：此合规声明已过时并已删除。

9.7.4.1.4 响应者 OpCode 验证

C9-82：在执行入站请求之前，响应者应验证 BTH 的 OpCode 字段。

OpCode 针对以下特征进行检查：
• 请求的功能（Send、RDMA、ATOMIC）是否被此接收队列支持，
• 如果请求是 RDMA READ 或 ATOMIC 操作，是否有足够的资源可用来接收它。

当数据包向上传递到传输层时，BTH OpCode 字段[7:5]已被检查以确保请求的操作是针对可靠服务的。如果不是，则数据包被静默丢弃。此检查在第 9.6 节中指定。因此，在数据包到达队列对进行本节规则验证之前，已知它是针对可靠服务的请求。

C9-83：对于 HCA 响应者中的可靠连接服务，如果请求的功能不被此接收队列支持，则应返回 NAK-Invalid Request。

o9-50：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，如果请求的功能不被此接收队列支持，则应返回 NAK-Invalid Request。

C9-84：对于 HCA 响应者中的可靠连接服务，且 BTH OpCode 字段[4:0]指定可靠连接保留 opcode 或可靠数据报保留 opcode，应返回 NAK-Invalid Request。

o9-51：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，且 BTH OpCode 字段[4:0]指定可靠连接保留 opcode 或可靠数据报保留 opcode，则应返回 NAK-Invalid Request。

C9-85：对于 HCA 响应者中的可靠连接服务，如果 BTH OpCode 字段[4:0]指定首或中间请求数据包（例如 SEND First 或 RDMA WRITE Middle），则填充计数位应验证为 b00，表示不存在填充字节。如果填充计数位非零，应返回 NAK-Invalid Request。

o9-52：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，如果 BTH OpCode 字段[4:0]指定首或中间请求数据包，则填充计数位应验证为 b00。如果填充计数位非零，应返回 NAK-Invalid Request。

C9-86：对于 HCA 响应者中的可靠连接服务，如果没有足够的资源来接收新的 RDMA READ 或 ATOMIC 操作请求，则应返回 NAK-Invalid Request。

o9-53：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，且没有足够的资源来接收新的 RDMA READ 或 ATOMIC 操作请求，则应返回 NAK-Invalid Request。

返回 NAK-Invalid Request 的行为如下：
• 错误的请求数据包不被执行。
• 在发出 NAK-Invalid Request 之前，必须执行并完成在错误请求之前接收到的任何请求数据包。这很重要，因为 NAK 有效地合并了对早期未完成请求的响应，并充当对先前未完成的 SEND、RDMA WRITE、ATOMIC 操作或 RDMA READ 请求的隐式响应。
• 返回 NAK-Invalid Request。
• 响应者不更新其预期 PSN。

C9-87：对于 HCA 响应者中的可靠连接服务，在包含无效 opcode 的数据包之前接收到的任何请求数据包必须在响应者发出 NAK-Invalid Request 之前执行并完成。

o9-54：如果在 CA 中实现可靠数据报服务，或者如果在 TCA 中实现可靠连接服务，则在包含无效 opcode 的数据包之前接收到的任何请求数据包必须在响应者发出 NAK-Invalid Request 之前执行并完成。

9.7.4.1.5 响应者 R_Key 验证

R_Key 违规由以下任何或所有条件导致，针对 RDMA READ、RDMA WRITE 或 ATOMIC 操作：
• RETH 的 R_Key 字段无效（对于 RDMA READ 或 WRITE）
• AtomicETH 的 R_Key 字段无效（对于 ATOMIC 操作）
• 指定的虚拟地址和长度或访问类型超出与 R_Key 关联的本地定义限制。对于 RDMA WRITE 请求，长度检查基于每个数据包进行，基于 LRH:PktLen 字段。对于 RDMA READ 请求，长度检查基于 RETH:DMA Length 字段。
• 对于 HCA，R_Key 违规还包括第 10.2.3 节"保护域"中定义的保护域违规。

C9-88：对于使用可靠连接服务的 HCA 响应者，对于每个零长度 RDMA READ 或 WRITE 请求，不应验证 R_Key，即使请求包含立即数据。

o9-55：如果 HCA 响应者实现可靠数据报服务，或者如果 TCA 响应者实现可靠连接和 RDMA 功能，或者如果 TCA 响应者实现可靠数据报服务和 RDMA 功能，响应者应表现如下。对于每个零长度 RDMA READ 或 WRITE 请求，不应验证 R_Key，即使请求包含立即数据。

C9-89：如果响应者的接收队列检测到 R_Key 违规，应使用错误请求数据包的 PSN 向请求者返回 NAK-Remote Access Error。

C9-90：在包含 R_Key 违规的数据包之前接收到的任何请求数据包应在响应者发出 NAK-Remote Access Error 之前执行并完成。

9.7.4.1.6 响应者 - 长度验证

C9-91：应检查 LRH 的 PktLen 字段以确认接收缓冲区（由接收 WQE 指定）中有足够的空间可用。如果接收 WQE 定义的缓冲区不足以容纳入站 SEND 请求，则应返回 NAK-Invalid Request。

C9-92：还应通过将数据包长度与 OpCode 进行比较来验证数据包长度：
如果 OpCode 指定"首"或"中间"数据包，则数据包有效载荷长度必须为完整的 PMTU 大小。
如果 OpCode 指定"仅有"数据包，则数据包有效载荷长度必须在零到 PMTU 字节之间。
如果 OpCode 指定"末"数据包，则数据包有效载荷长度必须在 1 到 PMTU 字节之间。

注：CA 不需要验证 GRH 数据包长度。

C9-93：如果检测到长度无效的数据包，该请求应为无效请求。

响应者在这种情况下的行为在第 9.9.3 节"响应者侧行为"中指定。

除了检查 LRH:PktLen 字段外，还按如下方式检查 RETH 的 DMA Length 字段。

对于 RDMA WRITE 请求，响应者可以选择性地检查 RETH 中的 DMA Length 字段，以确保它不指定大于 2^31 字节的传输长度。它还可以在传输结束时验证数据包有效载荷的总和是否等于 RETH 中的 DMALen 字段。如果响应者检测到这些条件中的任何一个，它可以将请求视为无效请求。

C9-94：对于验证入站 RDMA READ 请求的 HCA，应检查 DMA Length 字段。如果请求大于 2^31 字节，则应返回 NAK-Invalid Request。

o9-56：如果 TCA 实现 RDMA 操作，则对于入站 RDMA READ 请求，应检查 DMA Length 字段。如果请求大于 2^31 字节，则应返回 NAK-Invalid Request。

9.7.4.1.7 响应者本地操作验证

有效的入站请求仍然可能由于响应者本地的故障而无法完成，例如访问本地内存时的本地内存转换错误。所有本地响应者错误都作为 NAK-Remote Operational Error 报告给请求者。

9.7.5 响应者：生成响应

9.7.5.1 响应者侧行为

本节指定响应者在生成确认消息时必须遵循的所需行为。

9.7.5.1.1 为确认消息生成 PSN

当响应者为每个请求生成响应时，它应通过在响应的 BTH 中插入 PSN 来标识与响应关联的请求。

这允许请求者将接收到的响应数据包与其请求关联。

C9-95：对于 SEND 请求或 RDMA WRITE 请求的响应，响应者应在响应的 PSN 字段中插入被确认的最近请求数据包的 PSN。

此处以及整个第 9 章中使用的"最近请求的 PSN"特指最近接收到的新请求数据包的 PSN。（这将其与最近接收到的重复请求的 PSN 区分开来）。因此，最近接收到的请求的 PSN 标记了响应者感知到的最大前进进度点，同时忽略重复请求。

由于合并确认的规则（在第 9.7.5.1.2 节中给出），连续响应数据包的 PSN 不一定是连续的。

C9-96：对于 HCA 对 RDMA READ 请求的响应，响应数据包的 PSN 必须是连续的且单调递增的，从原始 RDMA READ 请求消息的 PSN 开始。

o9-57：如果 TCA 实现 RDMA READ 功能，则对于每个 RDMA READ 响应，响应数据包的 PSN 必须是连续的且单调递增的，从原始 RDMA READ 请求消息的 PSN 开始。

o9-58：由于 ATOMIC 操作请求需要显式响应，且 ATOMIC 操作请求限于单个数据包，响应数据包的 PSN 必须与请求的 PSN 相同。

9.7.5.1.2 合并确认消息

不要求每个请求数据包都有唯一的离散响应。相反，响应者可以用单个确认数据包确认多个未完成的请求数据包。这称为确认合并。

给定的响应数据包如下确认先前的未完成请求（即 PSN 早于响应数据包 BTH 中的 PSN 的请求）：
1) 如果存在 PSN 早于响应数据包 BTH 中 PSN 的未完成 RDMA READ 或 ATOMIC 操作请求，则响应数据包暗示对最旧的此类未完成 RDMA READ 或 ATOMIC 操作请求的否定确认。在此类 RDMA READ 或 ATOMIC 操作请求之后发布到发送队列的任何请求不被确认。
2) 它暗示对 PSN 早于响应数据包 BTH 中 PSN 的所有未完成 SEND 或 RDMA WRITE 请求数据包的肯定确认，除非此类未完成的 SEND 或 RDMA WRITE 请求在如上所述的未完成 RDMA READ 或 ATOMIC 操作请求之后。
3) 如果给定响应是 RDMA READ 响应消息，则是 RDMA READ 响应消息的第一个（或唯一）数据包隐式确认先前的未完成请求。
4) RDMA READ 响应消息的最后一个（或唯一）数据包仅显式确认 RDMA READ 请求。

9.7.5.1.3 确认 RDMA READ 请求

RDMA READ 响应与正常响应不同，因为它包含数据有效载荷。

每个 RDMA READ 请求消息都需要离散确认，称为 RDMA READ 响应，由一个或多个数据包组成。

C9-97：对于 HCA，如果 RDMA READ 响应包含多于一个数据包，第一个和最后一个数据包必须包含 AETH。

o9-59：在实现 RDMA 的 TCA 中，如果 RDMA READ 响应包含多于一个数据包，第一个和最后一个数据包必须包含 AETH。

C9-98：对于 HCA，如果 RDMA READ 响应本身是单个数据包，则该数据包必须包含 AETH。

o9-60：如果 TCA 实现 RDMA 功能，且 RDMA READ 响应本身是单个数据包，则该数据包必须包含 AETH。

单数据包 RDMA READ 响应中包含的 AETH 既隐式确认先前的未完成请求，也显式确认 RDMA READ 请求本身。

C9-99：HCA 响应者应生成与 opcode 一致的 RDMA READ 响应数据包有效载荷长度：
1) opcode 为"RDMA READ response only"的数据包应为零到（PMTU）字节长。
2) opcode 为"RDMA READ response first"或"RDMA READ response middle"的数据包应恰好为（PMTU）字节长。
3) opcode 为"RDMA READ response last"的数据包应为 1 到（PMTU）字节长。
4) 允许零长度 RDMA READ 请求。
5) 对零长度 RDMA READ 请求的响应应由 opcode 为"RDMA READ response only"的单个数据包组成。

o9-61：如果 TCA 实现 RDMA 功能，它应生成具有上述合规声明中描述的有效载荷长度的响应数据包。

C9-100：如果 HCA 响应者在返回多数据包 RDMA READ 响应的过程中检测到错误，它应通过不传输任何错误有效载荷数据并将发生错误的数据包的 opcode 强制为"acknowledge"而不是"RDMA READ response last"的 opcode 来强制 RDMA READ 响应的提前终止。插入适当的 NAK 代码。

o9-62：如果 TCA 实现 RDMA 功能，且在返回多数据包 RDMA READ 响应的过程中检测到错误，它应通过不传输任何错误有效载荷数据并将发生错误的数据包的 opcode 强制为"acknowledge"来强制 RDMA READ 响应的提前终止。插入适当的 NAK 代码。

由于 RDMA READ 请求的宽松排序规则，允许响应者开始执行在 RDMA READ 请求之后到达的一个或多个 SEND 或 RDMA WRITE 请求。

C9-101：对于 HCA，在执行 RDMA READ 请求之后的任何请求之前，必须验证 RDMA READ 请求的头部字段。这些请求在发送未完成的 RDMA READ 响应之前不得被确认。

o9-63：在执行 RDMA READ 请求之后的任何请求之前，必须验证 RDMA READ 请求的头部字段。这些请求在发送未完成的 RDMA READ 响应之前不得被确认。

9.7.5.1.4 确认重复请求

在验证重复请求后，对重复请求数据包的响应如下：

C9-102：在验证重复请求后，如果重复数据包有效，响应者应生成响应。

C9-103：在处理重复请求的整个过程中，响应者不应更新其预期 PSN；它保持为重复请求到达之前的值。即使响应者在生成对重复请求的响应过程中检测到错误，这也是正确的。

在生成适当的响应（如以下段落所述）后，响应者恢复等待具有匹配其预期 PSN 的 PSN 的新入站数据包。

响应者在等待新入站数据包时可能会收到另一个重复请求。这是完全有效的，应被视为另一个重复请求。此外，由于它是重复请求，不要求下一个接收到的请求与第一个重复请求的 PSN 顺序相连。然而，响应者需要为重复请求的响应生成维护与正常新请求相同的排序规则。

C9-104：特别是，重复的 RDMA READ 或 ATOMIC 操作请求必须在返回后续重复 SEND 或 RDMA WRITE 请求的确认之前用显式响应确认。

要生成的响应取决于重复请求消息的功能：

• SEND、RESYNC 或 RDMA WRITE 请求

C9-105：对于接收到重复入站 SEND 或 RDMA WRITE 请求的 HCA，或接收到重复入站 SEND 请求的 TCA，响应者不应重新执行请求，只为重复数据包生成响应数据包，等待任何未完成的重复 RDMA READ 请求或 ATOMIC 操作请求的响应。

o9-64：如果 TCA 响应者实现 RDMA 功能，它不应重新执行 RDMA WRITE 请求，只为重复数据包生成响应数据包，等待任何未完成的重复 RDMA READ 请求或 ATOMIC 操作请求的响应。

确认消息的 PSN 必须是最近完成的新请求的 PSN。可以将此过程视为合并确认能力的逻辑扩展。实际上，请求者在接收到对重复请求的响应时，将其完全视为任何其他合并确认；任何未完成的重复 RDMA READ 或 ATOMIC 操作请求被视为被 NAK。在这种情况下，通过返回最近完成的请求的 PSN，响应者告诉请求者它认为已经看到并执行了重复请求和最近完成的请求之间的所有请求。

C9-106：对于重复 SEND、RESYNC 或 RDMA WRITE 请求，如果响应者在返回响应过程中检测到错误，它应静默丢弃重复请求。这样做是为了避免与可能未完成的对新请求的 NAK 混淆。

• RDMA READ 请求

C9-107：HCA 响应者必须重新创建请求的读取响应数据。结果读取数据以 RDMA READ 响应返回给请求者。第一个 RDMA READ 响应数据包的 PSN 应与重复请求的 PSN 相同，后续响应数据包的 PSN 按为 RDMA READ 生成 PSN 的正常规则递增。

C9-108：如果 HCA 响应者在返回第一个响应数据包之前重新执行重复 RDMA READ 请求时检测到错误，响应者应静默丢弃重复请求。

C9-109：如果 HCA 响应者在返回一个或多个响应数据包之后重新执行重复 RDMA READ 请求时检测到错误，应中止 RDMA READ 响应操作，即不再返回更多响应数据包。

当生成 RDMA READ 请求时，根据 RDMA READ 响应中预期的数据包数量分配一定数量的连续 PSN 编号。这些 PSN 由响应者在生成 RDMA READ 响应数据包时使用。此外，原始请求包含 RETH 中定义的 DMA Length，表示被请求的数据范围。

要被视为重复 RDMA READ 请求，重复请求的 PSN 必须在响应者当前重复 PSN 区域内。此外，要被视为有效的重复 RDMA READ 请求，重复请求的 PSN 必须落在分配给原始 RDMA READ 响应的 PSN 范围内，且重复请求中请求的数据量必须完全包含在原始 RDMA READ 请求中请求的数据范围内。换句话说，重复 RDMA READ 请求中请求的数据必须是原始 RDMA READ 请求中请求的数据的真子集。

如果重复 RDMA READ 请求的起始 PSN 和长度不落在分配给原始 RDMA READ 响应的 PSN 范围内，则请求无效，响应者可以静默丢弃重复 RDMA READ 请求。

C9-110：响应者应按 PSN 顺序响应所有重复请求；即，具有（逻辑上）最早 PSN 的请求应首先执行。如果在响应新请求或重复请求时，接收到具有逻辑上更早 PSN 的重复请求，响应者应停止响应原始请求并开始响应具有逻辑上更早 PSN 的重复请求。

如果响应者被重复请求中断，不要求恢复被中断的请求。请求者有责任重新发送任何未确认的请求。

o9-65：如果 TCA 实现 RDMA 功能，RDMA READ 响应应符合前述 4 个 HCA 合规声明。

在重复 RDMA READ 响应之后，响应者可以用最近完成的请求的 PSN 确认任何后续的重复 Send 或 RDMA WRITE 请求。

• ATOMIC 操作请求

给定的接收队列可能只有资源支持有限数量的 ATOMIC 操作。当接收到重复的 ATOMIC 操作请求时，重复请求的 PSN 与最近执行的 ATOMIC 操作的 PSN 进行比较。

o9-66：如果重复 ATOMIC 操作请求的 PSN 恰好匹配最近执行的 ATOMIC 操作之一的 PSN，该操作的保存结果应返回给请求者。响应者不应重新执行请求。

o9-67：如果重复 ATOMIC 操作请求的 PSN 与最近执行的 ATOMIC 操作的 PSN 不匹配，则请求无效，重复请求数据包应被静默丢弃。只要请求者遵守未完成 ATOMIC 操作请求数量的限制，这就不应发生。

o9-68：如果本地错误阻止响应者重现原始 ATOMIC 操作请求数据，响应者必须静默丢弃重复请求。

在所有情况下，确认消息中返回的 PSN 是重复请求的 PSN。

9.7.5.1.5 生成 NAK

有几种情况会导致响应者生成 NAK。

C9-111：在除 RDMA READ 请求之外的所有情况下，NAK 数据包的 PSN 应包含响应者的预期 PSN。

C9-112：对于 RDMA READ 响应数据包的情况，NAK 响应数据包中给出的 PSN 应指向被 NAK 的 RDMA READ 响应数据包。

C9-113：当生成 RNR NAK 时，响应数据包的 PSN 应指向被 RNR NAK 的数据包的 PSN。

C9-113.a1：当生成 NAK 时，包含 NAK 代码的数据包应具有 Acknowledge 的 opcode。

这意味着，即使对于 RDMA READ 响应，如果响应者返回 NAK 代码，它通过用 Acknowledge 的 opcode 替代正常的 RDMA Read Response（first/middle/last/only）的 opcode 来实现。

一旦响应者返回了 NAK-sequence error 或 RNR NAK，它等待请求者发送具有响应者预期 PSN 的数据包。

响应者必须遵循的规则如下：

C9-114：一旦为 PSN 序列错误返回 NAK，响应者应忽略除重复请求之外的所有其他新请求，直到它接收到具有匹配其预期 PSN 的 PSN 的有效请求。它不应返回任何其他 NAK 数据包，除了响应具有匹配其预期 PSN 的 PSN 的有效请求。

C9-115：响应者必须继续按上述规定响应重复请求。然而，响应者不应在处理重复请求时发生的错误条件下返回 NAK。

9.7.5.1.6 确认消息调度

响应的调度本身不做规定；然而，请求者可以使用 BTH 中的 AckReq 位要求响应者调度响应。

C9-116：当响应者接收到设置了 AckReq 位的有效请求数据包时，它应为该请求调度响应数据包。响应的 PSN 必须等于或逻辑上大于（模 2^24）请求的 PSN。

在接收到设置了 AckReq 位的请求消息后，响应者仍然接受请求消息（包括标记了 AckReq 位的额外请求），同时准备传输响应数据包。这些额外的请求消息可能在响应者能够发送响应时导致合并 ACK。在这种情况下，单个响应消息可以满足多个未完成的请求消息。

当一个或多个请求到达时未设置 AckReq 位，响应者可以选择故意合并其响应；它甚至可以无限期地等待额外请求，直到到达一个需要调度响应的请求。

AckReq 位可以对请求者非常有用。例如，如果请求者正在发送发布到发送队列的最后一个请求 WQE 的最后一个数据包，建议请求者设置 AckReq 位或使用其他机制强制响应者返回响应。如果请求者不这样做，响应者可能选择无限期地合并响应。请求者可以使用的其他一些机制包括：
• 始终在每条消息的最后一个（或唯一）数据包上设置 AckReq 位
• 在给定请求之后发送设置了 AckReq 位的 NOP
• 用设置了 AckReq 位重试所需响应的请求。

对于 SEND 或 RDMA WRITE 请求，可以在数据实际写入响应者内存之前调度 ACK。ACK 仅表示数据已成功到达响应节点的故障域。即数据已被通道适配器接收，通道适配器将把该数据写入响应节点的内存系统，或至少会通知响应应用程序故障。

AckReq 位的缺失不禁止响应者生成响应数据包。一如既往，RDMA READ 和 ATOMIC 操作请求需要显式响应，因此 AckReq 位对这些请求没有影响。

9.7.5.1.7 响应格式

响应可以采用三种形式之一：
1) 用于正常 SEND、RESYNC 或 RDMA WRITE 操作的确认数据包，
2) RDMA READ 响应，以及
3) ATOMIC 操作的确认消息。

三种形式之间的关键区别在于正常确认数据包（用于 SEND、RESYNC 和 RDMA WRITE）不携带有效载荷字段，而 RDMA READ 和 ATOMIC 操作的响应则携带。这一观察影响了响应的格式和合并确认的规则。

确认数据包包含以下信息：
• 用于通知请求者给定请求消息成功或失败的综合征，
• 请求者用于将确认消息与其未完成请求列表关联的 PSN 值，
• 响应者用于通知请求者请求消息已完成的消息序列号，
• 可选的端到端流控制信用，
• 在 RDMA READ 响应或 ATOMIC 操作响应的情况下的有效载荷数据。

每种形式在以下各节中讨论。

9.7.5.1.8 SEND、RESYNC 或 RDMA WRITE 请求的响应格式

此格式用于确认 SEND、RESYNC 或 RDMA WRITE 请求数据包。正常确认消息由单个数据包组成，如图 96 所示。

格式：LRH GRH BTH RDETH AETH ICRC VCRC
注 1：GRH 可能出现也可能不出现，取决于 LRH Next Header 字段
注 2：RDETH 仅出现在可靠数据报操作中
注 3：DETH、RETH、EOP、PYLD 和 IMM 字段被禁止

9.7.5.1.9 RDMA READ 响应

此响应格式称为 RDMA READ 响应，用于确认 RDMA READ 请求。RDMA READ 响应消息由一个或多个数据包组成。

C9-117：对于 HCA，RDMA READ 响应数据包的 PSN 必须是连续的且单调递增的。如果响应消息由多于一个数据包组成，响应消息的第一个和最后一个数据包必须包含确认扩展传输头（AETH）。

o9-69：如果 TCA 实现 RDMA 功能，RDMA READ 响应数据包的 PSN 必须是连续的且单调递增的。如果响应消息由多于一个数据包组成，响应消息的第一个和最后一个数据包必须包含 AETH。

C9-118：对于 HCA，如果响应消息只包含单个数据包（"仅有"数据包），则该数据包必须包含 AETH。

o9-70：如果 TCA 实现 RDMA 功能，且响应消息只包含单个数据包，则该数据包必须包含 AETH。

RDMA READ 响应消息除了确认 RDMA READ 请求本身外，还隐式确认 RDMA READ 请求之前的请求。管理合并 ACK 的规则在第 9.7.5.1.2 节"合并确认消息"中给出。

第一个数据包或唯一数据包的到达触发第 9.7.5.1.2 节中指定的对任何未完成请求消息的隐式确认。这样做是为了减少完成任何未完成请求消息的延迟。

最后一个数据包或唯一数据包的到达触发对 RDMA READ 请求本身的显式确认。

9.7.5.2 AETH 格式

确认综合征携带在确认消息的 AETH 中。下表说明了 AETH 的综合征字段。

C9-119：当生成 AETH 时，实现 RC 服务的 HCA 响应者应按表 43"AETH 综合征字段"中所示编码 AETH 综合征字段。

o9-71：如果 TCA 响应者实现 RC 服务，或者如果 CA 响应者实现 RD 服务，响应者应按表 43 所示编码 AETH 综合征字段。

C9-119.a1：对于 HCA，AETH 综合征字段的最高位是保留的，应设置为零。

o9-71.a1：对于提供 RC 或 RD 服务的 TCA，AETH 综合征字段的最高位是保留的，应设置为零。

o9-72：如果 CA 实现可靠数据报服务，C CCCC 位设置为零，因为端到端信用未为 RD 服务定义。

位[4:0]的解释取决于位[6:5]中包含的代码。位[4:0]可能包含带有或不带有端到端流控制信用的肯定确认（取决于服务是 RC 还是 RD）、RNR NAK 定时器值、不带有端到端信用的肯定确认或 NAK 代码。

表 43 AETH 综合征字段
位 7 | 位 6:5 | 位 4:0 | 定义
0 | 00 | C CCCC | ACK（C CCCC = 信用计数）
0 | 01 | T TTTT | RNR NAK（T TTTT = 定时器值）
0 | 10 | X XXXX | 保留
0 | 11 | N NNNN | NAK（N NNNN = NAK 代码）

C CCCC = 编码的端到端流控制信用
T TTTT = RNR NAK 定时器字段
N NNNN = NAK 代码

代码 011 N NNNN（NAK）允许 MSN 与 NAK 数据包一起携带。

9.7.5.2.1 端到端流控制信用字段

如果 AETH 综合征字段的位[7:5]为零，则 AETH 综合征字段的位[4:0]携带从响应者到请求者的编码端到端流控制信用。此字段仅对可靠连接有效。编码 5b11111 表示信用字段无效。此编码也用于接收队列不支持端到端信用的情况。

9.7.5.2.2 NAK 代码

如果 AETH 综合征字段的位[6:5]为 b11，则位[4:0]携带 NAK 代码。该代码指导请求者选择恢复策略。以下各节描述所有可能的 NAK 代码。尽管 RNR NAK 有自己的 AETH 综合征（AETH[6:5] = b01），RNR NAK 也在本节中描述。

有效 NAK 代码列表在表 44 中提供。

表 44 NAK 代码
NAK 代码（AETH 位 4:0）| 定义
00000 | PSN 序列错误
00001 | 无效请求
00010 | 远程访问错误
00011 | 远程操作错误
00100 | 无效 RD 请求
00101 - 11111 | 保留

C9-120：如果请求者接收到包含保留代码的确认消息，它应认为确认数据包格式错误并应静默丢弃它。这最终可能导致请求者在等待丢失的确认数据包时超时，届时它将重新传输原始请求消息或停止该发送队列上的操作。

9.7.5.2.3 PSN 序列错误

当响应者检测到 PSN 序列不正确的数据包时，即 PSN 值既不等于预期 PSN 也不在有效重复数据包范围内时，会发生序列错误。

C9-121：响应者在构造响应序列错误的 NAK 数据包时，必须在 BTH 的 PSN 字段中插入其预期 PSN 值。这让请求者将其发送队列至少回退到故障点并从该点开始重新发送请求数据包。

PSN 序列错误可能被请求者重试多次。一旦重试计数已过期，请求者的传输会通知其客户端未成功传输消息。以下讨论指定重试计数过期之前的行为。

当响应者检测到序列错误时，对接收队列没有影响，也不消耗任何 WQE。相反，接收队列只是向请求者返回 NAK 数据包并恢复等待具有正确 PSN 值的入站请求数据包。

C9-122：一旦向请求者返回了序列错误的 NAK 数据包，响应者应丢弃不包含响应者预期 PSN 的所有后续请求，有效重复请求除外。

C9-123：如果响应者接收到 PSN 逻辑上小于其预期 PSN 的请求数据包（即有效重复请求数据包），它应根据重复数据包处理规则响应该请求。

9.7.5.2.4 远程访问错误

R_Key 违规由以下任何或所有条件导致，针对 RDMA READ、RDMA WRITE 或 ATOMIC 操作：
• RETH 的 R_Key 字段无效。
• 指定的虚拟地址和长度或访问类型超出与 R_Key 关联的本地定义限制。
• 对于 HCA，检测到保护域违规。

C9-124：对于 HCA 响应者，当报告 RDMA 远程访问错误时，确认消息的 BTH 字段必须包含导致远程访问错误的请求数据包的 PSN。

o9-73：如果 TCA 响应者实现 RDMA 功能，或者如果 CA 响应者支持 ATOMIC 操作，则当报告远程访问错误时，确认消息的 BTH 字段必须包含导致远程访问错误的请求数据包的 PSN。

9.7.5.2.5 无效请求

请求者请求了超出传输服务既定用途的操作 - 通常是响应者不支持的 OpCode 或长度超过可用接收缓冲区空间的请求。

C9-125：当报告无效请求时，确认数据包的 BTH 字段必须包含响应者的预期 PSN 值，即包含无效请求的请求数据包的 PSN。

9.7.5.2.6 远程操作错误

当响应者遇到阻止其接收队列完成当前请求的情况时，会发生远程操作错误。响应者可检测并报告为远程操作错误的错误条件列表未做规定，因为它是实现特定的。远程操作错误不能由请求者可能做的任何事情引起。相反，它们反映了响应者中的故障。

C9-126：当报告远程操作错误时，确认消息的 BTH 字段必须包含响应者检测到操作错误时正在执行的请求的 PSN。

9.7.5.2.7 无效 RD 请求

当响应者在以 RD 服务操作时检测到 Q_Key 或 RDD 违规，或者目标 QP 未配置为 RD 服务，或者目标 QP 不处于可以接受入站数据包的状态时，生成此 NAK 代码。

o9-74：如果响应者的 EE 上下文检测到无效 P_Key，请求数据包应被 EE 上下文静默丢弃。

如果未检测到 P_Key 违规，EE 上下文将数据包转发到 BTH 中指定的接收队列。

C9-127：如果 BTH 中指定的 QP 未配置为 RD 服务，则应返回 NAK-Invalid RD Request。

接收队列检查入站请求数据包的 Q_Key，并检查其当前 RDD 值是否与 EE 上下文的匹配。

如果响应者的接收队列检测到无效 Q_Key，或者接收队列的 RDD 值与 EE 上下文的不匹配，响应者应向请求者返回 NAK-Invalid RD Request。

9.7.5.2.8 RNR NAK

在某些情况下，接收队列可能暂时无法接受入站请求消息。例如，当前可能没有有效的接收 WQE 发布到接收队列。当发生这种情况时，响应者可能返回指示接收者未就绪（RNR NAK）的响应。注意：响应者可能为任何类型的请求（例如 SEND、RDMA READ 请求、RDMA WRITE 请求等）返回 RNR NAK。

在接收到 RNR NAK 后，请求者可以在等待至少 RNR NAK 中指定的间隔后重试相同的请求。"相同的请求"是指从响应者在其 RNR NAK 数据包中报告的相同 PSN 开始的精确相同的请求消息。

C9-128：对于使用可靠连接服务的 HCA 请求者，在接收到 RNR NAK 后，请求者不应通过重用相同的 PSN 来替换不同的请求消息。

o9-75：如果 TCA 请求者实现可靠连接服务，在接收到 RNR NAK 后，请求者不应通过重用相同的 PSN 来替换不同的请求消息。

对于可靠数据报服务，请求者可以精确重复请求、将 EEC 移至错误状态、放弃请求或如第 9.7.8 节所述暂停请求。

C9-129：使用可靠连接服务的 HCA 响应者，当生成 RNR NAK 时，应在 AETH 的定时器字段中指示适当的间隔。加载到 AETH 定时器字段中的值应如表 45"RNR NAK 定时器字段编码"所示。

o9-76：如果 CA 响应者实现可靠数据报服务，或者如果 TCA 实现可靠连接服务，当生成 RNR NAK 时，响应者应在 AETH 的定时器字段中指示适当的间隔。

C9-130：提供可靠连接服务的 HCA 请求者，在接收到 RNR NAK 后，必须等待至少 AETH 定时器字段中指定的间隔后才能重试请求。如果请求者未能在适当的超时间隔前等待就重试请求，响应者可以静默丢弃数据包。

o9-76.a1：提供可靠数据报服务的 HCA 请求者，在接收到 RNR NAK 后，必须等待至少 AETH 定时器字段中指定的间隔后才能重试请求。

o9-76.a2：提供可靠连接服务或可靠数据报服务的 TCA 请求者，在接收到 RNR NAK 后，必须等待至少 AETH 定时器字段中指定的间隔后才能重试请求。

C9-131：此合规声明已过时并已删除。

C9-132：使用可靠连接服务的 HCA 请求者应维护一个 3 位重试计数器，该计数器在连接建立期间由响应者提供的信息加载。此计数器用于限制请求者可以重试被 RNR NAK 的操作的次数。当接收到 RNR NAK 响应时，如果 RNR NAK 重试计数器不等于 7（表示无限重试），请求者应递减 RNR NAK 重试计数器。此后，当重试定时器过期时，如果重试计数器非零，请求者可以重新发出请求。

o9-77：如果 CA 请求者实现可靠数据报服务，或者如果 TCA 请求者实现可靠连接服务，它应维护一个 3 位重试计数器。

如果重试计数器在 RNR NAK 重试定时器过期时已递减到零，则请求者记录本地检测到的错误。

9.7.6 请求者：接收响应

9.7.6.1 验证入站响应数据包

在接收到入站确认数据包后，请求者按如下方式验证数据包：

C9-133：为了验证数据包的完整性，请求者应按第 9.6 节"数据包传输头验证"中的规定验证数据包。无效数据包应被请求者静默丢弃。

C9-134：对于使用可靠连接服务的 HCA 请求者，应检查 PSN 以检测乱序数据包。由于确认可能如第 9.7.5.1.2 节所述被合并，PSN 用于检测合并的响应。

o9-78：如果 TCA 请求者实现可靠连接服务，或者如果 CA 请求者实现可靠数据报服务，应检查每个确认数据包的 PSN 以检测乱序数据包。

C9-135：对于使用可靠连接服务的 HCA 请求者，应根据第 9.7.5.2.2 节"NAK 代码"中的表检查确认综合征的有效性。包含保留 NAK 代码的响应数据包应被简单丢弃。

o9-79：如果 TCA 请求者实现可靠连接服务，或者如果 CA 请求者实现可靠数据报服务，应根据第 9.7.5.2.2 节的表检查确认综合征的有效性。

o9-79.a1：如果 CA 实现可靠数据报服务，当接收响应数据包时，请求者应检查 BTH 中包含的目标 QP 编号与当前 EEC 的预期 QP 编号。如果响应数据包不是发往当前活动的请求者侧 QP，它应被请求者丢弃。

请求者还必须确保响应适合正在执行的操作类型。例如，在某些拥塞的 fabric 情况下，当期望 RDMA READ 或 Atomic 响应时可能到达 ACK。即使响应数据包中包含的 PSN 与请求者的预期响应 PSN 匹配，这也可能发生。反之亦然。

因此，请求者应验证接收到的响应数据包与未完成的请求一致。如果不一致，请求者的行为应如表 56"请求者侧错误行为"中所述。

C9-135.a1：此合规声明已过时。

如果数据包被确定为有效，则由请求者处理。在处理确认数据包时，请求者可能遇到本地错误。请求者在处理确认消息时可能遇到的本地错误列表未做规定，因为它是实现特定的，但包括由请求者侧故障导致的任何错误。

验证入站响应数据包的 PSN 依赖于识别 PSN 序列中的三个关键点：
1) 请求者的最大前进进度 - 请求者发送的任何请求的逻辑上最大（模 2^24）PSN。（这包括为 RDMA READ 响应分配的 PSN 空间。）它标记有效区域的"右手"边缘。
2) 最旧的未确认请求 - 最旧的未完成（未确认）请求的 PSN。这代表请求者所见的响应者所做的最大前进进度。此 PSN 的重要性在于它标记重复区域的结束，即 PSN 逻辑上小于此值的响应被视为无效或重复。
3) 最旧的有效请求 - 此点标记有效区域的"左手"边缘。

与请求数据包一样，每个响应数据包都携带 PSN。请求者在接收到响应数据包后检查 PSN 以确定响应是预期响应还是幽灵确认数据包。从概念上讲，请求者跟踪最旧的未确认请求数据包的 PSN 和截至目前发送的逻辑上最大（模 2^24）PSN，包括为 RDMA READ 响应保留的 PSN。这两个 PSN 定义了上图中标识为未确认区域的 PSN 范围的端点。如果响应数据包的 PSN 落在该范围内，则该数据包是预期响应数据包。如果响应不落在该区域内，则被视为重复响应或幽灵（无效）确认数据包并被请求者丢弃。

C9-136：对于使用可靠连接服务的 HCA 请求者，幽灵确认数据包应被请求者丢弃。

o9-80：如果 TCA 请求者实现可靠连接服务，或者如果 CA 请求者实现可靠数据报服务，幽灵确认数据包应被请求者丢弃。

9.7.6.1.1 重试请求的 PSN

在某些情况下（如下所述），请求者可能需要重试请求。在 PSN 术语中，这意味着请求者可能重新发送 PSN 逻辑上小于（模 2^24）截至目前传输的最大 PSN 的数据包。

在重试请求的过程中，请求者应维护一种标记最远 PSN 推进点（高水位标记）的方法，即使它在重试请求时逻辑上"回退"了 PSN 序列。这是必要的，因为对重试请求的响应可能具有逻辑上大于重试请求 PSN 的 PSN。这可能发生是因为在某些情况下，响应者可能将重试请求解释为重复请求。如果是这样，它（响应者）有义务返回标记其最远推进点的 PSN，该点可能超出重试请求的 PSN。

请求者需要维护这三个指针（最大推进点、最旧未确认请求的 PSN 和重试请求的 PSN）被称为三 PSN 范式。尽管各种实现可能找到不同的方式来实现三 PSN 范式，但请求者必须考虑到响应者可能返回对重试请求的响应，其 PSN 逻辑上大于重试请求本身的 PSN。

9.7.6.1.2 请求者对 NAK 消息的响应

请求者对否定响应消息的反应取决于返回的 NAK 代码以及队列对是配置为可靠连接还是可靠数据报服务。

NAK-Sequence Error 触发请求的自动重试。响应数据包中的 PSN 是请求者关于响应者认为它遗漏的请求数据包的指示，因此请求者可以重试该请求。为防止请求者永远重试相同的请求，请求者维护一个 3 位重试计数器，用于计算特定请求数据包已重试和超时的次数。

C9-137：使用可靠连接服务的 HCA 请求者应在响应者每次为给定请求数据包返回 NAK-Sequence Error 时递减其 3 位重试计数器。当给定的未完成请求被清除时，应重新加载计数器。如果不支持自动路径迁移，且如果 NAK-Sequence Error 再次返回，则请求者应声明本地检测到的错误。

o9-80.a1：支持可靠数据报服务的 HCA 请求者应在响应者每次为给定请求数据包返回 NAK-Sequence Error 时递减其 3 位重试计数器。

o9-81：实现可靠连接服务或可靠数据报服务的 TCA 请求者应在响应者每次为给定请求数据包返回 NAK-Sequence Error 时递减其 3 位重试计数器。

o9-82：如果 CA 支持自动路径迁移，则要求如下。如果在重试计数器递减到零后返回 NAK-Sequence Error，则通道适配器应尝试自动路径迁移。在自动路径迁移之后，请求者应重新加载重试计数器并重新开始该过程。如果请求者在多次重试后仍未成功发送请求，则请求者应声明本地检测到的错误。

对于其他 NAK 数据包，发送队列的响应取决于队列是提供可靠数据报还是可靠连接服务。

可靠数据报行为：可靠数据报需要使用 EE 上下文来维护数据包序列号从而确保可靠传递请求。取决于原因和相关操作，EE 上下文在检测到失败的请求数据包后可能采取以下任何选项：
a) 它可以从同一 QP 重试相同的失败数据包（注意并非所有 NAK 都可以重试，对于可以重试的，重试次数有限制），或
b) 它可以将 QP 和 EE 上下文转换到错误状态，以错误方式完成失败的请求消息，或
c) 它可以将当前 QP 置于 SQEr 状态并根据第 9.7.8 节中详述的规则生成 RESYNC 请求。在这种情况下，失败的 WQE 通常将以错误方式完成。

最后一种策略旨在允许请求者侧 EE 上下文继续服务，从而避免拆除 EE 上下文到 EE 上下文连接的需要。

如果要重试"相同的失败数据包"，请求者不需要从响应者 NAK 中指示的 PSN 开始其重传序列；相反，它可以从较早的请求数据包开始其重传。这些较早的请求数据包被响应者视为正常的重复数据包，不会造成不良副作用。

C9-138：对于使用可靠连接服务的 HCA 请求者，请求者必须接收并丢弃任何重复确认消息，不产生不良副作用。

o9-83：如果 TCA 请求者实现可靠连接服务，或者如果 CA 请求者实现可靠数据报服务，请求者必须接收并丢弃任何重复确认消息，不产生不良副作用。

可靠连接行为：对于可靠连接，请求者在接收到 NAK 时只有两种可能的替代方案。它可以重试相同的请求数据包，或者将当前 WQE 标记为以错误方式完成并通知其客户端。注意并非所有 NAK 都可以重试。

如果请求者重试相同的请求数据包，它不需要从响应者 NAK 中指示的 PSN 开始其重传序列；相反，它可以从较早的请求数据包开始其重传。

9.7.6.1.3 检测丢失的确认消息和超时

在某些错误条件下，请求者不会接收到对其一个或多个请求的确认消息。这可能由以下三个原因之一引起：
1) 响应者生成了确认消息，但该消息随后在 fabric 中丢失，或
2) 响应者由于某种原因失败，无法生成确认消息，或
3) 原始请求消息在被响应者接收之前在 fabric 中丢失。

所有这三种情况都被请求者检测为丢失的确认消息。

通常，这些错误由于确认合并而自动纠正；请求者接收到的下一个确认隐式确认所有未完成的请求。

然而，有几种情况下丢失的确认消息不会通过合并确认规则自动恢复。例如，在 fabric 中丢失的 NAK 消息不会通过确认合并解决，因为响应者侧规则要求响应者在给定时间内最多有一个 NAK 消息未完成。

C9-139：对于使用可靠连接服务的 HCA 请求者，为检测丢失的响应，每个发送队列都需要实现传输定时器来计时未完成的请求。

o9-84：如果 TCA 请求者实现可靠连接服务，为检测丢失的响应，每个发送队列都需要实现传输定时器。

o9-85：如果 CA 请求者实现可靠数据报服务，为检测丢失的响应，每个 EE 上下文都需要实现传输定时器。

由于 fabric 的可变性、调度算法和通道适配器的架构以及许多其他因素，以高度精确性计时未完成的请求既不可能也不可取。尽管如此，传输定时器是 ACK/NAK 协议的组成部分，通过提供确定性手段来检测丢失的请求或响应。

请求者不需要分别计时发送到 fabric 的每个请求，而只需在期望响应时启动定时器。一旦启动，每次接收到确认数据包时，只要还有预期的未完成响应，就重新启动定时器。定时器不检测特定预期确认数据包的丢失，而只是简单检测响应数据包的持续缺失。

定时器测量以下较小者：
• 自请求者在 BTH 中设置 AckReq 位发送数据包以来的时间，
• 或自最后一个有效确认数据包到达以来的时间。

操作如下：
请求者在定时器当前未运行时启动定时器，且：
1) 请求者在 Send 或 RDMA WRITE 请求中设置 AckReq 位，或
2) 请求者生成 RDMA READ 请求，或
3) 请求者生成 ATOMIC 操作请求。

此后，请求者在每次接收到新的入站确认数据包时重新启动定时器，只要仍有预期的未完成响应。

当没有预期的未完成响应时，停止定时器。

每个 QP 有一个与之关联的本地 ACK 超时值，用于推导传输定时器超时间隔 Ttr。

C9-140：对于使用可靠连接服务的 HCA 请求者，传输定时器超时间隔 Ttr 应定义为 4.096 μS × 2^(Local ACK Timeout)。Local ACK Timeout 应为 5 位值，零表示定时器被禁用。Local ACK Timeout 的最小可接受值（零除外）应由 CA 供应商定义。

C9-141：对于使用可靠连接服务的 HCA 请求者，QP 应提供检测超时条件的设施。

超时间隔应在请求被调度后开始。超时条件 To 应在不少于超时间隔 Tr 且不超过超时间隔的四倍 4Tr 的时间内被检测到。
因此，Ttr <= To <= 4Ttr。

一旦检测到给定请求数据包的超时，请求者可以重试请求。

C9-142：对于使用可靠连接服务的 HCA 请求者，为防止请求者永远重试请求，请求者应维护一个 3 位重试计数器。此计数器应在传输定时器每次为给定请求数据包过期时递减。当给定的未完成请求被清除时，应重新加载计数器。

C9-143：对于使用可靠连接服务的 HCA 请求者，如果不支持自动路径迁移，且如果传输定时器在重试计数器递减到零后过期，则请求者应声明本地检测到的错误。

o9-86：如果支持自动路径迁移，且如果传输定时器在重试计数器递减到零后过期，则通道适配器应尝试自动路径迁移。在自动路径迁移之后，请求者应重新加载传输定时器重试计数器并重新开始该过程。如果请求者在多次重试后仍未成功发送请求，则请求者应声明本地检测到的错误。

o9-87：如果 TCA 请求者实现可靠连接服务，则前述五个 HCA 合规声明应适用于该 TCA。

o9-88：如果 CA 请求者实现可靠数据报服务，则前述五个 HCA 合规声明应适用于该 CA。在这种情况下，所描述的功能适用于 EE 上下文而非队列对。

9.7.6.1.4 重复确认

第 9.7.1 节描述了如何生成重复请求。这些请求可能导致响应者返回重复确认。响应者还可能发送从请求者角度看像是"幽灵 ACK"的未经请求的 ACK。

C9-144：对于使用可靠连接服务的 HCA 请求者，如果响应者配置为生成端到端流控制信用，则请求者必须从重复确认中提取端到端流控制信用。

o9-89：如果 TCA 实现可靠连接服务，且响应者配置为生成端到端流控制信用，则请求者必须从重复确认中提取端到端流控制信用。

C9-145：对于使用可靠连接服务的 HCA 请求者，重复确认应被丢弃。

o9-90：如果 TCA 请求者实现可靠连接服务，重复确认应被丢弃。

9.7.7 可靠连接

所需的可靠性特征通过应用数据包序列号和 ACK/NAK 协议提供。

C9-146：对于 HCA，配置为可靠连接服务的每个 QP 必须符合第 9.7 节"可靠服务"中指定的要求、表 46"可靠连接服务特征"中给出的特征以及本节中给出的任何额外要求。

9.7.7.1 生成 MSN 值

对于可靠连接服务，消息序列号是响应者返回给请求者的数字，表示响应者完成的消息数。MSN 携带在 AETH 的最低三个有效字节中。MSN 作为服务提供给请求者，通过通知请求者响应者已完成的消息来协助其完成 WQE。

C9-147：使用可靠连接服务的 HCA 响应者应在包含 AETH 的任何响应数据包的 AETH 中返回 MSN。

o9-92：如果 TCA 响应者实现可靠连接服务，它应在包含 AETH 的任何响应数据包的 AETH 中返回 MSN。

逻辑上，请求者将连续的发送序列号（SSN）与发布到发送队列的每个 WQE 关联。SSN 与响应者在每个响应数据包中返回的 MSN 有一一对应关系。因此，当请求者接收到响应时，它将 MSN 解释为表示响应者最近完成的请求的 SSN，以确定哪些发送 WQE 可以完成。

注意，上述 SSN 仅是一个逻辑概念，旨在传达 MSN 如何应用的概念；实现不需要按所述方式实现它。

初始化后，发布到发送队列的第一个 WQE 的 SSN 为 1。响应者将其 MSN 计数器初始化为零。此后，响应者在每次完成入站请求消息的执行时递增其 24 位 MSN 值。

C9-148：使用可靠连接服务的 HCA 响应者应将其 MSN 值初始化为零。响应者应在每次成功完成处理新的有效请求消息时递增其 MSN。对于重复请求不应递增 MSN。递增的 MSN 应在 RDMA READ 或 Atomic 响应的最后一个或唯一数据包中返回。对于 RDMA READ 请求，响应者可以在完成验证请求后且在开始传输任何请求数据之前递增其 MSN，并可在第一个响应数据包的 AETH 中返回递增的 MSN。对于任何给定的请求消息，MSN 应仅递增一次。

o9-93：如果 TCA 响应者实现可靠连接服务，它应按前述合规声明中的描述计算和更新 MSN。

9.7.7.1.1 请求者接收新 MSN 时的行为

如上所述，响应数据包中新 MSN 值的存在可被请求者用作完成某些发布到其发送队列的 WQE 的信号。由于响应者可以选择合并确认，单个响应数据包实际上可能确认多个请求消息。因此，当它接收到新 MSN 时，请求者从其发送队列上最旧的未完成 WQE 开始评估并向前推进。

对于发送队列上有未完成的 RDMA READ 请求或 ATOMIC 操作请求的情况，情况稍微复杂。在这种情况下，请求者仅完成到第一个未完成的 RDMA READ 请求、ATOMIC 操作请求或 SSN 匹配响应数据包中 MSN 的 WQE 为止的未完成 WQE，以先到者为准。这是因为 RDMA READ 请求和 ATOMIC 操作请求都需要显式响应，因此在接收到此类显式响应之前无法完成。

C9-149：对于使用可靠连接服务的 HCA 响应者，无论响应是肯定确认、否定确认还是重复确认，MSN 计数器都应插入到 AETH 中。

o9-94：如果 TCA 响应者实现可靠连接服务，无论响应是肯定确认、否定确认还是重复确认，MSN 计数器都应插入到 AETH 中。

9.7.7.2 端到端（消息级别）流控制

IBA 为可靠连接提供端到端（或消息级别）流控制功能，响应者可以使用它来优化其接收资源的使用。基本上，请求者不能发送请求消息，除非它有适当的信用。

编码的信用通过确认消息中 AETH 的综合征字段从响应者传输到请求者。AETH 中携带的信用是相对于同一 AETH 的 MSN 字段的；因此，正确解释信用字段也需要解释 MSN 字段。

每个信用代表接收一个入站请求消息所需的接收资源。具体来说，每个信用代表发布到接收队列的一个 WQE。接收信用的存在并不一定意味着已分配足够的物理内存。

共享接收队列概念允许一组接收队列从公共的接收 WQE 池（即共享接收队列）中提取。因此，任何单独的接收队列都无法准确衡量共享接收队列中可用的 WQE 数量。出于这个原因，对于与共享接收队列关联的任何 QP，端到端流控制机制被禁用。

1) 端到端信用机制仅适用于可靠连接服务。
2) 端到端信用由响应者的接收队列生成，由请求者的发送队列消耗。
3) 对 CA 支持端到端流控制的要求在第 17 章"通道适配器"中给出。HCA 接收队列必须生成端到端信用（与共享接收队列关联的 QP 除外），但 TCA 接收队列不需要这样做。
4) 信用按每条消息发放，不考虑消息大小。
5) 端到端信用在 AETH 中作为编码的 5 位字段携带。
6) 响应者可以通过使用未经请求的确认数据包异步地向请求者发送信用。

C9-150.2.1：对于未与 SRQ 关联的 QP，每个 HCA 接收队列应生成端到端流控制信用。如果 QP 与 SRQ 关联，HCA 接收队列不应生成端到端流控制信用。

o9-95.2.1：每个 TCA 接收队列可以生成端到端信用，但与 SRQ 关联的 QP 除外。

C9-151：如果 TCA 的给定接收队列生成端到端信用，则相应的发送队列应接收并响应这些信用。这是对 CA 每个发送队列的要求。

9.7.7.2.1 从响应者向请求者传输信用

定义了两种将信用从响应者的接收队列传输到请求者的发送队列的机制。信用可以搭载在现有的确认消息上，或者可以由响应者生成特殊的未经请求的确认消息。

搭载信用：搭载端到端信用是指在正常确认数据包的 AETH 中将信用传输给请求者。信用携带在 AETH Syndrome[4:0] 中。当 AETH 中的 MSN 字段也有效时，信用可以搭载到任何确认数据包上。

未经请求的确认数据包：从 PSN 角度来看，未经请求的确认消息在请求者看来像是最近肯定确认消息的重复。然而，它始终具有 Acknowledge 的 opcode，即使最近的肯定确认是 RDMA READ 响应或 Atomic 响应。由于 ACK/NAK 协议禁止响应者发送重复的否定确认数据包（NAK），因此不能通过重新发送 NAK 数据包来创建未经请求的确认。

未经请求的确认可以由响应者在任何时间发送。请求者的发送队列只需从最近接收到的确认数据包中恢复信用字段和 MSN。

C9-152：未经请求的确认数据包的 MSN 字段必须具有有效的 MSN 字段。

9.7.7.2.2 协商连接：初始信用

对于建立的每个连接，端到端流控制的使用（或不使用）是为每个方向分别建立的。接收队列的能力决定了该连接一半的流控制特征。

C9-153：如果接收队列表示它期望生成信用，则相应的发送队列必须遵守端到端流控制规则。如果另一方面，接收队列表示它不会生成端到端流控制信用，则相应的发送队列可以随意传输请求消息而不考虑信用。

C9-154：如果 TCA 的接收队列不生成端到端信用，它应在 AETH Syndrome[4:0] 中放置值 5b11111，表示信用字段无效。

o9-95.2.2：如果 CA 支持 SRQ，且 QP 提供可靠连接服务并与 SRQ 关联，则 QP 不得生成端到端信用，应在 AETH Syndrome[4:0] 中放置值 5b11111。

C9-155：当接收队列处于 RESET 状态时，传输应将初始信用计数设置为零。一旦队列对转换到 INITIALIZED、RTR、SQD 或 RTS 状态，它应为发布到接收队列的每个接收 WQE 递增其信用计数。

9.7.7.2.3 响应者计算信用的算法

C9-157.2.1：对于在未与 SRQ 关联的 QP 上使用可靠连接服务的 HCA，接收队列应为发布到接收队列的每个 WQE 递增其信用计数。它应为接收到的每个消耗 WQE 的入站请求消息递减其信用计数。因此，当响应者接收到入站 RDMA READ 请求、不带立即数据的 RDMA WRITE 请求或 ATOMIC 操作请求时，不调整其信用计数。如果接收队列与 SRQ 关联，无论可能发布到接收队列的 WQE 如何，响应者不调整其信用计数。

o9-96：如果 TCA 实现可靠连接服务，且接收队列生成端到端流控制信用，它应为发布到接收队列的每个 WQE 递增其信用计数，为接收到的每个消耗 WQE 的入站请求消息递减其信用计数。

C9-158.2.1：对于在未与 SRQ 关联的 QP 上使用可靠连接服务的 HCA，对于生成的每个确认消息，无论是正常确认消息还是未经请求的确认消息，接收队列应在 AETH Syndrome[4:0] 中插入其当前编码的信用计数，如表 47 所示。如果 QP 与 SRQ 关联，接收队列应在 AETH 中插入 5 位值 5b11111。

9.7.7.2.5 请求者行为 - 受限发送 WQE

C9-162：当请求者在其发送队列上遇到没有可用信用的 WQE 时，该 WQE 被称为受限。发送队列遇到受限 WQE 时的行为如下：
• 如果受限请求 WQE 是 RDMA READ 请求、不带立即数据的 RDMA WRITE 请求或 ATOMIC 操作请求，可以正常发送而不考虑信用可用性。发送此类请求后，请求者递增其计算的 LSN 值，因为发送的请求不消耗接收 WQE 因此不消耗信用。
• 如果受限请求 WQE 是 SEND 请求，发送队列在必须停止传输并等待确认数据包之前应传输不超过请求消息的单个数据包。为确保响应者将生成响应，请求者应在该单个数据包中设置 AckReq 位。
• 如果受限请求 WQE 是带立即数据的 RDMA WRITE 请求，请求者可以在必须停止传输并等待确认数据包之前传输整个请求消息。这是允许的，因为实际消耗接收 WQE 的是请求的包含立即数据的单个数据包。为确保响应者将生成响应，请求者应在请求消息的最后一个数据包中设置 AckReq 位。

C9-163：对于使用可靠连接服务的 HCA，如果受限 WQE 是 SEND 请求，发送队列应传输不超过请求消息的单个数据包。在该单个数据包内，BTH 的确认请求（AckReq）位应被设置。请求者随后应停止传输并等待确认数据包。

C9-164：由于响应者的接收队列可能在任何时间生成未经请求的确认消息，请求者应准备在任何时间接收来自响应者的未经请求的确认消息，前提是接收队列已表示它将生成端到端流控制信用。

未经请求的确认仅用于将信用从响应者传输到请求者。在接收到未经请求的确认后，请求者如上所述重新计算其 LSN 并做出相应响应。缺乏信用不影响请求者重新传输先前传输的请求作为丢失数据包恢复的一部分的能力。端到端信用仅限制新请求消息的传输。

9.7.8 可靠数据报

可靠数据报提供可靠通信，即与可靠连接相同级别的可靠性和错误恢复，使用一对多范式。请求者的发送队列可以向不同的响应者、在相同或不同节点上的不同 QP 发送连续消息。响应者 QP 可以接收来自同一或不同端节点上多个请求者的消息。与不可靠数据报传输服务一样，源端节点和源 QP 提供给响应者。

使用可靠数据报的动机是为从事"全对全"通信的应用程序节省 QP 命名空间。考虑 N 个处理器节点，每个有 M 个进程。如果所有 M 个进程都希望与所有节点上的所有进程通信，可靠连接服务需要每个节点 M²×(N-1) 个 QP。相比之下，可靠数据报服务每个节点只需要 M 个 QP + N 个"端到端"（EE）连接，就可实现完全相同的通信。

可靠性使用至少一个每个远程端节点的"类 QP"上下文来实现 - 这称为端到端上下文（EE Context 或 EEC）。此上下文提供定位远程节点、序列化和交换确认以及维护可靠性所需的信息。

该服务仍使用 QP 来提供队列、WQE 指针、保护检查参数等。QP 和 EEC 一起包含将消息可靠地移动到目标所需的信息。但许多 QP 可以使用单个 EEC 进行发送或接收，QP 可以通过几个不同的 EEC 进行通信，每条消息选择一个。

9.7.8.3 可靠数据报操作

处理与可靠连接服务的定义非常相似。显著差异在于响应者处的重复数据包处理和请求者处重复请求的规则。

9.7.8.3.1 SEND 和带立即数据的 RDMA WRITE 处理

SEND 和带立即数据的 RDMA WRITE 的处理方式与可靠连接服务相同，只是端到端信用不返回给发送 QP。

o9-111：声称支持可靠数据报服务的 CA 应使用 NAK-RNR 协议来指示 RD 消息的接收队列溢出。

9.7.8.3.2 RDMA READ 处理

RDMA READ 的处理方式与可靠连接服务相同。与可靠连接服务不同，来自单个 QP 或 EEC 的未完成 RDMA READ 请求消息数应限制为一个。

9.7.8.3.3 ATOMIC 处理

ATOMIC 的处理方式与可靠连接服务相同。与可靠连接服务不同，来自单个 QP 或 EEC 的未完成 ATOMIC 请求数应限制为一个。

9.7.8.4 排序规则

接收队列是 FIFO 队列。一旦排队，WQE 应按 FIFO 顺序开始处理，但可以乱序完成。来自任何单个源 QP 的消息应始终有序。

o9-112：声称支持 RD 模式的 CA 应为 RD 消息提供上层支持乱序接收队列完成。

发送队列是 FIFO 队列。一旦排队，WQE 应按排队顺序处理发送。

o9-113：声称支持 RD 模式的 CA 应确保 RD 模式下发送队列上的 WQE 按顺序完成，无论它们是针对同一或不同端节点上的不同目标 QP 还是同一目标 QP。WQE 的完成应始终以 FIFO 顺序返回给传输消费者。

9.7.8.5 处理 QP 错误 - RESYNC

由于 RD 服务允许多个 QP 共享单个 EEC，因此希望局限于 QP 的错误不影响共享同一 EEC 的其余 QP。为此，RD 服务允许 EEC 在某些错误条件下"放弃"或"暂停"QP 上的操作。

消息被"放弃"是指它在源 QP 以错误方式完成，QP 编号转换到错误状态。

消息被"暂停"是指它在源 QP 未完成，但在同一 EEC 上启动了另一条消息。稍后恢复时，如果是 Send 或带立即数据的 RDMA Write 操作，暂停的消息必须从头重新开始。如果是 RDMA Read 或不带立即数据的 RDMA Write，它可以（实现选择）从中断处重新开始，但对响应者必须看起来是新消息。

"暂停"和恢复仅适用于 RNR NAK 条件，即响应者暂时无法执行与特定 QP 关联的请求，请求者可以通过在同一 EEC 上从其他 QP 发送消息来提高性能，同时等待 RNR NAK 超时。

请求者不允许暂停 Atomic 操作。

"放弃"或"暂停"的概念确实要求 RD 服务处理一类在数据包延迟、重复或两者兼有时可能发生的错误。为处理此问题，每当消息被"放弃"或"暂停"时，都需要进行 RESYNC 操作。

如果请求者简单地在原始 PSN 处开始新消息，可能会发生数据损坏。此外，当请求者稍后重新启动被 RNR NAK 的消息时，响应者将获得两份副本。

为处理此问题，在影响 QP 的任何错误之后使用 RESYNC，但让 EEC 能够继续操作。RESYNC 有几个目的：它使 EEC 两端的 PSN 同步，它让响应者知道先前的消息（如果不完整）将被放弃，它允许请求者确定性地确定当前消息在响应者处的状态。

o9-114.a1：对于声称支持 RD 模式的 CA，当 RD 模式下的消息发生与 QP 相关的错误时，请求者可以：
1) 将 QP 和 EEC 转换到错误状态并以错误方式完成消息，或
2) 实现第 9.7.8.5 节中描述的 RESYNC 过程。

o9-114.a2：对于支持 RD 传输服务的 CA，在发送 RESYNC 后，请求者应等待 RESYNC PSN 处的 ACK。没有其他响应有效。

RESYNC 应使用通常的重试计数进行超时和重试，如果没有到达正确的响应。

9.7.8.6 响应者生成 MSN

对于可靠数据报服务，消息序列号是响应者返回给请求者的数字，表示响应者在 EE 上下文处完成的消息数。MSN 携带在 AETH 的最低三个有效字节中。

o9-114.a3：实现可靠数据报服务的 CA 中的响应者应在每个响应数据包的 AETH 中返回 MSN。

o9-114.a4：使用可靠数据报服务的 CA 响应者应将其 MSN 值初始化为零。响应者应在每次成功完成处理新的有效请求消息时递增其 MSN。对于重复请求不应递增 MSN。递增的 MSN 应在 RDMA READ 或 Atomic 响应的最后一个或唯一数据包中返回。

9.7.8.6.1 请求者接收新 MSN 时的行为

响应数据包中新 MSN 值的存在可被请求者用作完成消息的信号。

"RESYNC"消息的响应中的 MSN 也可用于确定先前消息的完成状态：
• 如果由于早期重试，先前消息实际已完成，MSN 将比 NAK 响应中返回的最后一个值高两个。
  • 如果先前消息是 Send 或 RDMA Write，它应正常完成（意味着它毕竟没有被放弃或暂停）。
  • 如果先前消息是 RDMA Read，请求者没有获得返回数据，因此必须仍然暂停或放弃消息，取决于错误类型。
  • 如果先前消息是 Atomic，它必须以错误方式完成，附加信息"在响应者处已知完成"可返回给上层。
• 如果 MSN 与预期相同（比最后一个值大一），先前消息应被适当处理：
  • 如果被放弃，以错误方式完成，附加信息消息"在响应者处已知不完整"
  • 如果是 Send 或带立即数据的 RDMA Write，未启动，稍后重试
  • 如果是 RDMA Read 或不带立即数据的 RDMA Write，不完整，稍后作为新消息重新启动

如果 MSN 是任何其他值，响应被损坏，EEC 应置于错误状态。


9.8 不可靠服务（Unreliable Service）

IBA定义了两种类型的不可靠服务：不可靠连接（SEND、RDMA WRITE）和不可靠数据报（仅SEND）。这些服务具有以下特征：

1) 请求者不会收到消息接收的确认
2) 没有数据包顺序保证
3) 响应者正常验证入站数据包（验证适当的头部字段、CRC检查）。损坏的数据包可能会被静默丢弃，导致消息被丢弃。
4) 在检测到入站数据包中的错误（如丢弃/乱序数据包）时，响应者不会停止，而是继续接收入站数据包。
5) 响应者认为操作在以下情况下完成：它已按正确顺序接收到完整消息、所有数据已提交到本地故障区域、且所有适当的有效性检查（包括变体和不变CRC检查）已完成。对于不可靠连接服务，已完成消息的定义在第9.8.2.2.7节第389页给出。
6) 请求者认为消息操作在"最后一个"或"唯一"数据包已提交到网络结构时完成。

9.8.1 验证和执行请求（Validating and Executing Requests）

本节适用于不可靠连接和不可靠数据报服务。当两种服务之间存在差异时，这些差异会被注明。两种服务之间的主要区别在于：不可靠数据报服务仅限于单数据包消息，而不可靠连接服务没有此限制。此外，不可靠数据报服务仅限于使用Send功能，这进一步简化了请求验证过程。

以下描述了响应者验证入站请求数据包所需的要求。

C9-165：响应者应验证头部的各个字段，以验证数据包的完整性。此验证过程在第9.6节"数据包传输头部验证"（第269页）中指定。包含无效字段的数据包应被响应者静默丢弃。

C9-166：对于使用不可靠连接服务的HCA，应检查PSN以检测乱序数据包。通过检查PSN，响应者可以确定数据包是新请求还是无效数据包。有关此检查的描述，请参见第9.8.2.2.1节"响应者 - 验证PSN"（第382页）。

o9-115：如果TCA实现不可靠连接服务，应检查PSN以检测乱序数据包。通过检查PSN，响应者可以确定数据包是新请求还是无效数据包。有关此检查的描述，请参见第9.8.2.2.1节"响应者 - 验证PSN"（第382页）。

C9-167：对于使用不可靠连接服务的HCA，响应者应检查数据包OpCode以确定数据包OpCode序列是否有效。此检查不适用于不可靠数据报，因为该服务仅限于单数据包消息，因此OpCode序列的概念不适用。

o9-116：如果TCA实现不可靠连接服务，响应者应检查数据包OpCode以确定数据包OpCode序列是否有效。此检查不适用于不可靠数据报，因为该服务仅限于单数据包消息，因此OpCode序列的概念不适用。

C9-168：响应者应检查数据包OpCode以确定请求的操作是否受此接收队列支持。

C9-169：响应者应验证它有足够的可用资源来接收消息。必要的资源包括有效的接收WQE（用于SEND或带立即数据的RDMA Write），以及对于SEND请求，有足够的缓冲区空间来接收请求。

C9-170：对于使用不可靠连接服务的HCA响应者，如果请求是RDMA WRITE操作，响应者应检查R_Key。如果数据包被发现是有效的、按序的且有足够的资源可用，则由响应者执行。在执行过程中，响应者可能遇到本地错误。

o9-117：如果TCA响应者实现不可靠连接服务，且支持RDMA操作，则应按如下行为：如果入站请求是RDMA WRITE操作，响应者应检查R_Key。如果数据包被发现是有效的、按序的且有足够的资源可用，则由响应者执行。在执行过程中，响应者可能遇到本地错误。

C9-171：对于使用不可靠连接或不可靠数据报服务的HCA响应者，或对于使用不可靠数据报服务的TCA响应者，响应者应在验证入站请求数据包时遵循图112所示的序列。

o9-118：如果TCA响应者实现不可靠连接服务，响应者应在验证入站请求数据包时遵循图112所示的序列。

9.8.2 不可靠连接（Unreliable Connections）

不可靠连接由两个QP之间的一对一对应关系组成。数据包从一个QP发送到另一个QP，但目标QP不生成确认。主要特征是不向请求者提供传递保证。然而，响应者可以检测数据损坏和乱序数据包。

不可靠连接服务的特征总结在表49中。

表49 不可靠连接服务特征摘要

| 特征 | 说明 |
|------|------|
| 传递保证 | 不向请求者提供保证。响应者可能丢弃消息。 |
| 排序-请求者 | 无保证。请求者不能依赖消息按序到达。 |
| 排序-响应者 | 响应者检测并丢弃乱序数据包。 |
| 排序-响应者 | 丢弃的数据包可能导致消息被丢弃。 |
| 排序-响应者 | 丢弃数据包后，响应者从新消息的第一个数据包恢复。 |
| 支持的操作 | Send和RDMA WRITE（带或不带立即数据） |
| 消息大小 | 最大2^31字节。消息可由多个数据包组成。 |

C9-172：对于使用不可靠连接服务的HCA请求者，请求者必须在每个请求数据包的BTH:PSN字段中放置一个称为当前PSN的值。

o9-119：如果TCA请求者实现不可靠连接服务，请求者必须在每个请求数据包的BTH:PSN字段中放置一个称为当前PSN的值。

在连接建立期间，传输层的客户端必须将下一个PSN编程为零到16,777,215之间的任意值。

C9-173：对于使用不可靠连接服务的HCA请求者，由传输层客户端编程的初始PSN应作为请求者生成的第一个请求数据包中的BTH:PSN出现。

o9-120：如果TCA实现不可靠连接服务，由传输层客户端编程的初始PSN应作为请求者生成的第一个请求数据包中的BTH:PSN出现。

o9-121：如果TCA实现不可靠连接服务，传输层仅在发送队列处于适当状态可以传输请求数据包时才修改（更新）PSN。

C9-175：对于使用不可靠连接服务的HCA，请求者生成的每个请求数据包必须具有比前一个请求数据包的PSN值增加"1"（模2^24）的PSN值。

o9-122：如果TCA实现不可靠连接服务，请求者生成的每个请求数据包必须具有比前一个请求数据包的PSN值增加"1"（模2^24）的PSN值。

C9-180：对于使用不可靠连接服务的HCA请求者，请求者应在以下任一条件发生时认为消息Send（或RDMA WRITE）完成：请求者已将最后一个数据包的VCRC字段的最后一个字节提交到线路（且未检测到与消息传输相关的本地错误），或者请求者已检测到与消息传输相关的本地错误，导致请求者终止发送请求。

9.8.2.2 响应者行为（Responder Behavior）

本节指定响应者在接收入站请求时的必要行为。

9.8.2.2.1 响应者 - 验证PSN

响应者维护一个期望PSN值（ePSN），用于检测多数据包请求消息中的缺失数据包和检测丢弃的消息。由于UC服务中每个入站请求数据包的PSN是连续且单调递增的，PSN序列中的中断表示请求数据包丢失或被丢弃。

C9-181：对于使用不可靠连接服务的HCA响应者，响应者应维护一个期望PSN值（ePSN）。这是响应者期望在下一个入站请求数据包的BTH中找到的PSN。

o9-126：如果TCA响应者实现不可靠连接服务，响应者应维护一个期望PSN值（ePSN）。这是响应者期望在下一个入站请求数据包的BTH中找到的PSN。

响应者的期望PSN可以在连接建立时由传输的客户端初始化为零到16,777,215之间的任意值。然而，由于响应者将接受任何OpCode为"first"或"only"的有效数据包，并使用该数据包中包含的PSN值作为其期望PSN，因此不要求对响应者的初始期望PSN进行编程。有关在连接建立时加载期望PSN的机制的完整描述，请参见第12章：通信管理（第650页）。

初始期望PSN只能在队列处于Initialized状态时由客户端设置。客户端在队列处于任何其他状态时设置PSN的尝试可能会被传输层忽略。

C9-182：对于使用不可靠连接服务的HCA，传输层仅在接收队列处于适当状态可以接收入站请求数据包时才修改（更新）其期望PSN。例如，对于HCA，传输层在队列对处于Initialized状态时不修改PSN。

o9-127：如果TCA实现不可靠连接服务，传输层仅在接收队列处于适当状态可以接收入站请求数据包时才修改（更新）其期望PSN。例如，对于HCA，传输层在队列对处于Initialized状态时不修改PSN。

C9-183：对于使用不可靠连接服务的HCA响应者，如果入站请求数据包的PSN与响应者的当前ePSN不完全匹配，则应声明该数据包为乱序。

o9-128：如果TCA响应者实现不可靠连接服务，如果入站请求数据包的PSN与响应者的当前ePSN不完全匹配，则应声明该数据包为乱序。

C9-184：使用不可靠连接服务的HCA响应者应按如下行为：如果在数据包验证期间，发现入站请求数据包的OpCode为"first"或"only"，响应者应接受该数据包，并接受该请求消息的PSN作为其新的ePSN，无论入站数据包是否乱序。无论ePSN的先前值如何，都应执行此操作。

o9-129：实现不可靠连接服务的TCA响应者应按如下行为：如果在数据包验证期间，发现入站请求数据包的OpCode为"first"或"only"，响应者应接受该数据包，并接受该请求消息的PSN作为其新的ePSN，无论入站数据包是否乱序。无论ePSN的先前值如何，都应执行此操作。

9.8.2.2.2 响应者 - OpCode序列检查

请求数据包必须符合有效OpCode序列表。OpCode序列通过检查BTH:OpCode来确定。

C9-187：对于使用不可靠连接服务的HCA响应者，响应者应按以下第(1)到(5)项所述检查数据包OpCode序列：

1) 如果这是连接建立后的第一个数据包，则数据包OpCode必须指示"first"或"only"。"middle"或"last"的OpCode意味着当前消息的至少第一个数据包已丢失，表示无效的OpCode序列。

2) 如果最后一个有效接收的数据包的OpCode指示"first"，则当前OpCode必须指示"middle"或"last"。它还必须与最后一个有效数据包中指定的操作类型匹配（SEND、RDMA WRITE）。"first"或"only"的当前OpCode意味着前一个消息的至少最后一个数据包已丢失，表示无效的OpCode序列。

3) 如果最后一个有效接收的数据包的OpCode指示"middle"，则当前OpCode必须指示"middle"或"last"。它还必须与最后一个有效数据包中指定的操作类型匹配（SEND或RDMA WRITE请求）。"first"或"only"的当前OpCode意味着前一个消息的至少最后一个数据包已丢失，表示无效的OpCode序列。

4) 如果最后一个有效接收的数据包的OpCode指示"last"，则当前OpCode必须指示"first"或"only"。"middle"或"last"的当前OpCode意味着当前消息的至少第一个数据包已丢失，表示无效的OpCode序列。

5) 如果最后一个有效接收的数据包的OpCode指示"only"，则当前OpCode必须指示"first"或"only"。"middle"或"last"的当前OpCode意味着当前消息的第一个数据包被遗漏，表示无效的OpCode序列。

o9-133：如果TCA响应者实现不可靠连接服务，响应者应按上述第(1)到(5)项所述检查数据包OpCode序列。

C9-189：对于使用不可靠连接服务的HCA响应者，如果导致无效OpCode序列的当前数据包的OpCode为"first"或"only"，则应将其视为新请求消息的第一个数据包。

9.8.2.2.7 完成消息接收

响应者在以下情况下认为给定的入站消息已成功完成：

• 检测到有效消息的开始，由BTH中"First packet"或"Only packet"OpCode的存在指示，
• 检测到同一有效消息的结束，由BTH中"Only packet"或"Last packet"OpCode的存在指示，期间PSN序列没有跳跃，
• 成功按序接收"First packet"和"Last packet"之间（含）的所有数据包，或已成功接收"Only packet"。
• 消息有效载荷已无错误地提交到本地故障区域，
• 且已成功完成所有适当的有效性检查（包括变体和不变CRC）。

在这些步骤中的任何一步检测到的故障可能会也可能不会导致关联的WQE以错误完成。在某些情况下，例如缺少"first"数据包，响应者完全可能不会消耗任何WQE。注意，在存在错误的情况下，无法保证响应者内存的状态。在检测到错误之前，给定数据包的部分或全部可能已提交到响应者的内存中。

一旦入站消息接收成功完成，响应者即完成当前WQE。

9.8.3 不可靠数据报（Unreliable Datagrams）

不可靠数据报是一种通信形式，允许源QP向可能存在于同一或多个目标端节点上的多个目标QP中的一个发送每条消息。

• 对于要发送的每条消息，必须向请求者提供目标地址（参见第11.2.2.1节"创建地址句柄"，第559页）、目标QP、目标Q_Key等。有关HCA的参数，请参见第11.4.1.1节"提交发送请求"（第612页）。
• 响应者必须向客户端传递请求者的地址、QP等。有关HCA要求的更多详细信息，请参见第11.4.2.1节"轮询完成"（第623页）。

C9-200：源UD消息的设备应将UD消息大小限制为单个数据包。数据包应不大于源和目标之间的PMTU（否则将被丢弃）。

C9-201：源和汇UD消息的设备应满足基本不可靠服务的要求（参见第9.8节"不可靠服务"，第375页至第9.8.1节"验证和执行请求"，第375页）。

C9-202：源和汇IBA UD消息的设备应满足第9.9节"错误检测和处理"（第396页）中指定的要求。

第一种编码原始数据报的方法仅用于IPv6数据报。数据包有效载荷可以包含由IETF对IPv6头部"next header"字段编码定义的任何传输或网络协议，但不包括指示下一个头部是IBA传输头部的任何编码。

C9-208：CA不应生成其LRH指示原始数据报且其IPv6"next header"指示IBA传输的出站数据包，并将丢弃任何此类入站数据包。TCA可以以任何方式报告此错误。

第二种编码原始数据报的方法使用IBA定义的原始头部（RWH）。RWH包含16位Ethertype字段 - RWH在第5.3节"原始数据包格式"（第161页）中描述。RWH用于定义封装在数据包有效载荷中的协议头部。通常，第二种方法用于允许不受IPv6 next header支持的协议 - 应注意，任一方法都可用于传输IPv6数据报。

o9-145：如果CA实现原始数据报支持，原始数据报的数据包有效载荷必须始终为模4大小，因为LRH数据包长度以4字节增量描述长度。如果封装的有效载荷大小不是4字节的倍数，则有效载荷应填充到4字节的倍数。

o9-146：如果CA实现原始数据报支持，用于注入和消费原始数据报的QP应在本地管理，即QP与给定原始数据报服务的关联是实现相关的。

o9-147：如果CA实现原始数据报支持，它可以支持一个或多个QP用于原始数据报操作。

9.9 错误检测和处理（Error Detection and Handling）

IBA使用分层错误管理架构（LEMA）方法。每一层负责检测和管理适合该层的错误，然后将数据包或消息传递到协议栈中的下一层。

因此，传输层响应传输层特有的错误，包括数据包头部中的错误和未能正确传输消息的故障。

传输层检测到的错误会报告给传输层的客户端。在本节中，传输层与其客户端之间的接口在概念上表示为发送或接收队列。在HCA的情况下，传输层通过向完成队列（CQ）上的完成队列条目（CQE）写入完成代码来向其客户端指示错误。与往常一样，TCA可以按其认为合适的方式报告错误（或不报告）。

为了简化讨论，错误行为分别针对请求者和响应者端进行讨论。这导致以下各节中描述请求者和响应者侧错误的摘要表之间有少量重复。具体来说，当错误由响应者检测并报告给请求者时会出现重叠。然而，这些重叠区域严格限于可靠服务类别。

图115 请求者/响应者错误检测

[请求者侧：发送队列 → 请求数据包 → 本地错误检测 → CQ
响应者侧：确认数据包 → 远程错误检测响应 → 确认生成 → 接收队列 → 本地错误检测 → CQ]

请求者向其客户端报告的错误分为两类。第一类是本地检测的错误，即仅由请求者侧检测到的错误。本地检测错误的一个示例是请求者在发送请求期间访问其本地内存时检测到的保护故障。

第二类是远程检测的错误，即由响应者检测并通过响应数据包中的NAK代码报告给请求者的错误。远程检测的错误仅适用于可靠服务类别（可靠连接和可靠数据报）。

请求者侧有两类错误（本地和远程检测的），而响应者侧只有本地检测的错误。

响应者侧在检测到本地错误时，可能需要向请求者报告错误，或向其本地客户端报告错误，或两者都需要，或两者都不需要。错误报告对象的选择由服务类别（可靠与不可靠）和检测到的具体错误决定。

以下各节的关键重点是按错误向传输层客户端报告的方式进行分类，以及发送（接收）队列在检测到错误后必须表现的行为。因此，本节不仅按错误检测位置分类，还按报告对象分类。

9.9.1 向Verbs层报告错误（Reporting Errors to the Verbs Layer）

对于HCA，IBA软件接口定义了三种可以通过verbs层报告的错误类型。这些称为即时错误、完成错误和异步错误。在这三种类型中，传输层只能报告完成错误或异步错误。这是因为即时错误在WQE被提交到传输层之前就已由verbs层检测到。

表55 由传输层检测的软件错误类型

| IBA软件定义的错误类型 | 由IBA传输检测 |
|------------------------|--------------|
| 即时错误 | 否 |
| 完成错误 - 接口检查 | 是 |
| 完成错误 - 处理错误 | 是 |
| 异步错误 - 关联类型 | 是 |
| 异步错误 - 非关联类型 | 是 |

完成错误有两个类别：接口检查和处理错误。接口检查是在数据放置到链路之前检测到的提供给通道接口的信息中的错误。处理错误是通道接口在处理工作请求期间遇到的错误。

9.9.2 请求者侧错误行为（Requester Side Error Behavior）

如上所述，请求者检测来自本地或远程的错误。

9.9.2.1 请求者侧错误检测 - 本地检测的错误

本地检测的错误反映了请求者通道接口中发生的错误状况、来自响应者侧的缺失响应（超时）或针对序列错误或RNR NAK的过多重试。

请求者的本地检测错误可能在请求数据包生成期间、响应数据包处理期间或由于超时而发生。

C9-209：对于使用RC、UC或UD服务的HCA请求者，以及使用UD服务的TCA请求者，请求者应按如下行为：对于在传输请求数据包期间检测到的本地传输错误，CA应停止受影响QP的传输，存储与错误相关的状态直到任何先前未完成的WQE完成，最后完成受影响的WQE。如果错误类型要求，QP应置于错误状态。

o9-149：如果TCA请求者实现可靠连接或不可靠连接服务，则应按如下行为：对于在传输请求数据包期间检测到的本地传输错误，CA应停止受影响QP的传输，存储与错误相关的状态直到任何先前未完成的WQE完成，最后完成受影响的WQE。如果错误类型要求，QP应置于错误状态。

o9-150：如果CA请求者实现可靠数据报服务，则应按如下行为：对于在传输请求数据包期间检测到的本地传输错误，CA应停止受影响EEC的传输，存储与错误相关的状态直到任何先前未完成的WQE完成，最后完成受影响的WQE。如果错误类型要求，EEC应置于错误状态。

这是为了维护WQE的有序完成，并确保错误在发生错误的WQE中被正确报告。

9.9.2.1.1 请求者错误重试计数器

C9-210：对于使用可靠连接服务的HCA，为了检测过多重试，请求者应维护RNR NAK和错误重试计数器，执行第9.9.2.1.1节"请求者错误重试计数器"（第398页）中描述的逻辑功能。

o9-151：如果CA请求者实现可靠数据报服务，或如果TCA请求者实现可靠连接服务，请求者应按如下行为：为了检测过多重试，请求者应维护RNR NAK和错误重试计数器，执行第9.9.2.1.1节"请求者错误重试计数器"（第398页）中描述的逻辑功能。

实现可以按其选择的任何方式实现这些重试计数器，但为清楚起见，这里将其描述为递减计数器，初始化为在终止操作并创建最终完成错误之前允许的重试次数。有关HCA中这些计数器的编程，请参见第10.2.4.3节"修改队列对属性"（第437页）。

RNR NAK重试计数器在每次响应者返回RNR NAK时递减。如果请求者的RNR NAK重试计数器为零，且接收到RNR NAK数据包，则发生RNR NAK重试错误。每次RNR NAK被清除（即返回RNR NAK以外的确认消息）时，重试计数器重新加载。一个例外是如果RNR NAK重试计数器设置为7，此值表示无限重试，计数器不递减。

错误重试计数器在每次请求者因本地Ack超时、NAK-序列错误或隐含NAK而必须重试数据包时递减。如果请求者的重试计数器递减到零，可能实现以下两种情况之一。

如果不支持自动路径迁移，或自动路径迁移已完成，则应在完成中报告"传输重试计数器超出"错误。

o9-152：如果CA支持自动路径迁移，则在潜在可恢复错误及其重试之后，请求者可以迁移连接或EE上下文并再次执行错误重试，然后最终报告完成错误。有关更多信息，请参见第17.2.8节"自动路径迁移"（第1031页）。

每次数据包被正确确认时，重试计数器应重新加载。

9.9.2.2 请求者侧错误检测 - 远程检测的错误

远程检测的错误在响应者向请求者报告错误时发生。远程检测的错误是可靠服务类别所独有的。

远程检测的错误通过确认消息中携带的NAK代码报告。然而，并非所有NAK代码都导致向请求者的客户端报告错误。

在可能的NAK代码中，两个（NAK-序列错误和NAK-RNR）指示应由请求者自动重试的操作。NAK-序列错误和NAK-RNR以外的NAK代码指示必须立即向请求者的客户端报告且无法重试的故障。

请求者必须实现五种不同的错误响应行为集。对于任何给定错误执行哪种行为在上方表56中显示。本节指定错误响应行为。

9.9.2.4.1 请求者A类故障行为

A类错误是通过重试机制可由传输层恢复的错误。如果重试成功，对传输层的客户端（例如verbs层）没有可见影响。

唯一的A类错误是数据包序列错误、隐含NAK序列错误、本地Ack超时错误和RNR NAK。数据包序列错误和RNR NAK是远程检测的。本地Ack超时错误和隐含NAK序列错误由请求者本地检测。

C9-212：对于使用可靠连接服务的HCA，传输层每次重试请求者A类错误时，应递减重试计数器。有一个与数据包序列错误和本地Ack超时错误相关的重试计数器，以及一个与RNR NAK相关的不同重试计数器。只要重试计数未过期，传输层可以继续重试这些错误。重试这些错误的协议在第9.7节"可靠服务"（第280页）中给出。

o9-153：如果TCA请求者实现可靠连接服务，或如果CA请求者实现可靠数据报服务，传输层每次重试请求者A类错误时，应递减重试计数器。有一个与数据包序列错误和本地Ack超时错误相关的重试计数器，以及一个与RNR NAK相关的不同重试计数器。只要重试计数未过期，传输层可以继续重试这些错误。重试这些错误的协议在第9.7节"可靠服务"（第280页）中给出。

C9-213：对于使用可靠连接服务的HCA请求者，由于请求者A类错误是可恢复的，请求者不应将其报告给传输层的客户端，除非重试计数过期。

o9-154：如果TCA请求者实现可靠连接服务，或如果CA请求者实现可靠数据报服务，由于请求者A类错误是可恢复的，请求者不应将其报告给传输层的客户端，除非重试计数过期。

有关重试计数过期后HCA如何报告错误的讨论，请参见第10.10.2.2节"完成错误"（第531页）。

9.9.2.4.2 请求者B类故障行为

C9-214：响应请求者B类错误时，对于可靠数据报以外的服务，请求者应以错误完成当前WQE，将发送队列转换到错误状态，并将提交到发送队列的任何后续WQE标记为刷新。

o9-154.a1：对于实现可靠数据报服务的CA，请求者在响应请求者B类错误时，应执行第9.7.8.5节"处理QP错误 - RESYNC"（第368页）中描述的操作。如果在所述RESYNC过程之后，消息仍然出错，请求者应以错误完成当前WQE，将发送队列转换到错误状态，并将提交到发送队列的任何后续WQE标记为刷新。

对于HCA，错误以"完成 - 处理类型"与适当的错误类型一起报告（有关更多详细信息，请参见第10.10.2.2节"完成错误"，第531页）。

队列应由传输层转换到错误状态，以防止在客户端（例如HCA的verbs层）发现错误之前将更多WQE提交到发送队列时可能发生的竞态条件。这与图124"QP/EE上下文状态图"（第452页）中显示的发送队列状态图一致。

最后，发送队列中失败WQE之后的所有WQE也以"完成 - 刷新错误"状态完成。

对于RC模式，注意这些请求中的某些可能已由请求者提交到线路，甚至可能已由响应者执行和完成。无法防止这种情况，因为响应者可能在请求者检测到本地错误之前就已执行了请求。因此，响应者的本地状态必须被视为未知。

对于可靠数据报服务，请求者的EE上下文终止当前消息传输，向当前调度的发送队列发出错误信号，并将当前调度的发送队列从调度器中移除。然后EE上下文调度下一个请求服务的发送队列并继续。导致错误的发送队列按上述方式行为。

C9-215：当发送队列处于错误状态时，它必须静默丢弃到达的任何确认消息。

9.9.2.4.3 请求者C类故障行为

由于C类错误无法与任何特定WQE关联，因此无法将特定WQE标记为以错误完成。

C9-216：如果请求者C类错误可以与QP关联，则发送队列应转换到错误状态，所有未完成的WQE以"完成 - 刷新错误"状态完成。

o9-155：如果CA请求者实现可靠数据报服务，且请求者C类错误可以与EE上下文关联，则其发送侧应转换到错误状态，对于HCA，报告的错误为"关联异步错误"。

有关HCA错误报告的更多详细信息，请参见第11.6.3.2节"关联异步错误"（第639页）。

如果请求者C类错误无法与特定资源关联，则错误可能无法以任何详细方式报告，或根本无法报告。有关这种情况下的HCA错误处理，请参见第11.6.3.4节"非关联异步错误"（第641页）。

9.9.2.4.4 请求者D类故障行为

D类错误仅发生在可靠数据报服务中。

o9-156：如果CA请求者实现可靠数据报服务，则应按如下行为：对于请求者D类错误类别，传输层应将请求者的EE上下文转换到错误状态，终止当前消息传输，向当前调度的发送队列发出错误信号，并取消当前调度的发送队列。在保持错误状态期间，EE上下文继续将任何其他请求服务的发送队列转换到错误状态。

每个发送队列（QP）按B类错误行为；将其当前WQE标记为以错误完成，将QP转换到错误状态，并刷新所有后续WQE。

9.9.2.4.5 请求者E类故障行为

E类错误发生在请求者接收到PSN与其期望PSN不匹配的确认消息时。这些错误仅发生在可靠服务类别中。

这些错误不会报告给上层。PSN不匹配的确认消息被推定为"幽灵"确认消息或重复确认消息。

幽灵确认消息是在网络结构中存在足够长时间，以致于在连接销毁和随后新连接建立后仍然存活的确认消息。

重复确认消息发生在请求者认为其原始请求消息在网络结构中丢失时重新发送请求消息的情况下。如果两个请求消息最终都到达响应者，响应者可能为每个请求生成确认消息。

C9-217：对于使用可靠连接服务的HCA请求者，响应请求者E类错误时，请求者应丢弃确认消息。然而，此规则有一个例外。对于可靠连接服务，重复确认消息可能被响应者用来向请求者传递端到端流控制信用（"非请求确认"）。因此，如果确认消息的PSN比请求者期望PSN小1，则请求者必须恢复端到端信用并丢弃消息的其余部分。此行为在第9.7.7.2节"端到端（消息级别）流控制"（第347页）中详细说明。

o9-157：如果TCA请求者实现可靠连接服务，或如果CA请求者实现可靠数据报服务，响应请求者E类错误时，请求者应丢弃确认消息。然而，此规则有一个例外。对于可靠连接服务，重复确认消息可能被响应者用来向请求者传递端到端流控制信用（"非请求确认"）。因此，如果确认消息的PSN比请求者期望PSN小1，则请求者必须恢复端到端信用并丢弃消息的其余部分。此行为在第9.7.7.2节"端到端（消息级别）流控制"（第347页）中详细说明。

应注意，即使确认实际上是一个具有错误信用的幽灵确认，信用机制最终也会恢复，不会向上层报告错误。

9.9.2.4.6 请求者F类故障行为

C9-218：请求者F类错误发生在CQ不可访问或已满且尝试完成WQE时。受影响的QP应移至错误状态，并按第11.6.3.1节"关联异步事件"（第637页）中描述的方式生成关联异步错误。当前WQE和任何后续WQE处于未知状态。

9.9.3.1 响应者侧错误响应

响应者侧共描述了八个故障行为类别。故障行为根据是否向响应者侧的客户端报告错误、是否通过NAK代码向请求者报告错误，以及是否从接收队列消耗WQE进行分组。

9.9.3.1.1 响应者A类故障行为

A类错误可追溯到格式不正确或无效的WQE，或与接收者QP相关的其他错误。这些错误不是由发送者引起的。

对于可靠数据报服务，EEC继续操作。

如果响应者是HCA，这些错误作为"完成错误"或"关联异步错误"报告给verbs层。有关完成错误的讨论，请参见第10.10.2.2节"完成错误"（第531页）；有关异步错误的讨论，请参见第10.10.2.3节"异步错误"（第531页）。

• 不通知响应者的客户端
• 在不消耗当前接收WQE（如有）的情况下终止当前消息
• 等待新消息的第一个数据包（可能大于期望PSN）

"当前消息"指自最近接收到的"first"或"only"OpCode以来接收到的所有数据包，不包括当前数据包（导致D1类错误的数据包）。

9.9.3.1.10 响应者J类故障行为

此类本地检测的错误仅发生在RC QP上。J类错误报告给响应者侧客户端，但不通过NAK代码报告给请求者。

响应者J类错误发生在响应者接收到包含IETH中无效R_Key的SEND with Invalidate请求（SEND last with Invalidate或SEND only with Invalidate）时。

在错误优先级方面，与验证数据包头部和执行SEND操作（在其上附加了Invalidate）相关的所有其他错误在R_Key违规之前报告。

o9-162.2.1：以下声明构成检测和报告与SEND with Invalidate请求相关的R_Key违规的要求：

1) R_Key违规（如果发生）不应在附加了invalidate的SEND请求成功执行之前报告。执行SEND请求产生的任何错误必须在invalidate操作产生的错误之前报告。

2) 如果因执行底层SEND操作而发生错误，则不应报告与invalidate操作相关的错误。

3) 如果底层SEND操作正常执行，则无论关联的invalidate操作成功与否，都应消耗接收WQE。换句话说，由于执行了附加invalidate的底层SEND请求，接收WQE将已被消耗。因此，即使invalidate操作失败，接收WQE也始终被消耗。

4) 接收SEND with Invalidate请求的接收WQE不应在相应的invalidate操作完成之前完成。

5) 如果在执行invalidate操作过程中检测到错误，应发生以下操作：
   a) 经历SEND with Invalidate错误的WQE标记为以错误完成。
   b) QP转换到错误状态。
   c) 同一RQ上的后续WQE标记为完成 - 刷新错误。
   d) 不向请求者发送NAK。

9.11 静态速率控制（Static Rate Control）

随着网络结构中流量负载的增加，资源争用也会增加。拥塞管理用于平滑操作、提高网络结构效率、提高有效带宽和改善面对此类争用时的平均数据包延迟。

有多种机制可以解决此问题。对于此版本的IBA，仅定义静态速率控制。规范的未来版本可能提供额外机制的定义。

静态速率控制是端节点将数据注入网络结构的速率保持在预配置值以下的能力。

IBA在CA中支持静态速率控制，以减少由较高速CA以超过路径或目标CA吸收能力的速率向子网内的路径注入数据包而引起的拥塞。例如，具有10 Gbps链路的CA通过中间交换机向具有2.5 Gbps链路的CA传输数据包。在这种情况下，交换机需要引入"背压"（限制返回给较快链路的链路级流控制信用），以防止较慢的链路被溢出。

IBA提供三种机制来管理静态速率控制：

• 设备提供的端口速率信息（参见第16.3.3.1节"ClassPortInfo"，第991页）
• FM支持的源/目标对最佳可能速率报告（参见第15.2.5.16节"PathRecord"，第899页）
• CA"数据包间延迟"参数在连接设置MAD中（如下所述）

9.11.1 异构链路的静态速率控制

通道适配器具有限制注入数据包速率的能力。此速率基于子网本地目标端口。

o9-163：如果端口能够以大于2.5 Gbits/sec的速率向网络结构注入，则此端口应提供本节中定义的静态速率控制。

链路速率的支持由PortInfo:LinkWidthSupported和PortInfo:LinkSpeedSupported属性定义。有关这些属性的描述，请参见表145"PortInfo"（第822页）。

o9-164：如果端口配置为以大于2.5 Gbits/sec的速率向网络结构注入，则自上一个数据包从此源端口调度注入到同一目标端口以来，在经过可编程的时间量之前，不应调度数据包注入到本地子网。

链路速率的配置由PortInfo:LinkWidthActive和PortInfo:LinkSpeedActive属性定义。有关这些属性的描述，请参见表145"PortInfo"（第822页）。

在上述中，目标端口是指本地子网内目标端口的完整DLID（即基础DLID加路径位），即使对于全局路由的数据包也是如此；而源端口是指入口端口，与SLID无关（即适用于与此端口关联的所有SLID）。

等待传输后续数据包的时间基于传输当前数据包所需的时间。

o9-165：如果端口能够以大于2.5 Gbits/sec的速率向网络结构注入，则调度发往同一DLID并来自同一源端口的数据包之间的等待时间由数据包间延迟（IPD）确定。具体来说，如果数据包b要发送到与数据包a相同的DLID并使用相同的源端口，则数据包b不应在自数据包a被调度以来经过时间Ts之前被调度，其中Ts的计算方式为：（IPD + 1）乘以传输第一个数据包所需的时间。此外，传输数据包所需的时间计算为LRH:PktLen*4/Lr，其中Lr是从PortInfo:LinkWidthActive和PortInfo:LinkSpeedActive属性获得的端口速度。

数据包间延迟（IPD）值是8位整数，按下表所示进行解释。注意，所有256个可能的值都是合法的。还请注意，上表是示例，不包括所有必需的IPD。有关CA需要支持哪些IPD值，请参见第17.2.6节"静态速率控制"（第1029页）。
