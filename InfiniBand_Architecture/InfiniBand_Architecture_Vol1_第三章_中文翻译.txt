第三章：体系结构概述

本章提供了 InfiniBand 体系结构 (IBA) 特性、能力、组件和元素的自顶向下的描述，并描述了各种操作原理。这是一个高层次的概述，旨在作为参考指南，因此为了清晰起见，某些细节被有意省略。

IBA 定义了一个系统域网络 (SAN)，用于连接多个独立的处理器平台（即主机处理器节点）、I/O 平台和 I/O 设备（参见图 6）。IBA SAN 是一个通信和管理基础设施，支持一个或多个计算机系统的 I/O 和进程间通信 (IPC)。IBA 系统的范围可以从具有一个处理器和几个 I/O 设备的小型服务器到具有数百个处理器和数千个 I/O 设备的大规模并行超级计算机装置。此外，IBA 对互联网协议 (IP) 的友好特性允许桥接到互联网、内部网或连接到远程计算机系统。

IBA 定义了一个交换式通信 Fabric，允许许多设备在受保护的、远程管理的环境中以高带宽和低延迟并发通信。终端节点可以通过多个 IBA 端口进行通信，并可以利用通过 IBA Fabric 的多条路径。IBA 端口和网络路径的多重性被用于故障容错和增加数据传输带宽。

IBA 硬件将大部分 I/O 通信操作从 CPU 卸载。这允许多个并发通信，而没有与通信协议相关的传统开销。IBA SAN 为其 I/O 和 IPC 客户端提供零处理器拷贝的数据传输，无需内核参与，并使用硬件提供高度可靠、容错的通信。

图 6  IBA 系统域网络
一个 IBA 系统域网络由处理器节点和 I/O 单元组成，通过由级联交换机和路由器构成的 IBA Fabric 连接。I/O 单元的复杂度范围可以从单个 ASIC IBA 连接设备（如 SCSI 或 LAN 适配器）到在复杂度上可与处理器节点媲美的大内存丰富 RAID 子系统。

3.1 体系结构范围

本卷 InfiniBand 体系结构规范定义了互连 Fabric、路由元素、终端节点、管理基础设施以及通信格式和协议。它不指定 I/O 命令或集群服务。

例如，考虑一个 IBA SCSI 适配器。IBA 不定义磁盘 I/O 命令、SCSI 适配器如何与磁盘通信、操作系统 (OS) 如何看待磁盘设备，也不定义集群中哪个节点拥有磁盘适配器。IBA 是这些操作中每一个的必要基础，但不直接定义其中任何一个。相反，IBA 定义了数据和命令如何在处理器节点上的 I/O 驱动程序与 SCSI 适配器之间传输。

IBA 处理多计算机环境中 I/O 和 IPC 的数据通信。它支持 I/O 所需的高带宽和可扩展性。它满足 IPC 所需的极低延迟和低 CPU 开销。通过 IBA，OS 可以为其客户端提供绕过 OS 内核并直接访问 IBA 网络通信硬件的通信机制，从而实现高效的消息传递操作。

IBA 非常适合最新的计算模型，并将成为新形式 I/O 和集群通信的构建模块。IBA 允许 I/O 单元之间以及与系统中任何或所有处理器节点之间进行通信。因此 I/O 单元具有与任何处理器节点相同的通信能力。

该体系结构针对包含多个独立进程和线程（消费者）的单元进行了优化，如图 10 所示。每个通道适配器构成 Fabric 上的一个节点。该体系结构支持每个单元有多个通道适配器，每个通道适配器提供一个或多个连接到 Fabric 的端口，在这种情况下，处理器节点对 Fabric 表现为多个终端节点。

图 10  处理器节点

在处理器节点中，消息和数据服务是一个 OS 组件，不在本文档的范围之内。本文档指定消息和数据服务与通道适配器之间的语义接口。这个语义接口被称为 IBA 动词（Verbs）。动词描述了配置、管理和操作主机通道适配器所需的功能。这些动词标识了每个特定功能需要包含的适当参数。动词不是 API，而是为 OS 厂商提供指定 API 的框架。

IBA 被设计为一阶网络，因此它定义了主机行为（动词）并定义了内存操作，使得通道适配器可以尽可能靠近内存复合体。它在同意的消费者之间提供独立的直接访问，无论这些消费者是 I/O 驱动程序和 I/O 控制器，还是在对等基础上通信的软件进程。IBA 提供通道语义（发送和接收）和直接内存访问，并具有防止非参与消费者访问的保护级别。

3.2 通信

3.2.1 队列

IBA 操作的基础是消费者能够排入一组由硬件执行的指令。这个机制被称为工作队列。工作队列总是成对创建，称为队列对 (QP)，一个用于发送操作，一个用于接收操作。一般而言，发送工作队列保存导致数据在消费者的内存与另一个消费者的内存之间传输的指令，接收工作队列保存关于将从另一个消费者接收的数据放置在何处的指令。另一个消费者被称为远程消费者，即使它可能位于同一节点上。IBA 专门描述了主机通道适配器 (HCA) 的队列关系，但不描述 I/O 单元，因为 I/O 单元不一定受到主机环境中存在的第二方和第三方互操作性的约束（即 HCA 供应商、OS 供应商和 IHV 的 I/O 驱动程序或 ISV 的使用 IPC 的应用程序之间的互操作性）。以下描述 HCA 队列模型。

消费者提交一个工作请求 (WR)，这导致一个称为工作队列元素 (WQE) 的指令被放置在适当的工作队列上。通道适配器按照 WQE 被放置在工作队列上的顺序执行它们。当通道适配器完成一个 WQE 时，一个完成队列元素 (CQE) 被放置在完成队列上²。每个 CQE 指定工作完成所需的所有信息，并且直接包含该信息或指向包含该信息的其他结构（例如关联的 WQE）。

图 11  消费者队列模型
² WQE 和 CQE 不是体系结构化的实体，只有工作请求动词是体系结构化的。

每个消费者可以有自己的一组工作队列，每对工作队列彼此独立。每个消费者创建一个或多个完成队列，并将每个发送和接收队列与特定的完成队列关联。工作队列对的发送和接收队列不必使用同一个完成队列。

因为某些工作队列需要远程节点的确认，并且某些 WQE 使用多个数据包来传输数据，通道适配器可以同时有多个正在进行的 WQE，甚至来自同一工作队列。因此，CQE 被放置到完成队列的顺序是不确定的，除了同一工作队列的 CQE 通常按照对应 WQE 被放置到工作队列的顺序放置³。
³ 可靠数据报服务的接收完成是例外，因为在多个 EE 上下文上的并发接收可能导致乱序放置。

图 12  工作队列操作

发送队列操作分为三类：SEND、远程内存访问 (RDMA) 和内存绑定 (MEMORY BINDING)。

• 对于 SEND 操作，WQE 指定消费者内存空间中的一块数据，供硬件发送到目的地，让目的地已排队的接收 WQE 指定将数据放置在何处。

• 对于 RDMA 操作，WQE 还指定远程消费者内存中的地址。因此 RDMA 操作不需要涉及目的地的接收工作队列⁴。RDMA 操作有 3 种类型：RDMA-WRITE、RDMA-READ 和 ATOMIC。
    • RDMA-WRITE 操作规定硬件将数据从消费者的内存传输到远程消费者的内存。
    • RDMA-READ 操作规定硬件将数据从远程内存传输到消费者的内存。
    • ATOMIC 操作规定硬件对远程 64 位内存位置执行读取。目标返回读取的值，并通过将更新的值写回同一位置来有条件地修改/替换远程内存内容。
⁴ 带立即数据的 RDMA Write 确实涉及目的地的接收工作队列。

• MEMORY BINDING 指示硬件更改内存注册关系（参见第 10.6.6.2 节）。它将内存窗口（Memory Window）与现有内存区域（Memory Region）内的指定范围关联（绑定）。内存绑定允许消费者指定它与其他节点共享的已注册内存的哪些部分（即远程节点可以访问的内存），并指定读写权限。结果产生一个内存密钥 (R_KEY)，消费者将其传递给远程节点，供它们在 RDMA 操作中使用。

接收队列操作只有一种，即指定接收数据缓冲区。

• RECEIVE WQE 指定当另一个消费者执行 SEND 操作时，硬件将接收到的数据放置在何处。每次远程消费者成功执行 SEND 操作时，硬件从接收队列取出下一个条目，将接收到的数据放置在该接收 WQE 指定的内存位置，并在完成队列上放置一个 CQE，向消费者指示接收操作已完成。因此 SEND 操作的执行会导致远程消费者处的接收队列操作。

通常 RDMA 操作不会消耗目的地的接收 WQE，但有一个例外。即指定了立即数据的 RDMA WRITE 操作。立即数据是 SEND 或 RDMA WRITE 指令中可选提供的 32 位信息，作为操作的一部分传输，但不是将立即数据写入内存，而是将该数据作为 RECEIVE CQE 状态的特殊字段处理并返回。这意味着带立即数据的 RDMA WRITE 将消耗目的地的一个 RECEIVE WQE。

3.2.2 连接

IBA 支持面向连接和数据报服务。对于连接服务，每个 QP 与恰好一个远程消费者关联。在这种情况下，QP 上下文配置了远程消费者队列对的标识。远程消费者由端口和 QP 编号标识。端口由本地 ID (LID) 和可选的全局 ID (GID) 标识。在通信建立过程中，这些信息和其他信息在两个节点之间交换。

对于数据报服务，QP 不绑定到单个远程消费者，而是 WQE 中的信息标识目的地。需要与每个目的地进行类似于连接建立过程的通信建立过程来交换该信息。

3.3 通信栈

IBA 的通信栈如图 13 所示。该体系结构提供了多种 IBA 事务，消费者可以使用它们与远程消费者执行事务。消费者将工作队列元素 (WQE) 提交到 QP，通道适配器解释每个 WQE 以执行操作。

图 13  IBA 通信栈

对于发送队列操作，通道适配器解释 WQE，创建请求消息，必要时将消息分段为多个数据包，添加适当的路由头部，并通过适当的端口发送数据包。

端口逻辑通过链路发送数据包，交换机和路由器通过 Fabric 将数据包中继到目的地。

当目的地接收到数据包时，端口逻辑验证数据包的完整性。通道适配器将接收到的数据包与特定 QP 关联，并使用该 QP 的上下文来处理数据包和执行操作。如有必要，通道适配器创建响应（确认）消息并将该消息发送回发起方。

某些请求消息的接收导致通道适配器从接收队列消耗一个 WQE。当它这样做时，与被消耗的 WQE 对应的 CQE 被放置在适当的完成队列上，这导致向拥有该 QP 的消费者发出工作完成。

3.4 IBA 组件

IBA 系统中的设备分类为：
• 交换机
• 路由器
• 通道适配器
• 中继器
• 互连交换机、路由器、中继器和通道适配器的链路

管理基础设施包括：
• 子网管理器
• 通用服务代理

3.4.1 链路与中继器

链路互连通道适配器、交换机、中继器和路由设备以形成 Fabric。链路可以是铜缆、光缆或背板上的印刷电路布线。中继器是透明⁵设备，用于扩展链路的范围。InfiniBand 体系结构第 2 卷规定了各种介质类型的链路和中继器要求，并定义了 I/O 设备的各种模块外形尺寸。第 1 卷中描述的体系结构独立于链路类型和外形尺寸。
⁵ 透明的意思是中继器仅参与物理层协议级别，节点不知道它们的存在。

链路和中继器不可直接寻址，但链路状态可以通过链路两端的设备确定。

3.4.2 通道适配器

通道适配器是处理器节点和 I/O 单元中生成和消耗数据包的 IBA 设备。IBA 定义了两种类型的通道适配器：主机通道适配器 (HCA) 和目标通道适配器 (TCA)。HCA 提供消费者接口，该接口提供 IBA 动词指定的功能。IBA 不指定 TCA 的消费者接口语义。

通道适配器是一个可编程 DMA 引擎，具有特殊的保护功能，允许在本地和远程发起 DMA 操作。

图 14  通道适配器

通道适配器可以有多个端口。通道适配器的每个端口被分配一个本地 ID (LID) 或一个 LID 范围。每个端口都有自己的发送和接收缓冲区集，使得每个端口都能够并发发送和接收。缓冲通过虚拟通道 (VL) 进行，其中每个 VL 都有自己的流控制。

通道适配器提供内存转换与保护 (MTP) 机制，将虚拟地址转换为物理地址并验证访问权限。本文档不指定特定的内存管理机制，也不为 TCA 指定此类机制的要求。

通道适配器以队列对 (QP) 的形式向其消费者提供多个通信接口实例，队列对由一个发送和一个接收工作队列组成。

子网管理器为每个物理端口配置本地地址，即端口的 LID。与子网管理器通信以配置通道适配器的实体被称为子网管理代理 (SMA)。

每个通道适配器都有一个由通道适配器供应商分配的全局唯一标识符 (GUID)。由于子网管理器分配的本地 ID 不是持久的（即可能从一个上电周期到下一个发生变化），通道适配器 GUID（称为节点 GUID）成为用于持久标识通道适配器的主要对象。此外，每个端口都有一个由通道适配器供应商分配的端口 GUID。

3.4.3 交换机

交换机主要根据数据包本地路由头部中的目的地址传递数据包。交换机也消耗和产生管理交换机本身所需的数据包。可选地，交换机端口可以包含物理 TCA 端口的属性。

IBA 交换机是子网内路由（子网间路由由 IBA 路由器提供）的基本路由组件。交换机通过在链路之间中继数据包来互连链路。

图 15  IBA 交换机元素

交换机暴露两个或多个端口，数据包在这些端口之间被中继。子网内的每个目的地被配置一个或多个唯一的本地标识符 (LID)。数据包包含指定目的地 LID 的目的地址。从交换机的角度来看，目的 LID 代表通过交换机的路径。交换机元素配置了转发表。各个数据包根据数据包的目的 LID 和交换机的转发表被转发到交换机内的出站端口。

IBA 交换机支持单播转发，并且可以支持多播转发。单播是将单个数据包传递到单个目的地，多播是 Fabric 将单个数据包传递到多个目的地的能力。

子网管理器配置交换机，包括加载它们的转发表。

为了最大限度地提高可用性，可以在交换 Fabric 内部署终端节点之间的多条路径。如果交换机之间有多条可用路径，子网管理器可以使用这些路径进行冗余或基于目的 LID 的负载分担。在存在多条路径的情况下，子网管理器可以通过重新加载 Fabric 受影响区域中交换机的转发表来重新路由数据包，绕过故障链路。

3.4.4 路由器

与交换机一样，路由器不生成也不消耗数据包（管理数据包除外）。它们只是将数据包传递下去。路由器基于数据包的全局路由头部转发数据包，并且当数据包从一个子网传递到另一个子网时实际替换数据包的本地路由头部。

IBA 路由器是子网间路由（子网内路由由 IBA 交换机提供）的基本路由组件。路由器通过在子网之间中继数据包来互连子网。

图 16  IBA 路由器元素

路由器暴露一个或多个端口，数据包在这些端口之间被中继。路由器可以与其他设备（如通道适配器或交换机）嵌入在一起。

路由器对终端节点不是完全透明的，因为源必须指定路由器的 LID，并且还要提供目的地的 GID。

每个子网都用一个子网 ID（称为子网前缀）唯一标识。子网管理器通过 PortInfo 属性将子网前缀编程到所有端口。当与端口 GUID 组合时，此组合成为端口的自然 GID。端口可能还有其他本地管理的 GID。

从路由器的角度来看，GID 的子网前缀部分代表通过路由器的路径。IPv6 指定路由器之间执行的协议以导出它们的转发表。各个数据包根据数据包的目的 GID 和路由器的转发表被转发到路由器内的出站端口。

每个路由器将数据包通过下一个子网转发到另一个路由器，直到数据包到达目标子网。最后一个路由器使用与目的 GID 关联的 LID 作为目的 LID 发送数据包。

子网管理器为路由器配置有关子网的信息，例如要使用哪些 VL 和分区信息。

为了最大限度地提高可用性，可以在 Fabric 内部署子网之间的多条路径。如果有多条可用路径，路由器可能使用这些路径进行冗余或负载分担。在存在多条路径的情况下，路由器可以重新路由数据包，绕过故障子网。

3.4.5 管理组件

IBA 管理为子网管理器和支持多种通用管理服务的基础设施做了规定。管理基础设施要求每个节点中有一个子网管理代理，并定义了一个允许额外通用服务代理的通用服务接口。

该体系结构定义了一个通用管理数据报 (MAD) 消息结构，用于管理器和管理代理之间的通信。

3.4.5.1 子网管理器

子网管理器 (SM) 是连接到子网的实体，负责配置和管理交换机、路由器和通道适配器。SM 可以与其他设备（如通道适配器或交换机）一起实现。

IBA 支持每个子网有多个子网管理器的概念，并指定多个子网管理器如何协商一个成为主 SM。它不禁止协作 SM 之间用于管理主/备关系的其他方法。

主 SM：
• 发现子网拓扑，
• 为每个通道适配器端口配置一个 LID 范围、GID 子网前缀和 P_Key，
• 为每个交换机配置一个 LID、子网前缀及其转发数据库，
• 维护子网的终端节点和服务数据库，从而提供 GUID 到 LID/GID 的解析服务以及服务目录。

3.4.5.2 子网管理代理

每个节点提供一个子网管理代理 (SMA)，SM 通过一个称为子网管理接口 (SMI) 的众所周知的接口访问它。SMI 允许 LID 路由数据包和定向路由数据包。定向路由提供了在交换机和终端节点配置之前进行通信的手段。只有 SMI 允许定向路由数据包。

3.4.5.3 通用服务代理

每个节点可以包含额外的管理代理，称为通用服务代理 (GSA*)，可以通过一个称为通用服务接口 (GSI) 的众所周知的接口访问。GSI 仅支持 LID 路由。IBA 定义的通用服务类包括：

• 子网管理 (SA) - 这是 SM 提供的一项服务，允许节点访问有关子网的信息以发现其他节点和服务、解析路径以及注册其服务。
• 性能管理 - 监控和报告定义明确的性能计数器
• 基板管理 - 使用第 2 卷中定义的 IB-ML 提供机箱管理
• SNMP 隧道 - 通过定义发送和接收 SNMP 消息的方法来提供 SNMP 功能。
• 供应商定义 - 允许设备供应商可能用于远程配置和管理其设备的私有扩展。
• 通信管理 (Com Mgt) - 提供终端节点之间的连接建立和其他通信管理功能。
• 设备管理 (DevMgt) - 提供 I/O 资源管理

3.5 IBA 特性

3.5.1 队列对

QP 是硬件提供给 IBA 消费者的虚拟接口，它为消费者提供一个虚拟通信端口。该体系结构支持每个通道适配器最多 2^24 个 QP，每个 QP 上的操作与其他 QP 独立。每个 QP 提供与其他 QP 操作和其他消费者的高度隔离和保护。因此 QP 可以被视为分配给单个消费者的私有资源。一个消费者可能消耗多个 QP，如图 17 所示。

图 17  通信接口

消费者通过分配 QP 并指定其服务类别来创建此虚拟通信端口。通信在源 QP 和目的 QP 之间进行。对于面向连接的服务，每个 QP 紧密绑定到恰好一个其他 QP，通常在不同的节点上。消费者启动将 QP 与目的 QP 绑定所需的任何通信建立，并使用某些信息（如目的 LID、服务级别和协商的操作限制）配置 QP 上下文。

消费者向 QP 提交工作请求以通过该 QP 调用通信。

3.5.2 服务类型

每个 QP 被配置为特定的操作类别（称为服务类型），基于源和接收 QP 的交互方式。源和目的 QP 必须配置为相同的服务类型。每种服务类型基于以下属性：

• 面向连接 vs 数据报 - 对于面向连接的服务，QP 与恰好一个其他 QP 关联，提交到 QP 的所有工作请求都会导致消息发送到已建立的目的 QP。数据报操作允许单个 QP 用于向/从任何节点上的任何适当 QP 发送和接收消息。

• 确认 vs 未确认 - 对于确认服务，QP 在接收请求消息时返回响应消息。响应消息可能是正确认 (ACK)、否定确认 (NAK) 或包含响应数据。确认服务被称为可靠服务，因为传输协议保证未损坏的数据按顺序恰好传递一次。未确认服务被称为不可靠服务，因为传输协议不保证所有数据都被传递。它确实保证所有数据最多传递一次，且已传递的数据未损坏。此外，在某些情况下，Fabric 配置的变化可能导致数据乱序传递。

• IBA 传输 vs 其他传输 - IBA 传输服务为基于通道和基于内存的操作定义了特定的传输协议。IBA 还支持使用通道适配器作为数据链路引擎在节点之间发送原始数据包，这对于支持传统协议栈和传统网络很有用。

IBA 定义的服务类型在表 2 中指定。

表 2  服务类型
服务类型             面向连接   确认    传输
可靠连接             是        是      IBA
不可靠连接           是        否      IBA
可靠数据报           否        是      IBA
不可靠数据报         否        否      IBA
原始数据报           否        否      Raw

某些 IBA 操作仅在特定服务类别上有效。QP 拒绝对已配置服务类别无效的操作的 WQE。

面向连接的服务要求消费者在任何 QP 操作之前启动通信建立过程（连接建立）与目标节点关联 QP 并建立 QP 上下文。实际上，除原始数据报之外的所有服务类别都需要某种形式的通信建立来关联队列对。对于可靠数据报服务，节点执行通信建立过程以将端到端 (EE) 上下文与每个目标节点关联。所有配置为可靠数据报服务的 QP 使用已建立的 EE 上下文，工作请求指定该操作使用哪个 EE 上下文。

原始数据报类似于不可靠数据报，区别在于源 QP 不知道将接收和处理消息的 QP 的身份。原始数据报允许路由器将原始数据报数据包转发到不具备 QP 等价物的异构 Fabric（如 LAN 或 WAN）上的非 IBA 目的地。原始数据报有两种类型：IPv6 和 EtherType。IPv6 原始数据报包含全局路由头部，数据包载荷包含全局路由头部中标识的传输协议服务数据单元。EtherType 原始数据报包含以太网类型字段，数据包载荷包含以太网类型字段中标识的传输协议服务数据单元。

IBA 定义了通道（发送/接收）和内存 (RDMA) 语义。原始数据报和不可靠数据报服务不支持内存语义。

3.5.3 密钥

IBA 使用各种密钥来提供隔离和保护。密钥是由管理实体分配的值，以各种方式在消息中使用。密钥本身不提供安全性，因为密钥在穿越 Fabric 的消息中可用，因此任何能够进入 Fabric 内部的实体都可以获知密钥值。IBA 确实对应用程序如何访问某些密钥施加了限制。

密钥包括：

• 管理密钥 (M_Key)：执行主子网管理器的控制。由子网管理器管理，用于某些子网管理数据包。每个通道适配器端口有一个 M_Key，由 SM 设置然后启用。SM 可以为每个端口分配不同的密钥。启用后，端口拒绝不包含已编程 M_Key 的某些管理数据包。因此只有具有已编程 M_Key 的 SM 才能更改节点的 Fabric 配置。只要 SM 处于活动状态，SM 可以防止端口的 M_Key 被读取。端口维护一个超时值，使得如果 SM 失败，端口将恢复到未管理状态。交换机有一个 M_Key。

• 基板管理密钥 (B_Key)：执行子网基板管理器的控制。由子网基板管理器管理，用于某些 MAD。每个通道适配器端口有一个 B_Key，由基板管理器设置。基板管理器可以为每个端口分配不同的密钥。启用后，端口拒绝不包含已编程 B_Key 的某些管理数据包。因此只有具有已编程 B_Key 的基板管理器才能更改节点的基板配置。只要基板管理器处于活动状态，基板管理器可以防止端口的 B_Key 被读取。端口维护一个超时值，使得如果基板管理器失败，端口将恢复到未管理状态。交换机有一个 B_Key。

• 分区密钥 (P_Key)：执行成员资格控制。由分区管理器 (PM) 通过子网管理器管理。每个通道适配器端口包含一个由 PM 设置的分区密钥表。QP 需要配置为同一分区才能通信（QP0、QP1 和配置为原始数据报的端口除外），因此 P_Key 在每个 IB 传输数据包中携带。通信建立过程的一部分确定特定 QP 或 EEC 使用哪个 P_Key。EEC 包含可靠数据报服务的 P_Key，QP 上下文包含其他 IBA 传输类型的 P_Key。QP 或 EEC 中的 P_Key 被放置在每个发送的数据包中，并与每个接收的数据包中的 P_Key 进行比较。P_Key 比较失败的接收数据包被拒绝。每个交换机有一个用于管理消息的 P_Key 表，并可选地支持基于 P_Key 过滤数据包的分区执行表。

• 队列密钥 (Q_Key)：执行可靠和不可靠数据报服务（不包括原始数据报服务类型）的访问权限控制。由通道适配器管理。在数据报服务的通信建立期间，节点为特定队列对交换 Q_Key，节点在发送到该远程 QP 的所有数据包中使用为远程 QP 传递给它的值。同样，远程节点使用提供给它的 Q_Key。接收到具有不同于节点提供给远程队列对的 Q_Key 的数据包意味着该数据包无效，因此被拒绝。最高位设置的 Q_Key 被视为受控 Q_Key（如 GSI Q_Key），HCA 不允许消费者任意指定受控 Q_Key。尝试发送受控 Q_Key 会导致使用 QP 上下文中的 Q_Key。因此 OS 维护控制，因为它可以为特权消费者配置 QP 上下文中的受控 Q_Key。

• 内存密钥 (L_Key 和 R_Key)：启用虚拟地址的使用，并为消费者提供控制其内存访问的机制。这些密钥通过注册过程由通道适配器管理。消费者向通道适配器注册一个内存区域并接收 L_Key 和 R_Key。消费者在工作请求中使用 L_Key 来向 QP 描述本地内存，并将 R_Key 传递给远程消费者以在 RDMA 操作中使用。当消费者排入 RDMA 操作时，它指定远程消费者传递给它的 R_Key，R_Key 包含在到原始通道适配器的 RDMA 请求数据包中。R_Key 验证发送方访问目的地内存的权利，并为目的通道适配器提供将虚拟地址转换为物理地址的手段。

3.5.4 虚拟内存地址

IBA 针对虚拟寻址进行了优化。也就是说，IBA 消费者在工作请求中使用虚拟地址，通道适配器能够根据需要将虚拟地址转换为物理地址。为此，每个消费者向通道适配器注册虚拟内存区域，通道适配器向消费者返回 2 个称为 L_Key 和 R_Key 的内存句柄。消费者然后在每个需要访问该区域的内存的工作请求中使用 L_Key。参见第 3.5.3 节了解 L_Key 用法描述。

内存注册提供机制，允许 IBA 消费者描述一组虚拟连续的内存位置或一组物理连续的内存位置，以允许 HCA 使用虚拟地址将内存作为虚拟连续缓冲区访问。

IBA 还支持远程内存访问 (RDMA)，允许远程消费者访问已注册的内存。对于 RDMA，消费者将 R_KEY 和该内存区域中缓冲区的虚拟地址传递给另一个消费者。该远程消费者在其将访问原始节点内存的 RDMA WQE 中提供该 R_Key。参见第 3.5.3 节了解 R_Key 用法的详细描述。

3.5.5 保护域

内存注册不仅允许使用虚拟内存寻址，还提供了更高级别的保护，以防止疏忽和未授权的访问。

由于消费者可能与许多不同的目的地通信，但不希望让所有这些目的地对其已注册内存具有相同的访问权限，IBA 提供了保护域。保护域允许消费者控制其哪些内存区域和内存窗口集可以被其哪些 QP 集访问。

在消费者分配 QP 或注册内存之前，它创建一个或多个保护域。QP 被分配到保护域，内存被注册到保护域。特定内存域的 L_Key 和 R_Key 仅在为同一保护域创建的 QP 上有效。

3.5.6 分区

分区在共享 InfiniBand Fabric 的系统之间强制隔离。分区与子网、交换机或路由器建立的边界无关。相反，分区描述了 Fabric 内可以通信的一组终端节点。

每个终端节点的每个端口至少是一个分区的成员，也可以是多个分区的成员。分区管理器为每个通道适配器端口分配分区密钥 (P_Key)。每个 P_Key 代表一个分区。每个 QP⁶ 和 EE 上下文被分配到一个分区，并在其发送的所有数据包中使用该 P_Key，并检查其接收的所有数据包中的 P_Key。接收到无效 P_Key 会导致数据包被丢弃。
⁶ 除 QP0、QP1 和配置为原始数据报服务类型的 QP。

交换机和路由器可以可选地用于强制执行分区。在这种情况下，分区管理器为交换机或路由器编程 P_Key 信息，当交换机或路由器检测到具有无效 P_Key 的数据包时，它丢弃该数据包。

3.5.7 虚拟通道

虚拟通道 (VL) 提供了在单个物理链路内创建多个虚拟链路的机制。虚拟通道代表端口中的一组发送和接收缓冲区。所有端口支持 VL15，它专门保留给子网管理使用。还有 15 个其他 VL（VL0 到 VL14），称为数据 VL，所有端口至少支持一个数据 VL (VL0)，并且可以提供 VL1 到 VLn-1（其中 n 是端口支持的数据 VL 数量）。

端口实际使用的数据 VL 由 SM 配置，并基于数据包中的服务级别 (SL) 字段。默认是使用 VL0，直到 SM 确定链路两端支持的 VL 数量并编程端口的 SL 到 VL 映射表。

端口维护每个数据 VL 上的独立流控制，使得一个 VL 上的过多流量不会阻塞另一个 VL 上的流量。

图 18  虚拟通道

VL 分配仅存在于链路两端的端口之间，一条链路上的 VL 分配独立于其他链路上的分配。

每个数据包都有一个 SL，在数据包头部中指定。当数据包穿越 Fabric 时，其 SL 决定在每条链路上使用哪个 VL。每个端口维护一个 SL 到 VL 的映射表，使得数据包在适当的 VL 上发送。

当链路两端的端口支持不同数量的数据 VL 时，具有较多数量的端口降级到另一个端口支持的数量。因此对于仅支持单个数据 VL 的端口，所有数据流量默认到 VL0。

3.5.8 服务质量

IBA 提供了几种机制，允许子网管理器为面向连接和无连接服务管理各种服务质量保证。这些机制是服务级别、服务级别到虚拟通道映射和分区。IBA 不定义服务质量 (QoS) 级别（例如尽力而为）。

3.5.8.1 服务级别

IBA 定义了一个服务级别 (SL) 属性，允许数据包在 16 个服务级别之一上操作。每个服务级别的定义和目的不在体系结构范围之内，留作 Fabric 管理策略。因此服务级别的分配是每个节点的通信管理器及其与子网管理器协商的功能。

3.5.8.2 SL 到 VL 映射

与服务级别相关的另一个 IBA 机制是虚拟通道。每个数据包标识其 SL，当数据包穿越 Fabric 时，数据包的 SL 决定在下一条链路上使用哪个 VL。为此，每个端口（交换机、路由器、终端节点）都有一个由子网管理配置的 SL 到 VL 映射表。自然地，对于终止于仅支持一个数据 VL 的端口的所有链路，所有 SL 映射到 VL0。否则，子网管理策略决定每个 SL 到可用 VL 的映射。

发往 QP0 的数据包是子网管理数据包 (SMP)，专门使用 VL15，其 SL 被忽略。VL15（管理 VL）不是数据 VL，不用于非 QP0 寻址的数据包。

3.5.8.3 分区

可以与服务级别绑定的另一个 IBA 机制是分区。Fabric 管理可以为特定分区分配某些 SL。这允许 SM 隔离这些分区之间的流量，即使两个分区在相同的 QoS 级别上运行，每个分区也可以保证其公平份额的带宽，无论其他分区中的节点是否行为不当或过度订阅。

3.5.9 注入速率控制

IBA 定义了多种不同的链路比特率。最低比特率 2.5 Gb/sec 被称为 1x（一倍）链路。其他链路速率为 10 Gb/sec (4x) 和 30 Gb/sec (12x)。为了在 Fabric 内支持多种链路速度，IBA 定义了一种静态速率控制机制，防止具有高速链路的端口超过具有低速链路端口的容量。

作为路径解析过程的一部分，SubnAdm:PathRecord 为节点提供路径的 MTU 和速率信息。使用路径信息是因为交换机端口或终端节点都可能是限制因素。

图 19  速率匹配示例

每个目的地都有一个关联的超时值，该超时值基于源和目的比特率之间的比率。当源和目的比特率相等时，超时值为 0（不需要）。否则，当端口向目的地发送数据包时，它将该目的 LID 和超时值放入其静态速率控制表中。端口在超时期限到期后移除该条目。当条目仍在表中时，端口不向该目的地发送更多数据包（即，让位于不在表中的其他目的地的流量）。当表中没有条目时，端口通过将数据包放在适当的 VL 输出队列上来发送数据包。

3.5.10 寻址

每个终端节点包含一个或多个通道适配器，每个通道适配器包含一个或多个端口。此外，每个通道适配器包含多个队列对 (QP)。

每个 QP 有一个由通道适配器分配的队列对编号 (QPN)，在通道适配器内唯一标识该 QP。每个端口有两个众所周知的 QP（QP0 和 QP1），所有其他 QP 被配置为通过特定端口操作。对于可靠数据报服务，决定端口的是 EE 上下文而不是 QP 上下文。

除原始数据报外的数据包包含目的 QP 的 QPN。当通道适配器接收到数据包时，它使用目的 QPN 的上下文（可靠数据报的 EE 上下文）来处理数据包。

每个端口有一个由本地子网管理器分配的本地 ID (LID)。在子网内，LID 是唯一的。交换机使用 LID 在子网内路由数据包。本地子网管理器基于 LID 以及该端口相对于特定交换机的位置来配置交换机中的路由表。每个数据包包含一个源 LID (SLID)，标识将数据包注入子网的端口，以及一个目的 LID (DLID)，标识 Fabric 要将数据包传递到的端口。

IBA 还通过定义 LID 掩码控制 (LMC) 来提供物理端口内的多个虚拟端口。LMC 指定物理端口在验证数据包 DLID 是否与其分配的 LID 匹配时掩码（忽略）的 LID 最低有效位数。这些位不被交换机忽略，因此子网管理器可以基于这些最低有效位编程通过 Fabric 的不同路径。因此端口在通过 Fabric 路由方面表现为 2^LMC 个端口。

每个端口还有至少一个全局 ID (GID)，格式为 IPv6 地址。GID 是全局唯一的。每个数据包可选地包含一个全局路由头部 (GRH)，指定一个源 GID (SGID)和一个目的 GID (DGID)。路由器使用 GRH 在子网之间路由数据包。交换机忽略 GRH。

每个通道适配器有一个由通道适配器供应商分配的全局唯一标识符 (GUID)，称为节点 GUID。其每个端口也有一个端口 GUID。端口 GUID 与本地子网前缀组合成为端口的默认 GID。

子网管理提供 GUID 到 LID/GID 的解析服务。因此节点可以通过记住节点或端口 GUID 来持久标识另一个节点。

QP 的地址是端口地址 (GID + LID) 和 QPN 的组合。与 QP 通信需要一个信息向量，包括端口地址（LID 和/或 GID）、QPN、服务级别、路径 MTU，以及可能的其他信息。这些信息可以通过向子网管理发送路径查询请求来获得。

服务 ID 用于解析 QP。某些服务 ID 是众所周知的，有些在 I/O 控制器的服务条目列表中公布。子网管理器提供 GUID 到 GID/LID 的解析，但目标作为通信管理过程的一部分提供服务 ID 到 QP 的解析。

3.5.11 多播

多播是一种一对多/多对多的通信范式，旨在简化和提高一组终端节点之间的通信效率。

每个多播组由一个唯一的 GID 标识。节点通过管理操作加入和离开多播组，其中节点为将参与的每个端口提供 LID。此信息被分发到交换机。每个交换机被配置了多播流量的路由信息，指定数据包需要经过的所有端口。需注意确保没有环路（即单一生成树）。

节点在发送到该多播组的所有数据包中使用多播 LID 和 GID。当交换机接收到多播数据包时，它复制数据包并将其发送到除到达端口外的每个指定端口。通过这种方式，每个级联交换机复制数据包，使得数据包仅到达每个已订阅终端节点一次。

3.5.11.2 组管理

IBA 不定义实现加入和离开操作的多播组管理协议。但是，指定了实现多播组协议的管理接口和相关 MAD。

3.5.11.2.1 多播组创建
多播组创建在 IBA 中是一个显式操作，以提供对组特性的单一控制并允许成员隐蔽地加入。

3.5.11.2.2 多播组加入
多播组加入算法适用于 IBA 不可靠数据报多播组，包括验证多播组地址、验证 PMTU、建立多播路由树等步骤。

3.5.11.2.3 多播组离开
当应用程序离开多播组时，其 QP 被移除为多播组的目标。如果端口上没有更多 QP 参与，则通知多播管理实体更新转发表。

3.5.11.2.4 多播组删除
当不再需要多播组时，取消多播 LID 映射，并通知子网中的每个路由器。

3.5.11.3 多播裁剪
多播组管理实体应定期验证参与多播组的所有终端节点和路由器是否仍在参与，如果没有，应将它们从多播组中裁剪。

3.5.12 动词

IBA 通过一组称为动词 (Verbs) 的语义描述了主机通道适配器与操作系统之间的服务接口。动词描述了基于特定队列模型在主机通道适配器与其操作系统之间发生的操作。

动词的目的不是指定 API，而是充分描述接口，允许 OS 厂商定义利用该体系结构的适当 API。

动词描述了配置和管理通道适配器、分配队列对、配置 QP 操作、向 QP 提交工作请求、从完成队列获取完成状态所需的参数。

3.6 通道与内存语义

IBA 通信为用户提供通道语义和内存语义，因为两者对 I/O 和 IPC 都很有用。通道语义，有时称为发送/接收，指的是经典 I/O 通道中使用的通信方式——一方推送数据，目的方确定数据的最终目的地。

通过内存语义，发起方直接读取或写入远程节点的虚拟地址空间。远程方只需传达缓冲区的位置；它不参与数据的实际传输。

3.6.1 通信接口

发送队列支持的操作：
• 发送缓冲区 - 通道语义操作，将本地缓冲区推送到远程 QP 的接收缓冲区。
• RDMA 读 - 内存语义操作，读取远程节点上的虚拟连续缓冲区。
• RDMA 写 - 内存语义操作，写入远程节点上的虚拟连续缓冲区。
• 原子操作 - 内存语义操作，对远程 64 位字执行原子操作（比较并交换、FetchAdd）。
• 内存绑定 - 内存管理操作，更改内存窗口的绑定。

接收队列只有一种 WR：
• 提交接收缓冲区 - 通道语义操作，描述传入发送消息将写入的本地缓冲区。

零处理器拷贝数据传输，无需内核参与，是提供高带宽、低延迟通信的关键。IBA 操作支持使用虚拟地址和现有虚拟内存保护机制来确保对所有内存的正确和适当访问。

3.6.2 IBA 传输服务

IBA 传输机制提供多种通信服务类别：
• 可靠连接（确认 - 面向连接）
• 可靠数据报（确认 - 多路复用）
• 不可靠连接（未确认 - 面向连接）
• 不可靠数据报（未确认 - 无连接）
• 原始数据报（未确认 - 无连接）

可靠连接服务将本地 QP 与一个且仅一个远程 QP 关联。可靠连接是可靠的，因为通道适配器维护序列号并确认所有消息。确认确保数据在关联的 QP 之间可靠传递。确认不是消费者级别的确认——它仅意味着数据已到达目的地。

不可靠连接服务也将本地 QP 与一个远程 QP 关联，但不确认，因此没有重新发送丢失消息的能力。不支持 RDMA 读和原子操作。

不可靠数据报服务是无连接和未确认的，极大地提高了 IBA 的可扩展性。最大消息长度被限制为适合单个数据包的大小。

可靠数据报 (RD) 服务通过端到端上下文 (EEC) 进行多路复用。对于每个 RDD，每个消费者只需创建单个 QP，节点为其通信的每个平台创建一个 EE 上下文。

原始数据报服务是一种数据链路服务，允许 QP 发送和接收原始数据报消息（EtherType 和 IPv6 两种类型）。

3.7 IBA 分层体系结构

IBA 操作可以描述为一系列层。每层的协议独立于其他层。

图 25  IBA 体系结构层

3.7.1 物理层
物理层指定如何将位放在线路上以形成符号，并定义用于成帧、数据符号以及数据包之间填充的符号。物理层规范在第 2 卷中。

3.7.2 链路层
链路层描述了数据包格式和数据包操作协议，包括流控制以及数据包如何在子网内路由。有两种类型的数据包：
• 链路管理数据包 - 用于训练和维护链路操作，不受流控制约束，永远不会被转发。
• 数据数据包 - 传递 IBA 操作的数据包。

图 27  IBA 数据数据包格式
LRH | GRH | BTH | ETH | Payload | IData | ICRC | VCRC

本地路由头部 (LRH) 始终存在，标识本地源和目的端口，指定 SL 和 VL。
每个数据包有两个 CRC：不变 CRC (ICRC) 覆盖不应改变的字段；可变 CRC (VCRC) 覆盖所有字段。
链路级流控制是基于信用的方法，信用按 VL 分配。VL15 不受流控制约束。

3.7.3 网络层
网络层描述了在子网之间路由数据包的协议。全局路由头部 (GRH) 存在于穿越多个子网的数据包中。路由器修改 GRH 内容并替换 LRH，但源和目的 GID 不变并受 ICRC 保护。

3.7.4 传输层
传输层将数据包传递到正确的 QP 并指示 QP 如何处理数据。传输层负责在消息数据载荷大于路径 MTU 时进行分段。

基本传输头部 (BTH) 存在于除原始数据报之外的所有数据包中，指定目的 QP、操作码、数据包序列号和分区。

各种扩展传输头部 (ETH) 根据服务类别和操作码条件性存在。
立即数据 (IMMDT) 字段可选存在于 RDMA WRITE 和 SEND 消息中。

对于可靠连接服务，IBA 定义了端到端消息级流控制。

3.7.5 上层协议
IBA 支持任意数量的上层协议。IBA 还定义了某些管理功能的消息和协议。

3.7.5.1 子网管理
子网管理分为子网管理器 (SM) 应用和子网管理代理 (SMA)。每个子网只需一个子网管理器。子网管理使用称为子网管理数据包 (SMP) 的特殊类别管理数据报 (MAD)，定向到特殊队列对 (QP0)。SMA 处理在 QP0 上接收的 Get() 和 Set() SMP，发送 GetResp() 和 Trap() SMP。

3.7.5.2 通用服务
通用服务代理 (GSA*) 由多个管理服务代理组成。通用服务使用称为通用管理数据包 (GMP) 的消息格式，通常定向到特殊队列对 (QP1)，称为通用服务接口 (GSI)。GSA 能够将特定服务类别的 GMP 重定向到另一个队列对。

3.8 IBA 事务流

消费者通过队列对与 IBA 通道适配器交互。消息通过提交工作请求发起。通道适配器检测 WQE 提交、解释命令、验证虚拟地址、转换为物理地址、访问数据、分段为数据包、添加头部并发送。

数据包是通过 IBA Fabric 路由的信息单元。数据包是终端节点到终端节点的构造。交换机可能更改虚拟通道并替换 VCRC 但不触及 ICRC。路由器更改 LRH 和 GRH 中的字段，更新 VCRC 但不更改 ICRC。

在目的地，进行正常有效性检查和 CRC 完整性检查。传输头部标识目标 QP。对于可靠服务，如果检测到丢失的数据包，发送 NAK 消息。发起方收到确认后，创建 CQE 并退休 WQE。

3.9 IBA 管理基础设施

IBA 管理定义了通用管理基础设施，包括：
• 子网管理 - 提供子网管理器发现和配置 IBA 设备及管理 Fabric 的方法。
• 通用管理服务：
    • 子网管理 - 提供 SM 收集的信息和节点注册服务的登记处
    • 通信建立与连接管理
    • 发现和管理通道适配器"后面"的 I/O 设备的机制
    • 配置管理
    • 性能管理
    • 基板管理
    • SNMP 隧道

图 32  子网管理模型
每个通道适配器、交换机和路由器都有一个子网管理代理 (SMA)，响应子网管理数据包。通信使用子网管理接口 (SMI)，每个端口有一个 QP0。

保护 - 子网管理器可以在每个节点中放置一个密钥 (M_Key)，不能被其他节点读取，并防止没有 M_Key 的节点修改节点的配置。IBA 还提供租约过期机制。

管理模型：
• 被管代理模型 - 类管理器通过每个节点上的 GSA 管理节点。
• 对等代理模型 - 驻留在每个节点上的管理器之间相互通信（通信管理类使用此模型）。
• 被管服务模型 - 管理应用程序访问类管理器（子网管理和 I/O 资源管理类使用此模型）。

3.9.1 管理数据报
IBA 定义了支持通用处理的标准管理消息格式。每个 MAD 包含标识管理消息类别和方法的相同头部格式。RMPP 是允许管理实体交换超过单个 MAD 数据量的协议。

3.9.2 管理方法
IBA 定义了可被任何类采用的通用方法，包括 Get、Set、GetResp、Send、Trap、TrapRepress、Report 和 ReportResp。

3.9.2.1 Get 和 Set
SM 通过发送 SMP Get 请求消息轮询 Fabric 并了解其拓扑。通过发送 Set 请求消息配置 IBA 设备。

3.9.2.2 陷阱和通知
陷阱是管理代理在某些异步管理事件发生时发送给其类管理器的消息。通知是可以在被管节点的通知队列中排队的属性。

3.9.2.3 发送
Send() 是一种管理方法，一个实体在特定类的基础上向另一个实体发送数据。

3.9.2.4 报告
管理基础设施提供了管理实体向类管理器订阅以使类管理器转发其接收的陷阱的手段。

3.9.3 管理接口
IBA 定义了两个众所周知的 QP 用于管理接口。QP0 保留给子网管理，QP1 指定给通用管理服务。

3.9.4 子网管理接口
每个 IBA 端口有一个专用于子网管理的 QP0。QP0 永久配置为不可靠数据报服务；是所有分区的成员；仅接受 SMP；专门使用 VL15。

3.9.4.1 Fabric 初始化
子网管理器使用此服务接口轮询和配置 Fabric。交换机支持定向路由，允许在交换机配置转发数据库和节点被分配本地 ID 之前路由 SMP。SM 发现所有设备和互连方式后，配置每个节点的本地 ID 和交换机的路由表。

3.9.4.2 定向路由
SMP 可以指定它通过 Fabric 的路由。通过在 SMP 中包含一个端口号列表（路径向量）来定义通过子网的路径。数据包包含两个路径向量（正向和反向）、一个方向位和一个跳指针。

3.9.5 通用服务接口
每个 IBA 通道适配器有一个专用于通用 Fabric 服务的 QP1。QP1 永久配置为不可靠数据报服务；是端口所有分区的成员；仅接受 MAD；不使用 VL15。

3.9.5.1 重定向
IBA 定义了允许通道适配器将通用服务请求重定向到其他 QP 的机制。

3.10 I/O 操作

IBA I/O 架构支持从简单原生设备到大规模 I/O 子系统的一系列 I/O 实现。I/O 单元的模型如图 35 所示。I/O 单元由一个通道适配器和多个 I/O 控制器组成。I/O 单元的通道适配器被称为目标通道适配器 (TCA)。TCA 具有与 HCA 相同的功能，但与 HCA 不同的是，它不一定设计用于通用用途，这意味着它只需支持其控制器所需的功能。

图 35  I/O 单元

I/O 控制器代表处理 I/O 事务请求的硬件和软件。I/O 控制器的示例包括 SCSI 接口控制器、RAID 处理器、存储阵列处理器、LAN 端口控制器、磁盘驱动器控制器和控制台服务。

I/O 单元包含一个子网管理代理 (SMA) 响应 QP0 上接收的 SMP，还包含通用服务代理 (GSA*) 响应 GSI (QP1) 上接收的 GMP。GSA* 至少包含通信管理和 I/O 设备管理 (DevMgt)。每个 I/O 控制器向 DevMgt GSA 注册，使其能够以控制器的特定信息响应 DevMgt GMP。

通常处理器节点中的 I/O 资源管理器向 I/O 单元发送 DevMgt GMP 以发现控制器的属性。属性包含足够的信息供 I/O 资源管理器识别适当的 I/O 驱动程序。I/O 资源管理器加载驱动程序（如有必要），并用控制器的标识（I/O 单元和控制器 ID）配置 I/O 驱动程序。I/O 驱动程序然后在处理器节点上创建适当的通信端口（即 QP），并调用处理器节点的通信管理器与 I/O 控制器创建适当的连接。连接建立后，I/O 驱动程序通过连接交换控制消息和数据。

图 36  I/O 操作

驱动程序使用的通信端口数量是一个实现变量。I/O 驱动程序可以使用任何可用的服务类别（可靠连接、不可靠连接、可靠数据报或不可靠数据报），并可能为不同的通信端口使用不同的服务类别。
